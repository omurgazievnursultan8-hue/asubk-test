### Фаза 1 — Решение правительства: предложения по улучшению (для команды разработки)

- [ ] R1 🔴 Вложение документа решения
  - сейчас: поля для файла нет, есть только текстовое «Примечание»;
  - добавить загрузку файлов отдельно от «Примечания»;
  - формат только **PDF**, лимит размера (≈10–20 МБ), валидация при загрузке;
  - можно прикрепить **несколько** файлов (основное решение + приложения/изменения), ровно один помечается как «основной»;
  - на стадии «На рассмотрении» файл необязателен; для перехода в «**Действует**» (бывш. «Одобрен», см. R7) наличие основного документа **обязательно**;
  - замена файла — новой версией с сохранением истории (без перезаписи); у одобренного решения удаление файла запрещено;
  - кто и когда загрузил/заменил файл — фиксируется в журнале (см. R2).
- [ ] R2 🔴 Отклонение решения: подтверждение, причина, аудит, обратимость
  - сейчас: «Отклонить» срабатывает мгновенно, без причины и следа, необратимо (P1-03);
  - перед отклонением — диалог подтверждения с явными кнопками (защита от случайного клика);
  - обязательное поле «Причина отклонения» — без него «Отклонить» недоступна; причина видна в просмотре записи и в списке;
  - журнал аудита по каждому переходу статуса (одобрение/отклонение/реактивация): кто, когда, действие, причина — это же закрывает аудит загрузки файла из R1;

- [ ] R3 🔴 Поле «Код» — системная генерация вместо ручного ввода
  - сейчас: код вводится **вручную** (поле редактируемо, обязательно при создании), без проверки уникальности, есть пустые legacy-записи (P1-02);
  - код должен генерироваться **системой** при создании, ручной ввод убрать, поле сделать read-only (заодно убирает проблему «серого» стиля поля — P1-01);
  - формат `{ТИП}-{ГГГГ}-{NNNN}`: ТИП — короткий код вида решения (ПОСТ/РАСП/ОСН/РКМКР/РПКР/ПКМКР), ГГГГ — год из «Даты решения», NNNN — сквозной порядковый номер внутри пары тип+год с ведущими нулями (пример: ПОСТ-2026-0007);
  - уникальность гарантируется счётчиком;
  - старые пустые коды оставляем как legacy (предстоит полное обнуление данных, миграция не нужна).
- [ ] R4 🟠 Удаление решений: защита ссылок, статус, soft-delete

  - серверная проверка ссылок: запретить удаление, если на решение ссылается хотя бы одна кредитная программа (или иная downstream-сущность) — проверка на сервере, не только скрытием кнопки;
  - при попытке удалить используемое решение — внятное сообщение («Нельзя удалить: на решение ссылаются N программ»), а не молчаливый отказ;
  - выровнять с редактированием: для одобренных/закрытых записей удаление недоступно (кнопка неактивна);
  - soft-delete вместо физического удаления — запись помечается удалённой, остаётся в БД, действие пишется в журнал аудита (см. R2);
  - уточнить полный список downstream-ссылок на решение при реализации фаз 2+.

  - валидация и на клиенте (календарь не даёт выбрать будущее), и на сервере (чтобы не обойти через API);
  - при попытке ввести будущую дату — внятное сообщение («Дата решения не может быть в будущем»), а не молчаливый сброс;

    - статусы (6): Черновик → На рассмотрении → На доработке → **Действует** → Отклонён → Закрыт;
    - переходы (10): Создать/Отправить/Отозвать/Одобрить(→Действует)/Отклонить/Вернуть/Отправить повторно/Закрыть/Удалить/Дублировать;

### Фаза 2 — Кредитные программы: предложения по улучшению (для команды разработки)

Нумерация `P2-R*` — отдельный счётчик фазы (как `P2-xx` у дефектов); счётчик

- [ ] P2-R1 🟡 Визуальная семантика полей формы-мастера
  - сейчас: ~40 полей на 9 вкладках в одинаковом сером «disabled»-стиле; обязательные/опциональные/readonly визуально неотличимы; обязательные видно только после неудачного сохранения (P2-01);
  - ввести **3 состояния поля** с разным видом:
    - **обязательное-редактируемое** → белый фон + рамка + красный `*` у подписи;
    - **опциональное-редактируемое** → белый фон + рамка, без `*`;
    - **только для чтения / вычисляемое** → серый фон, без рамки (напр. «Валюта» на вкл. 2, «Выбранные шаблоны», вся вкл. «Предпросмотр»);

  - **единый стиль с R6** (Фаза 1): один компонент-стиль полей дизайн-системы ASUBK, применяемый в обоих модулях;
  - применить ко всем 9 вкладкам мастера;
  - тот же корень, что P1-01/P1-06.
- [ ] P2-R2 🟠 Валидация «Дата с» / «Дата по»
  - сейчас: у пикеров нет min/max и нет проверки порядка — «Дата по» раньше «Даты с» не блокируется (P2-02);
  - **«Дата по» ≥ «Дата с»** (равенство допускается — программа на 1 день) — на клиенте (пикер «Дата по» не даёт выбрать раньше «Даты с») и на сервере;
  - **«Дата с» ≥ «Дата решения»** связанного решения правительства — нижняя граница (у решения нет периода действия, только дата принятия, проверено 2026-06-17, поэтому она и есть нижний порог);
  - будущие даты для программы **допустимы** (в отличие от решения, R5) — это проверка порядка/согласованности, а не запрет будущего;
  - при нарушении — внятное сообщение, а не молчаливый сброс.
- [ ] P2-R3 🟡 Семантика и подписи кнопок сохранения
  - факт (проверено 2026-06-17 на тест-записи): «OK» = сохранить **и закрыть** редактор; «Сохранить» = сохранить **и остаться**; обе сохраняют данные — это штатный паттерн Jmix, не потеря данных (P2-03);
  - проблема: разница не очевидна, подписи непоследовательны (англ. «OK» рядом с рус. «Сохранить»/«Отмена»);
  - сделать: единый язык подписей и понятная семантика — напр. «Сохранить» (остаться) + «Сохранить и закрыть» + «Отмена»; либо тултипы на кнопках;
  - мелочь: после «OK» переход идёт на `/` (главную), а не в список программ — вернуть в список.
- [ ] P2-R4 🟡 Отдельная страница деталей программы («Просмотр»)
  - сейчас: в тулбаре только Создать/Изменить/Удалить — нет режима просмотра, инспекция вынуждает входить в редактирование (риск случайной правки) (P2-04); расходится с UX решений правительства, где «Просмотр» есть;
  - сделать: **отдельную страницу деталей** программы (не вкладку «Предпросмотр» и не read-only мастер) — единый читаемый обзор всех данных программы;
  - открытие из списка: **кнопка «Просмотр»** в тулбаре (выбрать строку → Просмотр) **и двойной клик по строке**;
  - со страницы просмотра — кнопка **«Изменить»** (переход в редактирование), выровнять с моделью Фазы 1 (R6 «Просмотр + действия»).
- [ ] P2-R5 🟠 Формализовать воркфлоу статусов программы
  - сейчас: в списке только «Черновик», нет Одобрить/Отклонить; единственный переход — «Отправить на подтверждение» на вкладке «Предпросмотр»; полный набор статусов, переходы после отправки и роль подтверждающего в UI не определены (P2-05);

  - факт (проверено 2026-06-17): удаление программы со ссылкой даёт диалог «Непредвиденная ошибка» с сырым `DeletePolicyException: Unable to delete LoanProgram because there are references from GovDecision` (англ., техническое) (P2-07);

- [ ] P2-R8 🟠 Баг: направление delete-policy для программы
  - факт (проверено 2026-06-17): только что созданная программа (ID 24) **без** downstream-ссылок не удаляется, ошибка винит «references from GovDecision» — хотя это программа ссылается на решение, а не наоборот (P2-07);
  - похоже на инвертированную политику удаления / back-reference в модели;
  - сделать: разработчику проверить направление связи LoanProgram↔GovDecision и delete-policy; программа без реальных downstream-ссылок (заявки/кредиты) должна удаляться;
  - отдельно от P2-R7 (там только подача ошибки, здесь — корректность самой политики).
- [ ] P2-R9 🔴 Выбор «Решения правительства» только из одобренных — серверный enforcement
  - факт (проверено 2026-06-19): фильтр «Статус = Одобрен» в picker'е — **только дефолтный UI-фильтр и снимается** одним кликом по чипу; после снятия в выборе появляются все решения, включая «На стадии рассмотрения» и «Закрыт» (7 → 13 строк), их можно выбрать и **программа сохраняется без серверной проверки** (P2-08);
  - сделать: серверную валидацию при сохранении — связанное решение должно быть **одобренным/действующим**, иначе сохранение отклоняется с внятным сообщением (не полагаться на UI-фильтр);
  - в picker'е неодобренные решения **скрывать** (жёсткий фильтр загрузчика), а не просто ставить съёмный дефолт; либо блокировать выбор;
  - picker открывает **полный экран-справочник** с действиями Создать/Изменить/Удалить/Одобрить/Отклонить — это слишком широкий доступ из формы программы; ограничить до режима выбора;
  - увязать со статусами решения (Фаза 1 R7: «Одобрен» → «Действует») — ссылаться можно только на «Действует».

Нумерация `P3-R*` — отдельный счётчик фазы (как `P3-xx` у дефектов). Дефекты —
`notes/qa-findings.md` (P3-xx). Модуль зрелее предыдущих: уже есть статус-степпер,
4-eyes-голосование комиссии, inline-валидация, режим просмотра у комиссии._
- [ ] P3-R1 🟡 Визуальная семантика полей формы заявки
  - сейчас: в диалоге «Новая заявка» редактируемые поля (Адрес, Номер телефона, Источник финансирования) в сером «disabled»-стиле; обязательные/опциональные/readonly визуально неотличимы (P3-01);
  - **область — только форма создания** («Новая заявка»), где серый стиль подтверждён визуально; страница деталей и экран комиссии — вне этой задачи;
  - применить **тот же единый стиль полей дизайн-системы**, что вводится в P1-R6/P2-R1: обязательное-редактируемое → белый фон + рамка + красный `*`; опциональное → белый фон + рамка; readonly/вычисляемое → серый;
  - переиспользовать **общий компонент** из P1-R6/P2-R1 (не плодить новый), задача остаётся отдельной по фазе для трекинга;
  - прим.: обязательные поля здесь уже показывают `*` и inline «Поле является обязательным» — остаётся только серый базовый стиль (тот же корень, что P1-01/P2-01).
- [ ] P3-R2 🟡 Отдельная страница деталей заявки («Просмотр»)
  - сейчас: в тулбаре заявок Создать/Изменить/Удалить/Отправить в комиссию/Отправить в залоговую комиссию — нет «Просмотр», инспекция вынуждает входить в редактирование (`/loan-applications/{id}`, без `?mode=readonly`) (P3-02);
  - **расхождение внутри фазы**: соседний экран «Комиссии по заявкам» «Просмотр» и readonly-режим **имеет**;
  - сделать: **отдельную страницу деталей** заявки (как решили для программ в P2-R4 и для решений в Фазе 1 R6) — единый читаемый обзор всех данных заявки;
  - открытие из списка: **кнопка «Просмотр»** в тулбаре (выбрать строку → Просмотр) **и двойной клик по строке → Просмотр (readonly)**;
  - со страницы просмотра — кнопка **«Изменить»** (переход в редактирование);
  - выровнять с P2-R4 и Фазой 1 (R6 «Просмотр + действия»).
- [ ] P3-R3 🟠 Валидация дат и чисел заявки
  - факт (проверено 2026-06-18): у полей «Запрашиваемая сумма» и «Запрашиваемый срок» нет min/max/step, у пикера даты комиссии нет min — клиент не блокирует 0/отрицательное/прошлые даты (P3-03);
  - **числа заявки**: «Запрашиваемая сумма» > 0 и «Запрашиваемый срок» > 0;
  - **границы программы**: запрашиваемая сумма и срок должны укладываться в min/max суммы и срока, заданные в связанной кредитной программе (Фаза 2) — кросс-проверка с правилами программы;
  - **даты комиссии**: «Крайний срок» голосования ≥ сегодня; «Дата заседания» не раньше создания комиссии (порядок дат);
  - валидация на клиенте и на сервере, с внятными сообщениями (не молчаливый сброс);
  - прим.: у самой заявки своих дат нет (только срок-число), поэтому проверка завязана на границы программы, а не на окно её действия.
- [ ] P3-R4 🟠 Правила работы комиссии (финальное решение, 4-eyes)
  - голосование реализовано (Прогресс голосования, Члены/Председатель, Решение + Крайний срок, Финальное решение, Протокол), но правила в UI не видны (P3-04);
  - позиция заказчика согласована 2026-06-19 + проверено на стенде 2026-06-19 (`scripts/inspect/verify-p3r4*.mjs`); scope разработки сузился до двух пунктов:
  - **#1 — «Финальное решение» ручное + аудит** 🟠 (разработка):
    - UI-поле уже ручное (отдельная секция «Финальное решение (Председатель комиссии)», голосование — отдельная кнопка «Проголовать»); править UI не нужно;
    - добавить **аудит-запись** при сохранении финала: кто/когда/какое решение + причина, особенно когда финал расходится с тэлли голосов (через UI не виден — вероятно нужен backend);
  - **#2 — 4-eyes предупреждение** 🟠 (разработка):
    - если автор/податель заявки оказался членом/председателем комиссии по своей заявке — не блокировать, показать предупреждение + пометить в аудите;
    - проверено: такого предупреждения в UI сейчас нет → новая разработка;
  - **#3 — кворум: без порога** ✅ no-op (подтверждено заказчиком + стендом): голосование закрывает председатель в любой момент, голоса справочно; текста про кворум/порог в UI нет, авто-блокировки по тэлли нет — только задокументировать;
  - **#4 — порядок комиссий: параллельно, независимо** ✅ no-op (подтверждено заказчиком + стендом): обе кнопки «Отправить в комиссию»/«Отправить в залоговую комиссию» активны независимо, гейтинга нет; кредитная и залоговая параллельны (2 статуса заявки) — только задокументировать;
  - **#5 — «Крайний срок» члена: напоминание/эскалация** 🟠 (разработка, согласовано 2026-06-19):
    - дедлайн принуждается **уведомлением**: при приближении/истечении срока — нотификация члену + председателю;
    - голос **не меняется автоматически** (без авто-«воздержался» — иначе конфликт с «председатель закрывает голосование в любой момент»); голос открыт пока не подан или председатель не закрыл;
    - новая разработка: таймер/планировщик + механизм нотификаций;
  - ✅ готово к разработке; реальный код — #1 (аудит) + #2 (warning) + #5 (напоминание/эскалация).
