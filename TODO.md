### Фаза 1 — Решение правительства: предложения по улучшению (для команды разработки)
_Полное обоснование в `requirements/features/01-government-decision.md` → Recommendations._
- [ ] R1 🔴 Вложение документа решения
  - сейчас: поля для файла нет, есть только текстовое «Примечание»;
  - добавить загрузку файлов отдельно от «Примечания»;
  - формат только **PDF**, лимит размера (≈10–20 МБ), валидация при загрузке;
  - можно прикрепить **несколько** файлов (основное решение + приложения/изменения), ровно один помечается как «основной»;
  - на стадии «На рассмотрении» файл необязателен; для перехода в «**Действует**» (бывш. «Одобрен», см. R7) наличие основного документа **обязательно**;
  - замена файла — новой версией с сохранением истории (без перезаписи); у одобренного решения удаление файла запрещено;
  - кто и когда загрузил/заменил файл — фиксируется в журнале (см. R2).
- [ ] R2 🔴 Отклонение решения: подтверждение, причина, аудит, обратимость
  - сейчас: «Отклонить» срабатывает мгновенно, без причины и следа, необратимо (P1-03);
  - перед отклонением — диалог подтверждения с явными кнопками (защита от случайного клика);
  - обязательное поле «Причина отклонения» — без него «Отклонить» недоступна; причина видна в просмотре записи и в списке;
  - журнал аудита по каждому переходу статуса (одобрение/отклонение/реактивация): кто, когда, действие, причина — это же закрывает аудит загрузки файла из R1;
  - обратимость: дать администратору (по праву/роли) вернуть запись из «Закрыт» обратно «На рассмотрении» с обязательной записью в журнал — мис-клик не должен навсегда хоронить юр. основание (⚠️ расходится с моделью R7, где «Закрыт» терминальный, а обратимость — через «Дублировать»; выбрать одну модель в R7).
- [ ] R3 🔴 Поле «Код» — системная генерация вместо ручного ввода
  - сейчас: код вводится **вручную** (поле редактируемо, обязательно при создании), без проверки уникальности, есть пустые legacy-записи (P1-02);
  - код должен генерироваться **системой** при создании, ручной ввод убрать, поле сделать read-only (заодно убирает проблему «серого» стиля поля — P1-01);
  - формат `{ТИП}-{ГГГГ}-{NNNN}`: ТИП — короткий код вида решения (ПОСТ/РАСП/ОСН/РКМКР/РПКР/ПКМКР), ГГГГ — год из «Даты решения», NNNN — сквозной порядковый номер внутри пары тип+год с ведущими нулями (пример: ПОСТ-2026-0007);
  - уникальность гарантируется счётчиком;
  - старые пустые коды оставляем как legacy (предстоит полное обнуление данных, миграция не нужна).
- [ ] R4 🟠 Удаление решений: защита ссылок, статус, soft-delete
  - сейчас: «Удалить» активна даже на одобренных/закрытых; редактирование уже корректно заблокировано, удаление — нет (P1-05);
  - серверная проверка ссылок: запретить удаление, если на решение ссылается хотя бы одна кредитная программа (или иная downstream-сущность) — проверка на сервере, не только скрытием кнопки;
  - при попытке удалить используемое решение — внятное сообщение («Нельзя удалить: на решение ссылаются N программ»), а не молчаливый отказ;
  - выровнять с редактированием: для одобренных/закрытых записей удаление недоступно (кнопка неактивна);
  - soft-delete вместо физического удаления — запись помечается удалённой, остаётся в БД, действие пишется в журнал аудита (см. R2);
  - уточнить полный список downstream-ссылок на решение при реализации фаз 2+.
- [ ] R5 🟠 Валидация «Даты решения»
  - сейчас: календарь/поле принимают будущие даты (P1-04);
  - запретить будущие даты: «Дата решения» не может быть позже сегодняшнего дня;
  - валидация и на клиенте (календарь не даёт выбрать будущее), и на сервере (чтобы не обойти через API);
  - при попытке ввести будущую дату — внятное сообщение («Дата решения не может быть в будущем»), а не молчаливый сброс;
  - нижний порог даты пока не вводим.
- [ ] R6 🟠 Страница деталей решения — переработать по мокапу
  - сейчас: страница — это набор неактивных форм без единого режима просмотра;
  - сделать страницу в режиме **«Просмотр + действия»** по готовому мокапу: **`mockups/decision-detail.html`** (передать команде разработки как образец вёрстки и дизайн-системы);
  - компоненты страницы:
    - **слева** — дерево-навигация модуля (Кредиты → Программы и решения → Решение правительства);
    - **шапка** — хлебные крошки + номер решения;
    - **вкладки**: «Общая информация», «Кредитные программы», «Документы», «История»;
    - **Общая информация** — режим просмотра (подпись → значение, без редактируемых полей): Вид решения, Номер, Дата, Орган, Наименование, Примечание, «Действующее решение» (бейдж Да/Нет);
    - **Кредитные программы** — таблица связанных программ;
    - **Документы** — таблица вложений (увязать с R1);
    - **История** — таймлайн действий с иконками по типу (создание/изменение/загрузка/смена статуса), увязать с журналом аудита R2;
    - **футер** — «Изменить» (вход в режим редактирования) + «Назад»; в режиме редактирования поля становятся инпутами дизайн-системы, футер `OK / Отмена`;
  - мокап самодостаточный (HTML/CSS/JS, дизайн-система ASUBK gov-blue), отрисовку в продукте делать на стеке проекта по образцу мокапа.
- [ ] R7 🔴 Воркфлоу статусов решения — модель жизненного цикла
  - спецификация: **`mockups/decision-workflow.html`** (статусы, переходы, роли, матрица прав, поведение тулбара) — передать команде разработки;
  - сейчас в системе 3 статуса («На рассмотрении», «Одобрен», «Закрыт»); модель добавляет недостающие и переименовывает:
    - статусы (6): Черновик → На рассмотрении → На доработке → **Действует** → Отклонён → Закрыт;
    - переходы (10): Создать/Отправить/Отозвать/Одобрить(→Действует)/Отклонить/Вернуть/Отправить повторно/Закрыть/Удалить/Дублировать;
    - роли: Автор, Согласующий (+ опц. Наблюдатель);
    - матрица прав роль×статус; тулбар: **скрывать по роли, гасить по статусу** (с подсказкой на неактивных);
  - правила: редактирование только в «Черновик»/«На доработке»; удаление только «Черновик»; «Действует» только закрывается; «Закрыт» достижим только из «Действует»;
  - **расхождения, которые надо снять при реализации (увязать с R1, R2):**
    - терминология: «Одобрен» → «Действует» — синхронизировать с R1 (обязательность файла теперь на переходе в «Действует»);
    - обратимость: выбрать одну модель — R2 предлагал возврат из «Закрыт» в «На рассмотрении» (админ); спека делает «Закрыт»/«Отклонён» терминальными, а обратимость — через «Дублировать» отклонённое в новый черновик;
    - именование: статус «Действует» vs. булев флаг «Действующее решение (Да/Нет)» из карточки — решить, заменяет ли статус флаг или это разные сущности;
    - журнал переходов (кто/когда/комментарий) — это тот же аудит, что в R2 и таймлайн «История» в R6; не дублировать, сделать единый журнал;
  - открытые вопросы (из спеки, требуют решения заказчика): обязательность комментария-причины при «Отклонить»/«Вернуть»; нужна ли роль «Наблюдатель»; нужно ли «Дублировать»; кто уполномочен закрывать решение.
