# Руководство — Управление доступом (меню, страницы, кнопки, поля, данные)

> Для кого: администраторы, настраивающие права в Кредитном модуле АСУБК.
> Основано на боевом тестовом стенде (admin), обследовано 2026-06-20 через
> `mcp__plugin_ecc_playwright` на `https://fkftest.okmot.kg/`.
> Приложение использует **штатную безопасность Jmix** — права через роли, а не
> флаги на пользователе.

---

## 1. Модель (прочитать первым)

Доступ **никогда** не задаётся прямо на пользователе. Вы создаёте **роли**, затем
назначаете роли пользователям. Три уровня:

| Уровень | Путь в меню | Что контролирует | На какой вопрос отвечает |
|---------|-------------|------------------|--------------------------|
| **Ресурсные роли** | Безопасность → Ресурсные роли (`/sec/resourcerolemodels`) | пункты меню, страницы, кнопки, CRUD сущностей, поля | *Что пользователь может делать?* |
| **Роли уровня строк** | Безопасность → Роли уровня строк (`/sec/rowlevelrolemodels`) | какие **записи** видны/доступны для правки | *Какие строки видит пользователь?* |
| **Пользователи (Users)** | Приложение → Users (`/users`) | учётные записи + назначение ролей | *Кому какие роли?* |

```
Пользователь ──(Назначения ролей)──> Ресурсная роль(и) → действия / экраны / сущности / поля
          └──────────────────────────> Роль уровня строк → JPQL-фильтр строк
```

> **Колонка «Источник».** Роль бывает:
> - **Аннотированный класс** — задана в Java-коде, **только для чтения** в UI
>   (напр. `Full Access`, `ui-minimal`, `report-run`). Здесь её не отредактировать.
> - **База данных** — создаётся/правится в этом UI. Именно ими вы управляете.
>
> Для управления доступом вы создаёте/правите роли с источником **База данных**.

---

## 2. Создание ресурсной роли

1. Безопасность → **Ресурсные роли**.
2. **Создать**.
3. Заполните шапку:
   - **Название** — человекочитаемое имя (напр. `Специалист кредитного отдела`).
   - **Код** — уникальный технический код (напр. `credit-specialist`). Обязателен.
   - **Область действия** — отметьте **UI** для веб-приложения; **API** — только
     если роль используется через REST.
   - **Описание** — необязательно.
4. Добавьте политики на вкладке **Ресурсные политики** (следующий раздел).
5. **OK** — сохранить.

Роль **без политик не даёт ничего** — см. «Подводные камни» (§7).

---

## 3. Что ограничиваем → какой тип политики

На вкладке **Ресурсные политики** нажмите **Создать** и выберите **Тип** политики.
Каждый тип нацелен на свою часть UI:

| Что нужно контролировать | **Тип** политики | **Ресурс** (что вписать) | **Действие** |
|--------------------------|------------------|--------------------------|--------------|
| **Пункт бокового меню** (виден ли пункт) | `menu` | id меню, напр. `Bank.list` | `Доступ` |
| **Страница / экран** (можно ли открыть) | `screen` | id экрана, напр. `Bank.list`, `Bank.detail` | `Доступ` |
| **Операции с данными** сущности (управляет кнопками CRUD) | `entity` | имя сущности, напр. `Bank` | `Чтение` / `Создание` / `Изменение` / `Удаление` |
| **Поле** на форме / колонка | `entityAttribute` | `Сущность.поле` или `Bank.*` для всех | `Изменение` / `Чтение` |
| **Кнопка / своё действие / функция** | `specific` | id специфичного права | `Доступ` |

### Как именно контролируются кнопки

Кнопки **разные** — сопоставляйте кнопку с нужной политикой:

- **Стандартные кнопки списка/формы — Создать, Изменить, Удалить.** Управляются
  политикой **`entity`** автоматически. Кнопка активна только при наличии
  соответствующего действия на сущности:
  - кнопка **Создать** ⇐ entity `Создание`
  - **Изменить** / сохранение ⇐ entity `Изменение`
  - кнопка **Удалить** ⇐ entity `Удаление`
  - список/форма видны ⇐ entity `Чтение` (+ политика `screen`)

  > Проверено на роли **банки**: у неё entity `Bank` со всеми четырьмя действиями
  > (Чтение/Создание/Изменение/Удаление), поэтому все её кнопки CRUD активны.

- **Нестандартные кнопки / не-CRUD действия** (напр. *Одобрить*, *Отклонить*,
  *Назначить на пользователей*, экспорт, доменное действие) управляются политикой
  **`specific`**. Разработчик регистрирует именованное «специфичное право»; вы
  выдаёте его, добавив политику `specific` с этим id. Без неё кнопка скрыта/неактивна.

  > Примечание: список id `specific` определяет команда разработки. Если нужную
  > кнопку не покрывает действие сущности, запросите у разработчиков id её
  > специфичного права и добавьте политику `specific`.

### Пример — выдать роли полный доступ к экрану **Bank**

Добавьте политики (именно так устроена штатная роль **банки**):

| Тип | Ресурс | Действие |
|-----|--------|----------|
| menu | `Bank.list` | Доступ |
| screen | `Bank.list` | Доступ |
| screen | `Bank.detail` | Доступ |
| entity | `Bank` | Чтение |
| entity | `Bank` | Создание |
| entity | `Bank` | Изменение |
| entity | `Bank` | Удаление |
| entityAttribute | `Bank.*` | Изменение |

Вариант «только чтение»: уберите Создание/Изменение/Удаление, оставьте `Чтение`
и поставьте `entityAttribute Bank.*` → `Чтение`. Кнопки Создать/Изменить/Удалить
тогда исчезнут у такого пользователя автоматически.

## 4. Переиспользование через дочерние роли

Вкладка **Дочерние роли** позволяет роли **наследовать** все политики других
ролей. Делайте небольшие узкие роли (напр. по одной на экран) и собирайте из них
должностную роль, вместо повторного ввода политик. Должностная роль может состоять
**только** из дочерних ролей, без собственных политик.

---

## 5. Ограничить, какие строки видит пользователь (уровень строк)

Ресурсные роли решают, *что* пользователь может читать `Bank`. Роли уровня строк
решают, *какие именно* записи `Bank`.

1. Безопасность → **Роли уровня строк** → **Создать**.
2. Шапка: **Название**, **Код**.
3. Вкладка **Политики уровня строк** → **Создать**:
   - **Тип**: `JPQL` (предикат) — или тип, который даёт ваша команда разработки.
   - **Имя сущности**: напр. `Bank`.
   - **Действие**: `Read` (и/или другие).
   - **Выражение Where** (JPQL where) — фильтр, напр.
     `{E}.department.id = :current_user_department`. **Пустой Where не фильтрует
     ничего** (см. «Подводные камни»).
   - **Выражение Join** — необязательный доп. JPQL-join.
4. **OK**.

## 6. Назначение ролей пользователю

Два равнозначных пути:

- **Из списка ролей:** выберите роль → **Назначить на пользователей**.
- **Из пользователя:** Приложение → **Users** → выберите пользователя →
  **Назначения ролей** → добавьте ресурсные и/или строковые роли.

Изменения вступают в силу при следующем входе пользователя (или обновлении сессии).

> В списке Users каждый пользователь связан с **Сотрудник** → **Подразделение**.
> Используйте эту связь в JPQL уровня строк, чтобы ограничивать данные по подразделению.

## 7. Подводные камни (замечено на тестовом стенде — избегать)

- **Пустая роль = нет доступа.** Роль без политик не даёт ничего. На стенде роли
  `СУГС` и `Специалист кредитного отдела` **без политик** — пользователь только с
  ними видит пустое приложение. Всегда добавляйте политики (или дочерние роли).
- **Меню без экрана/сущности.** Если выдать только политику `menu`, пункт виден,
  но страница не откроется без `screen` + entity `Чтение`. Выдавайте набор вместе
  (см. пример в §3).
- **Забытый атрибут сущности.** Форма откроется, но поля будут пустыми/только для
  чтения без `entityAttribute`. Добавьте `Сущность.*` (или конкретные поля).
- **Пустой Where уровня строк ничего не делает.** Политика уровня строк с пустым
  **Выражением Where** не фильтрует строки (ошибка роли `test` на стенде).
- **Не редактируйте роли с источником `Аннотированный класс`** — они принадлежат
  коду; изменения вносятся в исходники приложения, а не здесь.
- **Важна область действия.** Если роль используется в веб-UI, должна быть отмечена
  **Область действия → UI**, иначе политики там не применяются.

---

## 8. Краткий чек-лист

- [ ] Роль с источником **База данных** (редактируемая).
- [ ] Задан **Код**, отмечена **Область действия = UI**.
- [ ] Политики `menu` + `screen` для каждой страницы, куда пускаем пользователя.
- [ ] entity `Чтение` (+ Создание/Изменение/Удаление по необходимости) для каждой сущности.
- [ ] `entityAttribute` для полей (используйте `Сущность.*` для всех).
- [ ] Политики `specific` для нестандартных кнопок/действий.
- [ ] (Опц.) роль уровня строк с **непустым** Where.
- [ ] Роль назначена пользователю; пользователь перезашёл.
