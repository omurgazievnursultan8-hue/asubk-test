## Детально по задачам

### R1 — Вложение документа решения ✅
Появилась вкладка **«Документы»** (и в форме создания, и в карточке просмотра).
Грид с колонками, точно по спецификации:
`Файл · Основной · Тип · Версия · Размер · Дата загрузки · Кто загрузил`.
Кнопки: **«Загрузить PDF»** (формат PDF в подписи), **«Открыть»**,
**«Сделать основным»** (флаг «основной»), **«Удалить»**.
Покрыты: множественность файлов, флаг основного, версионность, аудит загрузки.

**Не проверено автотестом** (требует ручной проверки): фактическая загрузка PDF,
лимит размера (≈10–20 МБ), отклонение не-PDF, обязательность основного документа
при переходе в «Действует».

### R2 — Отклонение: причина, аудит, обратимость ✅
- **Диалог** «Отклонение решения» с полем **«Причина отклонения»**, поле
  **обязательное** (при пустом — «Поле является обязательным»). Молчаливого
  отклонения (старый дефект **P1-03**) больше нет.
- **Аудит** — вкладка «История», грид:
  `Дата и время · Пользователь · Действие · Старый статус · Новый статус · Причина / комментарий · Файл`.
  Проверено на живой записи (E5): зафиксированы 3 перехода —
  `Создание решения → Черновик`, `Отправка на рассмотрение: Черновик → На рассмотрении`,
  `Отклонение: На рассмотрении → Отклонён` с причиной «QA автотест: проверка причины отклонения».
  Причина **сохраняется** и в журнале, и в колонке списка «Последняя причина / комментарий».
- **Обратимость** — кнопка **«Вернуть на рассмотрение»** активна у статусов
  «Отклонён» и «Закрыт» (модель админ-возврата из R7).

### R3 — Код генерируется системой ✅
Поле **«Код»** теперь **read-only** (снимает и косметический дефект P1-01).
При создании записи (Вид решения = «Основное», дата сегодня) система присвоила
код **`ОСН-2026-0001`** — формат `{ТИП}-{ГГГГ}-{NNNN}` соблюдён
(ОСН = тип, 2026 = год, 0001 = сквозной номер). Ручной ввод убран.
Старые записи с legacy-кодами («3», «1111», «№123 о-к-к») оставлены как есть — ок.

### R4 — Удаление: защита и причина ✅
- «Удалить» **гаснет** у статусов «Действует» и «Отклонён», **активна** у «Черновик»
  (проверено по тулбару). Закрывает дефект **P1-05** (раньше «Удалить» был активен
  всегда).
- Клик «Удалить» открывает диалог **«Удаление решения»** с **обязательной**
  «Причиной удаления» — не голый `confirm()` (по мокапу).

**Не проверено**: серверная проверка ссылок (сообщение «Нельзя удалить: на решение
ссылаются N программ») и фактический soft-delete — удаление не коммитил, чтобы не
портить данные.

### R5 — Валидация «Даты решения» ✅ (клиент)
`vaadin-date-picker` отдаёт атрибут **`max = 2026-06-30`** (= сегодня): будущие даты
в календаре заблокированы. Значение по умолчанию = сегодня. Закрывает P1-04/P1-11
на стороне клиента.

**Не проверено**: серверная валидация (отклонение будущей даты через API в обход UI).

### R6 — Страница деталей по мокапу ✅
Просмотр открывается отдельным режимом **`/gov-decisions/{id}?mode=readonly`**.
4 вкладки — точно как в мокапе `mockups/decision/decisions.html`:
**Общая информация · Кредитные программы · Документы · История**.
В режиме просмотра поля нередактируемые (0 редактируемых инпутов — чистый «view»).
Вкладка «Кредитные программы» — грид связанных программ (`Наименование · Статус · Дата решения`).

### R7 — Воркфлоу статусов ✅
- **Статусы** в системе: Черновик, На рассмотрении, Действует, Отклонён (+ Закрыт,
  На доработке как цели переходов). Старая 3-статусная модель заменена.
- **Тулбар — 11 действий**: Создать · Изменить · Просмотр · Удалить · Отправить ·
  Отозвать · Одобрить · Отклонить · Вернуть на доработку · Закрыть · Вернуть на рассмотрение.
  Цветовое кодирование: Создать/Одобрить — синий/зелёный, Отклонить — красный.
- **Гашение по статусу** проверено:
  - **Черновик**: активны Изменить/Просмотр/Удалить/Отправить; остальные гаснут.
  - **На рассмотрении**: активны Одобрить/Отклонить/Отозвать/Вернуть на доработку;
    Изменить/Удалить гаснут (модель maker/checker).
  - **Действует**: активна только Закрыть; Изменить/Удалить гаснут.
  - **Отклонён**: активна «Вернуть на рассмотрение».
- Переход **Отправить** (Черновик → На рассмотрении) отработал вживую (тост
  «Решение отправлено на рассмотрение»).
- Флаг **«Действующее решение (Да/Нет)»** — отдельная колонка, не выводится из
  статуса (E5 = «Нет» при «Отклонён», legacy «тест» = «Да»). Соответствует решению
  по R7 (флаг и статус — разные сущности).

### P1-12 — список показывал только «Одобрен» ✅ исправлено
Список теперь возвращает **17 строк всех статусов** (видны Черновик, На рассмотрении,
Действует, Отклонён). Записи в работе доступны, ряд-действия согласования работают.
Остаётся сохранённый фильтр «фильтр по статусу» с **пустым** значением (показывает
всё). Технически ок, но пользователь может выбрать значение и снова сузить выдачу —
см. замечание ниже.

## Отклонения от мокапа и замечания (мелкие)

1. **R2/R4 — кнопка подтверждения не гаснет при пустой причине.** В мокапе кнопка
   «Отклонить»/«Удалить» была *disabled* пока причина пуста. В продукте кнопка «OK»
   активна, а пустая причина блокируется **валидацией при отправке** (красное «Поле
   является обязательным»). Функционально эквивалентно — без причины действие не
   проходит. 🔵 косметика.
2. **P1-12 — сохранённый фильтр по статусу остаётся** (с пустым значением). Лучше
   убрать предустановленный фильтр или сделать дефолт «все, кроме удалённых», чтобы
   очередь согласования не пряталась случайным выбором. 🔵 на усмотрение.

## Что осталось проверить вручную (автотест не покрыл)
- **R1**: загрузка реального PDF; лимит размера; отклонение не-PDF; обязательность
  основного документа для перехода в «Действует»; замена файла = новая версия.
- **R2**: ограничение по роли (тестировалось под `admin`); обязательность причины
  при «Вернуть на доработку».
- **R4**: серверная защита ссылок (сообщение про N программ) и фактический soft-delete.
- **R5**: серверная валидация будущей даты (в обход клиента).
- **R7**: переходы Одобрить (→Действует), Закрыть (→Закрыт), Отозвать,
  Вернуть на доработку, оба админ-возврата — по отдельности не прогонялись
  (кнопки присутствуют и корректно гасятся).
