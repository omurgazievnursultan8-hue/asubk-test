## 1. Назначение

Кредитная программа — **шаблон кредитного продукта**, определяющий все параметры,
на основе которых создаются кредитные заявки и кредиты. Программа ссылается ровно на
одно Решение правительства (раздел 01) — причём только на решение в статусе
**«Действует»** (P2-R9) — и является родительской сущностью для кредитных заявок
(раздел 03) и кредитов (раздел 05).

Программа задаёт: наименование и правовое основание, назначение и вид кредита, сроки
действия, параметры суммы и срока кредита (фиксированные или диапазонные), процентную
и штрафные ставки, льготные периоды, параметры графика платежей, перечни документов и
требования к залоговому обеспечению.

Запись проходит управляемый жизненный цикл согласования: создаётся автором в статусе
«Черновик», отправляется на согласование, **другой** пользователь (согласующий)
одобряет / отклоняет / возвращает на доработку — самосогласование запрещено (принцип
«4 глаз», P2-R5). Каждый переход статуса фиксируется в журнале аудита.

## 3. Поля

Поля сгруппированы по вкладкам мастера создания. Обозначения типов:

- `text` — однострочное текстовое поле (`vaadin-text-field`)
- `textarea` — многострочное текстовое поле (`vaadin-text-area`)
- `date` — поле выбора даты (`vaadin-date-picker`)
- `select` — выпадающий список (`vaadin-select`); значения перечислены в §5.7
- `lookup` — поле выбора из справочника (`jmix-value-picker`)
- `multi-combo` — комбо с множественным выбором (`vaadin-multi-select-combo-box`)
- `int` — целочисленное поле (`vaadin-integer-field`)
- `decimal` — поле с десятичным числом (`vaadin-big-decimal-field`)
- `checkbox` — флажок (`vaadin-checkbox`)
- `grid` — редактируемая таблица с кнопками «Добавить» / «Удалить» / «Исключить»
- `readonly` — поле только для чтения

### 3.1 Визуальная семантика полей (P2-R1)

Все поля мастера показываются в **трёх визуально различимых состояниях** — сразу
при открытии формы, а не только после неудачной валидации:

| Состояние | Вид | Примеры |
|---|---|---|
| **Обязательное-редактируемое** | белый фон + рамка + красный `*` у подписи | Наименование на русском, Решение правительства |
| **Опциональное-редактируемое** | белый фон + рамка, без `*` | Наименование на английском, Комментарий |
| **Только для чтения / вычисляемое** | серый фон, без рамки | «Валюта» на вкл. 2, «Выбранные шаблоны» (вкл. 7), вся вкл. 9 «Предпросмотр» |

> Единый компонент-стиль дизайн-системы ASUBK, общий с Фазой 1 (R6). Применяется ко
> всем 9 вкладкам. Обязательность по вкладкам определена в §5.1.

Колонка «Обяз.» в таблицах ниже отражает **целевой** набор обязательных полей
(P2-R10): **Да** — обязательно для отправки на согласование; **Условно** —
обязательно при включении соответствующего блока (льгота / залог); **Нет** —
необязательное; **readonly** — выводится системой.

## 4. Страница деталей (P2-R4)

Режим **«Просмотр + действия»**: данные показываются как «подпись → значение», без
редактируемых инпутов; редактирование — отдельным режимом по кнопке «Изменить».

- **Слева** — дерево-навигация модуля (Кредиты → Программы и решения → Кредитные программы).
- **Шапка** — хлебные крошки + наименование программы + бейдж статуса.
- **Вкладки просмотра** — те же 9 разделов мастера в режиме чтения + вкладки
  «Заявки» (downstream-ссылки на раздел 03) и «История» (журнал аудита).
- **Футер** — действия жизненного цикла (по роли и статусу, §6/§7) + «Изменить» + «Назад».

Открытие: кнопка **«Просмотр»** в тулбаре (выбрать строку → Просмотр) **и двойной
клик по строке**. Действия жизненного цикла доступны **и в списке, и на странице
деталей**.

## 5. Бизнес-правила и расчёты

### 5.1 Прогрессивная валидация мастера (P2-R10)

**Двухуровневая валидация:**

- **«Сохранить» (черновик)** — проверяет только вкладки 0..текущую; дальше можно
  оставить незаполненным. Запись сохраняется в статусе «Черновик».
- **«Отправить на согласование»** — валидирует **все** вкладки целиком; при наличии
  незаполненных обязательных полей отправка отклоняется.

**Прогрессивный гейт навигации:** переход «Далее» / клик по вкладке вперёд возможен
только если обязательные поля текущей и предыдущих вкладок заполнены (иначе — прыжок
на первую незаполненную вкладку + подсветка). Навигация назад — свободна.

**UX ошибок (P2-R1):** вкладки с невалидными полями помечаются красным индикатором
со счётчиком ошибок; под каждым невалидным полем — inline-текст «Обязательное поле»
(не только красная рамка).

Валидация выполняется **на клиенте и на сервере**; серверная проверка — обязательное
условие отправки и одобрения (UI-гейту не доверять).

Сводный целевой набор обязательных полей по вкладкам:

| Вкладка | Обязательные поля |
|---|---|
| 1 Основные | 11 полей (см. §3.1, итог вкл. 1) |
| 2 Сумма и срок | тип суммы + тип срока + ≥1 значение каждого (или мин+макс при «Диапазон») |
| 3 Ставки | тип ставки + значение (≥1 из справочника / мин+макс) |
| 4 Штрафы | тип штрафа осн. + значение, макс. размер %, тип штрафа проц. + значение |
| 5 Льготный период | условно: при включённом блоке — тип + продолжительность |
| 6 Платежи | периодичность, день платежа, метод расчёта дней, метод погашения, тип расчёта процентов, очередь погашения (+ месяцы при Ежекв./Ежегодно) |
| 7/8 Залог | условно: при включённом гейте — ≥1 вид залога + требования ру/кырг |

### 5.2 Параметры суммы кредита

- **Тип «Фиксированная»**: грид «Фиксированные суммы» — обязателен ≥1 элемент.
- **Тип «Диапазон»**: грид скрывается, появляются обязательные
  **«Минимальная сумма»** и **«Максимальная сумма»** (`vaadin-big-decimal-field`);
  правило `Макс ≥ Мин`.
- Валюта наследуется с вкл. 1 (readonly).

### 5.3 Параметры срока кредита

- **Тип «Фиксированная»**: грид «Фиксированные сроки» — обязателен ≥1 элемент.
- **Тип «Диапазон»**: грид скрывается, появляются обязательные
  **«Минимальный срок»** и **«Максимальный срок»** (`vaadin-integer-field`);
  правило `Макс ≥ Мин`.

### 5.4 Параметры процентной ставки

- **Тип «Фиксированная»**: ≥1 значение из регламентированного справочника ставок.
- **Тип «Диапазон»**: появляются обязательные **«Значение ставки с»** и
  **«Значение ставки по»**; правило `по ≥ с`.
- Флаг «Использовать плавающую ставку» — опциональная заготовка.

### 5.5 Параметры штрафных ставок

Все четыре поля обязательны (P2-R10): тип + значение штрафа по основной сумме;
максимальный размер штрафа по процентам (% от суммы); тип + значение штрафа по
процентам. Числа — `vaadin-big-decimal-field`.

### 5.6 Льготный период

Три независимых флага. При включении флага блок становится условно-обязательным:
требуются тип и продолжительность льготы (P2-R10). При выключенном флаге блок
исключается из валидации.

### 5.7 Перечисляемые значения select-полей

| Поле | Значения |
|---|---|
| Тип суммы / срока / основной ставки / штрафной ставки (осн. и проц.) | Фиксированная · Диапазон |
| Периодичность платежей | Ежемесячно · Ежеквартально · Ежегодно |
| Обработка выходных дней | Переносить на следующий рабочий день · Переносить на предыдущий рабочий день |
| Метод погашения кредита | Аннуитетный · Дифференцированный |

### 5.8 Валидация дат (P2-R2)

- **«Дата по» ≥ «Дата с»** (равенство допускается — программа на 1 день): на клиенте
  (пикер «Дата по» не даёт выбрать раньше «Даты с») и на сервере.
- **«Дата с» ≥ «Дата решения»** связанного решения правительства — нижняя граница
  (у решения есть только дата принятия, поэтому она и есть нижний порог).
- Будущие даты для программы **допустимы** (в отличие от решения, Фаза 1 R5) — это
  проверка порядка/согласованности, а не запрет будущего.
- При нарушении — внятное сообщение, а не молчаливый сброс.

### 5.9 Выбор решения правительства — серверный enforcement (P2-R9)

- Связанное решение должно быть в статусе **«Действует»**; иначе сохранение
  отклоняется на сервере с внятным сообщением (не полагаться на UI-фильтр).
- В picker'е неодобренные решения **скрываются жёстким фильтром загрузчика**
  (а не снимаемым дефолтным чипом).
- Picker ограничен **режимом выбора** — без действий Создать/Изменить/Удалить/
  Одобрить/Отклонить из формы программы.

### 5.10 Удаление программы (P2-R7 / P2-R8)

- Серверная защита ссылок: удаление запрещено, если на программу ссылается хотя бы
  одна заявка или кредит. Перехват `DeletePolicyException` → **общее** локализованное
  сообщение («Нельзя удалить: на программу есть ссылки»), без сырого стека (P2-R7).
- **Баг к исправлению (P2-R8):** программа без реальных downstream-ссылок должна
  удаляться; текущая инвертированная политика винит «references from GovDecision»
  (программа ссылается на решение, а не наоборот) — проверить направление связи
  `LoanProgram ↔ GovDecision` и delete-policy.
- Удаление доступно только в статусе «Черновик»; soft-delete + запись в журнал
  (по образцу Фазы 1 R4).

### 5.11 Семантика кнопок сохранения (P2-R3)

Единый язык подписей: **«Сохранить»** (сохранить и остаться) ·
**«Сохранить и закрыть»** · **«Отмена»**. После «Сохранить и закрыть» переход — в
**список программ** (не на главную `/`, как в as-is).

## 6. Функции и действия

### 6.1 Действия жизненного цикла (P2-R5)

Доступны из списка и со страницы деталей. Видимость — по роли; доступность — по статусу.

| Действие | Из статуса | В статус | Роль |
|---|---|---|---|
| Создать | — | Черновик | Автор |
| Отправить на согласование | Черновик | На рассмотрении | Автор |
| Отозвать | На рассмотрении | Черновик | Автор |
| Вернуть на доработку | На рассмотрении | На доработке | Согласующий |
| Отправить повторно | На доработке | На рассмотрении | Автор |
| Удалить | Черновик | (удалено, soft-delete) | Автор |
| Одобрить | На рассмотрении | Действует | Согласующий |
| Отклонить | На рассмотрении | Отклонена | Согласующий |
| Закрыть | Действует | Закрыта | Согласующий / уполномоченный |
| Вернуть на согласование | Отклонена | На рассмотрении | Администратор |
| Вернуть на согласование | Закрыта | На рассмотрении | Администратор |

> Самосогласование запрещено: пользователь, отправивший программу, не может её
> одобрить — требуется **другой** пользователь (4 глаза, P2-R5).
> Два «Вернуть на согласование» — служебный админ-возврат из терминальных статусов
> (единый паттерн с Фазой 1).

### 6.2 Подтверждения и причины

- **Отклонить**, **Закрыть**, **Вернуть на доработку** и **Вернуть на согласование**
  (админ) — через диалог подтверждения с **обязательной** причиной; причина видна в
  просмотре и пишется в журнал аудита (по образцу Фазы 1 R2).

### 6.3 Кнопки формы

| Кнопка | Описание |
|---|---|
| **Сохранить** | Сохранить и остаться в форме |
| **Сохранить и закрыть** | Сохранить и вернуться в список программ (P2-R3) |
| **Отмена** | Закрыть без сохранения |
| **Отправить на согласование** | Только на вкл. 9; валидирует все вкладки (§5.1) |

### 6.4 Фильтрация (список)

- **Добавить условие поиска** — конструктор фильтра.
- Бейдж статуса в строке списка (P1-08).
- Пагинация, настройки столбцов — стандартные Jmix-компоненты.

## 7. Роли и матрица прав (P2-R5)

| Роль | Назначение |
|---|---|
| **Автор** | Создаёт и редактирует программы, отправляет на согласование, отзывает, дорабатывает, удаляет черновик |
| **Согласующий** | Одобряет / отклоняет / возвращает на доработку / закрывает программу; не может одобрять собственную отправку (4 глаза) |
| **Администратор** | Служебный возврат «Отклонена» / «Закрыта» → «На рассмотрении» с причиной в журнал |

**Матрица прав** (по статусу):

| Статус | Автор | Согласующий |
|---|---|---|
| Черновик | Изменить · Удалить · Отправить на согласование | только просмотр |
| На рассмотрении | Отозвать | Одобрить · Отклонить · Вернуть на доработку |
| На доработке | Изменить · Отправить повторно | только просмотр |
| Действует | только просмотр | Закрыть |
| Отклонена | только просмотр | только просмотр |
| Закрыта | только просмотр | только просмотр |

> Тулбар: скрывать кнопки по роли, гасить по статусу. Единый паттерн жизненного
> цикла с Фазой 1 (R7). Возврат из «Отклонена» / «Закрыта» — вне матрицы
> Автор/Согласующий, выполняется Администратором.

## 8. Статусная модель (P2-R5)

```
        ┌───────────┐  Создать
        │ Черновик  │◄────────────── (новая запись)
        └─────┬─────┘
 Отправить     │   ▲  Отозвать
               ▼   │
        ┌──────────────────┐
        │ На рассмотрении  │◄──── Отправить повторно ──┐
        └──┬────┬─────┬────┘                           │
Одобрить    │    │     │  Вернуть                ┌──────┴───────┐
            ▼    │     └────────────────────────►│ На доработке │
       ┌──────────┐│ Отклонить                   └──────────────┘
       │ Действует ││───────────► ┌───────────┐
       └─────┬─────┘              │ Отклонена │
  Закрыть    │                    └─────┬─────┘
             ▼                          │ Вернуть на согласование (админ)
       ┌──────────┐                     │
       │  Закрыта │─────────────────────┘
       └────┬─────┘  Вернуть на согласование (админ) → На рассмотрении
            └─────────────────────────────────────►
```

| Статус | Описание | Доступные действия |
|---|---|---|
| Черновик | Создаётся и редактируется автором, ещё не отправлено | Изменить, Удалить, Отправить на согласование |
| На рассмотрении | Отправлено согласующему, ожидает решения | Отозвать; Одобрить, Отклонить, Вернуть на доработку |
| На доработке | Возвращено автору с замечаниями | Изменить, Отправить повторно |
| Действует | Утверждена, по программе можно создавать заявки/кредиты | Закрыть; просмотр |
| Отклонена | Отказано согласующим (из «На рассмотрении», с причиной) | Возврат на согласование (админ); просмотр |
| Закрыта | Действовавшая программа отменена/утратила силу (из «Действует») | Возврат на согласование (админ); просмотр |

> Соответствует статусной модели решения (Фаза 1 R7) с поправкой на род терминов
> программы («Отклонена» / «Закрыта»). Бейджи мокапа
> `mockups/loan-program/loan-program.html` (Черновик / На рассмотрении / Одобрена /
> Отклонена / Закрыта) подлежат правке: «Одобрена» → «Действует», добавляется
> «На доработке» — унификация с Фазой 1 (P2-R5).

## 9. Бизнес-правила (сводка)

1. **Обязательные поля** — целевой набор по вкладкам (§5.1, P2-R10); двухуровневая
   валидация: черновик проверяет вкладки 0..текущую, отправка — все.
2. **Начальный статус** — «Черновик»; статус меняется только действиями
   жизненного цикла (§6.1), не ручной правкой формы.
3. **Редактирование** возможно только в статусах «Черновик» / «На доработке».
4. **Самосогласование запрещено** (4 глаза, P2-R5).
5. **Связанное решение** — только в статусе «Действует», серверный enforcement (§5.9).
6. **Даты** — `Дата по ≥ Дата с ≥ Дата решения`; будущие даты допустимы (§5.8).
7. **Удаление** — только «Черновик», soft-delete, серверная защита ссылок,
   локализованное сообщение (§5.10).
8. **Отклонение / закрытие** — подтверждение + обязательная причина + аудит (§6.2).
9. **Журнал аудита** — каждый переход статуса, отклонение, закрытие, удаление
   фиксируются (кто, когда, действие, причина); отображаются на вкладке «История».

## 10. Открытые вопросы (требуют решения заказчика)

1. **Кто уполномочен закрывать** программу («Закрыть»: Согласующий / отдельная
   уполномоченная роль).
2. **Роль «Наблюдатель»** (только просмотр) — вводить ли.
3. **Продолжительность льготного периода** на уровне программы vs транша: где
   хранится «источник истины» (программа задаёт рамку, транш — факт; §5.6, раздел 06).

> Решено (унификация с Фазой 1, P2-R5, 2026-06-24): статус «Действует» вместо
> «Одобрена»; роль «Согласующий»; переход «Отправить на согласование»; добавлены
> «На доработке» (возврат согласующим автору) и обратимость «Отклонена» / «Закрыта»
> → «На рассмотрении» через Администратора. Прежние открытые вопросы об обратимости
> и о статусе «На доработке» сняты.
