1. Назначение и границы

1.1 Назначение модуля «Заявка»

Заявка — центральная сущность фазы 3. Собирает заёмщика, условия кредита, документы, залог и (для групповых программ) состав группы, проводит их через проверку и коллегиальное решение и после одобрения порождает кредит(ы). В целевой модели заявка — управляемый конечный автомат (статус-степпер), а не свободно редактируемая карточка.

Комиссия по заявкам — коллегиальный орган и модуль (commission.html), где ведётся голосование и выносится единственное итоговое решение по заявке (кредит + залог вместе), протоколируемое кворум-гейтом.

1.2 Два потока происхождения заявки (спека «Два потока заявок»)

  • Поток A — по программе: вход из карточки/списка кредитной программы → программа зафиксирована и наследуется в заявку конвертом условий; после одобрения кредит формируется сразу.
  • Поток B — индивидуальная: вход «Создать индивидуальную заявку» (гейт по роли) → программы нет, условия свободные; после одобрения кредит не формируется — заявка переходит в «Ожидает решения/программы», ждёт привязки нового решения + программы, затем идёт как поток A. Признак потока — app.flow (мокап).

1.3 Ключевые отличия от as-is

  1. Двух-комиссионная модель отменена (заказчик 2026-07-06; P3-R4 #4 / P3-R26): одна комиссия судит и кредит, и залог — один голос, одно решение, один протокол. Убраны колонка «Статус залога», вторая кнопка «Отправить в залоговую комиссию», параллельные статусы. Залог — не отдельный статус, а гейт отправки в комиссию (коэффициент покрытия + обязательные залоговые документы).
  2. Статус-степпер управляемый (P3-R22): переходы только через действия ЖЦ с аудитом, не свободная правка статуса; редактируемость полей гейтится статусом (P3-R10).
  3. Воркфлоу комплектации (мокап; спека 2026-07-10): документы → подтверждение головного филиала → заключения отделов → комиссия (предв.) → комитет — 5-узловой пайплайн (appStage), ортогональный статусу.
  4. Единый движок решения живёт в модуле «Комиссии по заявкам» (commission.html); в карточке заявки фаза «Заседание комитета» — read-only зеркало (модель B, P3-R47).

1.4 Границы (что НЕ входит)

  • Формирование кредита/траншей, график выданного кредита, кураторы — модуль «Кредит» (условия наследуются при выдаче).
  • Реестр субъектов/заёмщиков, справочники залога/областей/отделов — внешние модули (в заявке — пикеры и справочники-источники).
  • Печатные формы и шаблоны договоров — поздняя стадия, отдельные задачи.

2. Роли и фазы (сквозная модель доступа)

2.1 Роли (ROLES, мокап)

Модель расширена до 4 рабочих ролей + наблюдатель. Роли gf/dept — надстройка комплектации (спека 2026-07-10 §6), в закрытом графе Топика 4 их ещё нет. Роль в проде берётся из auth (Jmix security); в мокапе — переключатель setRole (сбрасывает _editMode, откатывает несохранённое _revertEdit, перерисовывает деталь). Не сверено на live.

Ключ Роль Зона ответственности
spec Кредитный специалист создаёт/правит заявку в черновике, все действия тулбара, назначает отделы на заключения
gf Головной филиал 2-й уровень комплектации: подтверждает укомплектованность документов (confirmDoc)
dept Отдел (заключение) вносит заключение только за свой отдел (_deptKey), после подтверждения ГФ
com Комиссия правит опер-условия и голосует на «На рассмотрении»
ro Остальные только чтение

_deptKey (setDept) выбирает, от чьего имени действует роль «Отдел».

2.2 Фазы (condPhase(status), мокап)

Фаза Статусы Кто правит меняемые условия
draft Новый · Требуется доп. информация Кредитный специалист
review На рассмотрении · Ожидает решения/программы Кредитная комиссия (экран специалиста read-only)
locked Одобрена · Отклонена · Отозвана · На оформлении · Оформлена Никто (зафиксировано)

2.3 Матрица прав can(app) (роль × фаза → право)

Каждое право = потенциальное (роль × фаза) И включённый режим «Изменить» (_editMode) там, где помечено (edit). Пусто = не даётся.

Право Кому/когда spec·draft com·review gf·draft dept·draft com·locked
editReq (edit) заявитель, документы, члены, залог ✅
editOp (edit) опер-условия + комментарий комиссии ✅
editSched (edit) окно правки графика (P3-R13) ✅ ✅
vote голос члена комиссии ✅
confirmDoc ГФ подтверждает укомплектованность ✅
assignDepts spec назначает отделы ✅
editConcl(dept) dept вносит заключение (после confirmedMet) ✅ (свой)
withdrawConcl(dept) dept отзывает своё submitted-заключение ✅ (свой)
toolbar действия тулбара заявки (§3.4) ✅ (любая фаза)

Правила гарда (мокап): - rawReq = spec && draft; rawOp = com && review; rawSched = (spec && draft) || (com && locked); фактические editReq/editOp/editSched = потенциальное И _editMode. - Роль/отдел фиксируются снимком в момент вызова can() (curRole/curDept) — замыкания editConcl/withdrawConcl не «уезжают» при смене роли. - editConcl/withdrawConcl дополнительно требуют _docStats(app).confirmedMet.

2.4 Иммутабельные поля (спека Топик 4; P3-R10)

Кредитный специалист не меняет никогда, даже в draft: - Субъект (заявитель) — фиксируется при создании; - Поля, изменённые комиссией — 3-й слой diff-провенанса заблокирован для специалиста.

По статусу: {Новый, Требуется доп. информация} → правится по линзе наследования конверта; все прочие статусы → полный read-only (гард полей P3-R10).

3.1 Перечень статусов заявки (STATUS_META, одна дорожка)

Нормализация (STATUS_ALIAS): legacy-значения приводятся к каноническим при загрузке (r.status = STATUS_ALIAS[r.status] || r.status): Одобрено → Одобрена, Отклонено → Отклонена. Правило: новые синонимы добавлять в STATUS_ALIAS, не в бизнес-логику; сравнения по статусу — только по каноническим строкам. Инвариант касается и статкарточек (§4.7): считать после алиаса, иначе статус разъедется на две плитки.

Права can(rec) (роль × статус): vote — член состава по нерешённой комиссии, ещё не голосовавший; decide — только председатель при pending (под гейтом кворума); protocol — председатель/секретарь при pending; manage — секретарь.

Инвариант (критично): UI-гейт без серверной проверки кворума — не защита. Отдельной задачей подтвердить, что сервер валидирует кворум перед сменой «Тип решения».

Эволюция правила (P3-R4 #3 ↔ P3-R38): as-is (P3-R4 #3, live 2026-06-19) — кворум без порога (председатель закрывает в любой момент, голоса справочно). To-be ядро P3-R38 вводит кворум-гейт ⌊n/2⌋+1. Осознанное ужесточение — при реализации согласовать каноническое правило с заказчиком.

9.6 Вкладка «Комиссия» карточки заявки (renderCommission, tab-6)

Строка-пайплайн стадий над контентом: «1. Предварительное изучение → 2. Заседание комитета · решение · протокол». Вход: commissionFor(app)==null (Новый/Отозвана) → баннер «Заявка ещё не отправлена в комиссию»; _comStateOf(app) держит проекцию (phase·votes·comments·final·scan·open).

  • Фаза A «Предварительное изучение» (движок в карточке): список обяз. документов (_reqDocList: осн. секции + документы членов + залоговые) со статус-бейджем; «Запросить замену» (comRequest → возврат документа специалисту → заявка «Требуется доп. информация»; только роль «Комиссия»); «Вынести на заседание комитета» (comAdvance, в общем тулбаре, решение 2026-07-10; гейт = комплект подтверждён ГФ И все заключения внесены). Гарды продублированы в модели (_comActor) — UI прячет кнопки, функции не верят рендеру.
  • Фаза B = read-only зеркало: движка нет; показывает прогресс, тэлли-чипы, грид состава (read-only), «Финальное решение» (бейдж), скан протокола; баннер направляет в модуль. По-членный чек-лист подписей убран (итог = голосование + один скан).
  • Расхождение зеркала (хвост): прогресс-полоса фазы B в карточке одноцветная, без кворум-строки — в отличие от commission.html (стек-сегменты + кворум). При реализации зеркало читает те же данные из записи комиссии; эталон прогресса — commission.html (P3-R39/40).
