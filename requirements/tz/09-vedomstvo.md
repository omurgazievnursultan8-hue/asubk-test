## 1. Назначение и границы

### 1.1 Назначение модуля

**«Информация о ведомстве»** — карточка самой организации-оператора (ОАО
«Госфинхолдинг»): кто мы, чем зарегистрированы, по каким счетам принимаем и
отправляем деньги, какими документами действуем, как с нами связаться.

Это **справочная витрина организации**, а не операционный экран: записи здесь
меняются редко, но на них ссылается всё остальное — платёжные поручения (счёт
организации), печатные формы договоров (реквизиты, наименование), публичные
контакты (антикоррупционная линия, обращения граждан), титульные листы отчётов.

### 1.2 Кто пользуется

| Роль | Что делает |
|---|---|
| **Администратор** | Единственный, кто правит собственные блоки карточки; видит вкладку «Журнал изменений» |
| **Наблюдатель** (любой сотрудник) | Читает карточку, копирует реквизиты счёта, печатает карточку организации, скачивает файлы документов |

### 1.3 Границы (что НЕ входит)

- **Оргструктура** — своя сущность и свой модуль; здесь только **зеркало**, режим
  чтения (§7).
- **Показатели** — считает модуль статистики; здесь только **зеркало
  зафиксированных срезов** (§8).
- **Пользователи и роли** — подсистема администрирования.
- **Сотрудники (персоналии)** — карточек сотрудников модуль не ведёт: у узла
  оргструктуры ФИО руководителя показывается как публичный атрибут узла, ссылки на
  чувствительную кадровую карточку нет.
- **Хранилище файлов** — вложения документов лежат в подсистеме документооборота;
  модуль хранит только ссылку и метаданные.

## 3. Модель данных

### 3.1 Сущность «Ведомство» — singleton (P10-R1)

Одна запись на систему. Кнопок «Создать» и «Удалить» **нет ни в каком виде**;
доступны только чтение и редактирование блоков.

Карточка складывается из **собственных блоков** (хранятся здесь) и **зеркал**
(хранятся в чужих модулях, здесь только читаются).

| Блок | Кардинальность | Владелец данных |
|---|---|---|
| Общая информация | 1:1 | свой |
| Регистрационные данные | 1:1 | свой |
| Хронология (события организации) | 1:N | свой |
| Банковские счета | 1:N | свой |
| Документы ведомства | 1:N | свой |
| Контакты (телефоны, e-mail, адреса, режим работы) | 1:N + 1:1 | свой |
| Оргструктура | — | **зеркало** (модуль оргструктуры) |
| Показатели | — | **зеркало** (модуль статистики) |
| Журнал изменений | — | **представление** (подсистема администрирования) |

### 3.2 Общая информация (1:1)

| Поле | Тип | Обязательно | Примечание |
|---|---|---|---|
| Полное наименование | текст | да | «Открытое акционерное общество «Государственный финансовый холдинг»» |
| Краткое наименование | текст | да | выводится в шапке, печатных формах |
| Краткое описание | текст (многострочный) | да | чем занимается организация |
| Логотип | файл (SVG/PNG) | нет | загружается пользователем; показывается в шапке и в печатной карточке |

**Двуязычие наименования (кыр/рус) не вводится** — только русский (решение
зафиксировано в мокапе, `ORG`).

### 3.3 Регистрационные данные (1:1)

| Поле | Формат | Валидатор (мокап `V`) |
|---|---|---|
| Номер свидетельства о регистрации | текст | — |
| Дата свидетельства | дата | `V.date` |
| Орган регистрации | **ссылка в справочник** `REG_ORGS` | P10-R3 |
| ИНН | 14 цифр | `V.inn` |
| ОКПО | 8 цифр | `V.okpo` |

### 3.4 Хронология (1:N)

Событие: **дата** + **заголовок** + **описание**. Сортировка по дате по убыванию.
Свободный список — используется в публичных материалах и справках («Учреждение
ОАО», «Переход на АСУБК», «Открытие Ошского филиала»). Единственный блок, где
удаление записи допустимо (событие — не финансовый факт), но и оно пишется в журнал.

### 3.5 Банковские счета (1:N) — см. §5

### 3.6 Документы ведомства (1:N) — см. §6

### 3.7 Контакты

| Блок | Кардинальность | Состав |
|---|---|---|
| Телефоны | 1:N | тип (**справочник**: приёмная, горячая линия, антикоррупционная линия, канцелярия, факс) + номер (`V.phone`, формат +996) + примечание |
| Электронная почта | 1:N | тип (**справочник**: общий, для обращений граждан, СЭД) + адрес (`V.email`) + примечание |
| Адрес сайта | 1:1 | `V.url` |
| Юридический адрес | 1:1 | текст |
| Фактический адрес | 1:1 | текст |
| Режим работы | 1:1 | текст |
| Часы приёма граждан | 1:1 | текст |

Телефон **антикоррупционной линии** — требование к госоргану: его отсутствие даёт

## 4. Справочники вместо свободного ввода (P10-R3 🔴)

Значения перечисленных полей **берутся из модуля справочников**, ввод текстом
запрещён:

| Поле | Справочник (мокап `REF`) |
|---|---|
| Банк счёта | `BANKS` — **вместе с БИК**: БИК принадлежит банку, а не нашему счёту, и отдельным полем не вводится |
| Валюта счёта | `CCY` |
| Назначение счёта | `ACCT_PURPOSE` (расчётный, спецсчёт бюджетных средств, валютный, депозитный…) |
| Орган регистрации | `REG_ORGS` |
| Тип телефона | `PHONE_TYPES` |
| Тип e-mail | `EMAIL_TYPES` |
| Тип документа | `DOC_TYPES` (устав, свидетельство, лицензия, разрешение, постановление) |

**Почему это блокер, а не удобство.** Free-text даёт расхождение написаний между
модулями («ОАО «РСК Банк»» / «РСК банк» / «RSK»), после чего ни один отчёт по
этому полю не сходится, а платёжка уходит с реквизитами, которых нет в справочнике
банков.

## 5. Банковские счета (P10-R2 🔴)

### 5.1 Состав записи

Банк (справочник, БИК подтягивается) · Номер счёта (16 цифр, `V.acct`) · Валюта
(справочник) · Назначение (справочник) · Признак «основной» · Дата открытия ·
Дата закрытия (пусто = действующий).

### 5.2 Append-only: счёт не удаляется, а закрывается

Удаления счёта **нет**. Счёт выводится из обращения простановкой **даты закрытия**.

**Почему.** Реквизит из платёжного поручения трёхлетней давности должен оставаться
разрешимым: если удалить счёт, старый платёж ссылается в пустоту, а печатная форма
договора перестаёт воспроизводиться.

### 5.3 Инвариант «основного» счёта

> **Не более одного «основного» счёта в разрезе валюты. Считается ТОЛЬКО по
> действующим счетам** (мокап `checkMainAccount`).

- Закрытый счёт основным быть не может: при закрытии флаг «основной» **снимается
  автоматически**.
- Нарушение инварианта — **блокирующая ошибка** сохранения, а не предупреждение;
  на вкладке выводится красный баннер с перечнем валют, где инвариант нарушен.
- Основной счёт по валюте — тот, что подставляется в платёжные документы по
  умолчанию; двух умолчаний быть не может.

### 5.4 UI вкладки «Реквизиты» (мокап `renderReqs`)

- Фильтр **«Действующие / Все»** (`state.acctFilter`) — по умолчанию действующие;
  закрытые счета показываются приглушённо с датой закрытия.
- Действие **«Копировать реквизиты»** — блок реквизитов счёта в буфер обмена
  (доступно и наблюдателю).
- Действия администратора: «Добавить счёт», «Изменить», «Закрыть» (вместо
  «Удалить»).

## 6. Документы ведомства (P10-R4 🟠)

### 6.1 Состав записи

Тип (справочник) · Наименование · Номер · Дата выдачи · Срок действия (может быть
пустым = бессрочный) · Выдавший орган · Файл (вложение подсистемы документооборота:
имя, размер, скачивание).

### 6.2 Статус — производный, не хранимый (мокап `docStatus`)

| Условие | Статус | Индикация |
|---|---|---|
| срок действия не задан | Бессрочный | нейтральный |
| до истечения > 90 дней | Действует до ДД.ММ.ГГГГ | зелёный |
| до истечения ≤ 90 дней | Истекает через N дн. | жёлтый |
| срок прошёл | Истёк ДД.ММ.ГГГГ | красный |

Статус **вычисляется на текущую дату** и в базе не хранится — иначе он протухает
ровно в тот день, когда становится важен.

### 6.3 Сводка по вкладке

Над списком — один индикатор состояния комплекта: сколько документов просрочено,
сколько истекает в ближайшие 90 дней, либо «все действительны». Просроченная
лицензия — это остановка кредитных операций, а не косметика.

## 7. Оргструктура — зеркало (P10-R5 🟠)

- Вкладка **только для чтения**. Редактирование — в модуле оргструктуры
  (`mockups/org-structure/org-structure.html`); здесь ссылка, а не копия данных.
- Источник **версионен**: узлы создаются, переименовываются, ликвидируются;
  руководители сменяются. Витрина показывает **срез на выбранную дату**
  (мокап `nodesAt` / `nameAt` / `headAt`), а не «текущее состояние».
- Глубина — **до подразделений, без персоналий**: ФИО руководителя — публичный
  атрибут узла, а не ссылка на кадровую карточку сотрудника.
- **Вакансия — валидное состояние** узла («— вакансия»), а не ошибка данных.
- На дату, когда организации ещё/уже не существовало в этом виде, показывается
  пустое состояние, а не сегодняшнее дерево.

## 8. Показатели — зеркало зафиксированных срезов (P10-R6 🟠)

### 8.1 Срезы, а не расчёт на лету

Агрегаты считает **модуль статистики батчем в нерабочее время** и складывает
срезами с датой расчёта (мокап `METRIC_SNAPSHOTS`). Витрина берёт **последний срез
на дату ≤ выбранной** и показывает дату расчёта. Считать «на лету» по живой базе
на этой странице запрещено — это витрина, а не отчётный движок.

### 8.2 Состав (только всеорганизационные агрегаты)

| Показатель | Единица | Направление «хорошо» |
|---|---|---|
| Объём портфеля | сом | рост |
| Выдано с начала года | сом | рост |
| Доля просрочки | % от портфеля | снижение |
| Количество заёмщиков | активных | рост |

Разрезов по подразделениям здесь нет — они относятся к модулю статистики.

### 8.3 Дельта к предыдущему срезу

- Сравнение — с **предыдущим срезом**, дата сравнения подписана.
- **Доля просрочки меняется в процентных пунктах (п.п.)**, а не в «процентах от
  процента» (мокап `deltaHtml`, `mode:'pp'`) — 4,5 % → 4,2 % это «↓ 0,3 п.п.», а не
  «↓ 6,7 %».
- Направление окрашивается по смыслу показателя: рост портфеля — хорошо, рост
  просрочки — плохо.
- Первый зафиксированный срез сравнивать не с чем — так и пишем, а не рисуем 0 %.

## 9. Валидация и полнота карточки

### 9.1 Валидаторы форматов (P10-R8 🟡, мокап `V`)

| Поле | Правило |
|---|---|
| ИНН | 14 цифр |
| ОКПО | 8 цифр |
| БИК | 6 цифр (приходит из справочника банков) |
| Номер счёта | 16 цифр |
| E-mail | стандартный формат |
| Адрес сайта | http/https |
| Телефон | +996, далее цифры/скобки/дефисы |

Проверка **жёсткая на сохранении** + подсветка поля при потере фокуса. Кнопку
«Сохранить» не блокировать — проверка всё равно на сохранении, лишнее состояние не
плодим.

### 9.2 Полнота карточки — исполняемая проверка (P10-R9 🟡, мокап `completeness`)

В шапке организации — индикатор **«Карточка заполнена N/M»** со всплывающим списком
незаполненного. Чек-лист обязательного минимума:

полное наименование · краткое наименование · краткое описание · ИНН · ОКПО ·
свидетельство о регистрации (номер + дата) · юридический адрес · хотя бы один
**действующий** счёт · хотя бы один телефон · телефон **антикоррупционной линии** ·
электронная почта · адрес сайта · документ типа **«Устав»**.

Индикатор — не декорация: пункты вычисляются теми же валидаторами, что и формы, а
отсутствие антикоррупционной линии дополнительно выводится предупреждением на
вкладке «Контакты».

## 10. Журнал изменений (P10-R7 🟠)

### 10.1 Общесистемное хранилище, а не свой лог

Записи аудита всех модулей лежат **в одном хранилище подсистемы администрирования**
и различаются полем `scope`. Вкладка ведомства — **отфильтрованное представление**
(`scope = 'vedomstvo'`), собственного журнала модуль не заводит.

### 10.2 Состав записи

| Поле | Назначение |
|---|---|
| Когда | дата и время изменения |
| Кто | пользователь |
| Блок | «Общая информация», «Хронология», «Регистрационные данные», «Банковские реквизиты», «Документы», «Контакты» |
| Объект | `objId` + человекочитаемый `objLabel` («Счёт EUR · Айыл Банк», «Телефон · Горячая линия») |
| Поле | что именно изменено |
| Было → Стало | старое и новое значение |

**`objId` + `objLabel` обязательны для блоков с множественностью** (счета, документы,
контакты, хронология): без них по записи журнала невозможно найти объект, к которому
она относится, и журнал превращается в поток бессвязных строк.

### 10.3 Это журнал, а не версионность

Откат по записи журнала не предусмотрен: журнал отвечает на вопрос «кто и когда
изменил», а не «вернуть как было». Версионируется только источник оргструктуры — и
это его собственное свойство, а не свойство ведомства.

### 10.4 UI (мокап `renderAudit`)

Фильтры: блок · пользователь · период (с / по). Пагинация по 10 записей.
Выгрузка **CSV**. Вкладка видна **только администратору** (`adminOnly`).

| 6 | Оргструктура | зеркало | никто (источник — модуль оргструктуры) |
| 7 | Показатели | зеркало | никто (источник — модуль статистики) |
| 8 | Журнал изменений | представление | никто (пишется автоматически); видна только администратору |

### 11.2 Режим «только чтение»

Наблюдатель видит те же вкладки без кнопок правки (в заголовке секции — метка
«только чтение»). Доступные ему действия — **не редактирование**, и отбирать их не
нужно (мокап `READ_ACTS`):

- копирование реквизитов счёта,
- печать карточки организации,
- скачивание файла документа,
- выгрузка журнала в CSV (в пределах видимости вкладки),
- пагинация и фильтры.

### 11.3 Срез «на дату»

В верхней панели — выбор даты просмотра (по умолчанию сегодня, кнопка «Сегодня»
возвращает к текущей). Дата влияет **только на зеркала** (оргструктура,
показатели): собственные блоки версионности не имеют и всегда показываются в
актуальном состоянии.

## 12. Печатная карточка организации (P10-R10 🟡)

Кнопка **«Карточка организации»** — в шапке организации на первой вкладке, доступна
и наблюдателю. Формирует одностраничную выжимку для печати/PDF (мокап
`openPrintCard`):

наименования и описание · регистрационные данные (ИНН, ОКПО, свидетельство, орган
регистрации) · **действующие** счета с реквизитами · контакты (телефоны, почты,
адреса, сайт, режим работы) · дата формирования.

Закрытые счета в печатную форму не попадают — это карточка «как есть сейчас», по
ней платят.

## 14. Открытые вопросы

| # | Вопрос | Почему важно |
|---|---|---|
| 1 | Кто, кроме администратора, вправе править контакты (например, канцелярия — часы приёма)? | Сейчас модель «правит только администратор»; если полномочия шире — нужна вторая роль |
| 2 | Нужна ли история значений собственных блоков (версионность), или журнала изменений достаточно? | Журнал отвечает «кто изменил», но не даёт «покажи карточку на 01.01.2025» |
| 3 | Порог предупреждения об истечении документа — 90 дней? | Зависит от сроков перевыпуска лицензий и разрешений |
| 4 | Кто отвечает за состав чек-листа полноты карточки (§9.2)? | Список обязательного минимума должен утвердить владелец |
| 5 | Требуется ли публичная (внешняя) выдача карточки организации на сайт? | Определяет, нужен ли отдельный read-only API/экспорт |
