## 1. Назначение и границы

### 1.1 Назначение модуля «Заявка»

**Заявка** — центральная сущность фазы 3. Собирает заёмщика, условия кредита, документы,
залог и (для групповых программ) состав группы, проводит их через проверку и
коллегиальное решение и после одобрения порождает кредит(ы). В целевой модели заявка —
**управляемый конечный автомат** (статус-степпер), а не свободно редактируемая карточка.

**Комиссия по заявкам** — коллегиальный орган и модуль (`commission.html`), где ведётся
голосование и выносится **единственное** итоговое решение по заявке (кредит + залог
вместе), протоколируемое кворум-гейтом.

### 1.2 Два потока происхождения заявки (спека «Два потока заявок»)

- **Поток A — по программе:** вход из карточки/списка кредитной программы → программа
  зафиксирована и наследуется в заявку конвертом условий; после одобрения кредит
  формируется сразу.
- **Поток B — индивидуальная:** вход «Создать индивидуальную заявку» (гейт по роли) →
  программы нет, условия свободные; после одобрения кредит **не** формируется — заявка
  переходит в «Ожидает решения/программы», ждёт привязки нового решения + программы,
  затем идёт как поток A. Признак потока — `app.flow` (мокап).

### 1.3 Ключевые отличия от as-is

1. **Двух-комиссионная модель отменена** (заказчик 2026-07-06; P3-R4 #4 / P3-R26):
   одна комиссия судит и кредит, и залог — один голос, одно решение, один протокол.
   Убраны колонка «Статус залога», вторая кнопка «Отправить в залоговую комиссию»,
   параллельные статусы. Залог — не отдельный статус, а **гейт** отправки в комиссию
   (коэффициент покрытия + обязательные залоговые документы).
2. **Статус-степпер управляемый** (P3-R22): переходы только через действия ЖЦ с аудитом,
   не свободная правка статуса; редактируемость полей гейтится статусом (P3-R10).
3. **Воркфлоу комплектации** (мокап; спека 2026-07-10): документы → подтверждение
   головного филиала → заключения отделов → комиссия (предв.) → комитет — 5-узловой
   пайплайн (`appStage`), **ортогональный** статусу.
4. **Единый движок решения** живёт в модуле «Комиссии по заявкам» (`commission.html`);
   в карточке заявки фаза «Заседание комитета» — read-only зеркало (модель B, P3-R47).

### 1.4 Границы (что НЕ входит)

- Формирование кредита/траншей, график выданного кредита, кураторы — модуль «Кредит»
  (условия наследуются при выдаче).
- Реестр субъектов/заёмщиков, справочники залога/областей/отделов — внешние модули
  (в заявке — пикеры и справочники-источники).
- Печатные формы и шаблоны договоров — поздняя стадия, отдельные задачи.

## 2. Роли и фазы (сквозная модель доступа)

### 2.1 Роли (`ROLES`, мокап)

> Модель расширена до 4 рабочих ролей + наблюдатель. Роли `gf`/`dept` — надстройка
> комплектации (спека 2026-07-10 §6), в закрытом графе Топика 4 их ещё нет. Роль в
> проде берётся из auth (Jmix security); в мокапе — переключатель `setRole` (сбрасывает
> `_editMode`, откатывает несохранённое `_revertEdit`, перерисовывает деталь).
> **Не сверено на live.**

| Ключ | Роль | Зона ответственности |
|---|---|---|
| `spec` | Кредитный специалист | создаёт/правит заявку в черновике, все действия тулбара, назначает отделы на заключения |
| `gf` | Головной филиал | 2-й уровень комплектации: подтверждает укомплектованность документов (`confirmDoc`) |
| `dept` | Отдел (заключение) | вносит заключение только за свой отдел (`_deptKey`), после подтверждения ГФ |
| `com` | Комиссия | правит опер-условия и голосует на «На рассмотрении» |
| `ro` | Остальные | только чтение |

`_deptKey` (`setDept`) выбирает, от чьего имени действует роль «Отдел».

### 2.2 Фазы (`condPhase(status)`, мокап)

| Фаза | Статусы | Кто правит меняемые условия |
|---|---|---|
| `draft` | Новый · Требуется доп. информация | Кредитный специалист |
| `review` | На рассмотрении · Ожидает решения/программы | Кредитная комиссия (экран специалиста read-only) |
| `locked` | Одобрена · Отклонена · Отозвана · На оформлении · Оформлена | Никто (зафиксировано) |

### 2.3 Матрица прав `can(app)` (роль × фаза → право)

Каждое право = потенциальное (роль × фаза) **И** включённый режим «Изменить» (`_editMode`)
там, где помечено *(edit)*. Пусто = не даётся.

| Право | Кому/когда | spec·draft | com·review | gf·draft | dept·draft | com·locked |
|---|---|:--:|:--:|:--:|:--:|:--:|
| `editReq` *(edit)* | заявитель, документы, члены, залог | ✅ | | | | |
| `editOp` *(edit)* | опер-условия + комментарий комиссии | | ✅ | | | |
| `editSched` *(edit)* | окно правки графика (P3-R13) | ✅ | | | | ✅ |
| `vote` | голос члена комиссии | | ✅ | | | |
| `confirmDoc` | ГФ подтверждает укомплектованность | | | ✅ | | |
| `assignDepts` | spec назначает отделы | ✅ | | | | |
| `editConcl(dept)` | dept вносит заключение (после `confirmedMet`) | | | | ✅ (свой) | |
| `withdrawConcl(dept)` | dept отзывает своё submitted-заключение | | | | ✅ (свой) | |
| `toolbar` | действия тулбара заявки (§3.4) | ✅ (любая фаза) | | | | |

Правила гарда (мокап):
- `rawReq = spec && draft`; `rawOp = com && review`; `rawSched = (spec && draft) || (com && locked)`;
  фактические `editReq/editOp/editSched` = потенциальное **И** `_editMode`.
- Роль/отдел фиксируются снимком в момент вызова `can()` (`curRole`/`curDept`) — замыкания
  `editConcl`/`withdrawConcl` не «уезжают» при смене роли.
- `editConcl`/`withdrawConcl` дополнительно требуют `_docStats(app).confirmedMet`.

### 2.4 Иммутабельные поля (спека Топик 4; P3-R10)

Кредитный специалист не меняет никогда, даже в `draft`:
- **Субъект (заявитель)** — фиксируется при создании;
- **Поля, изменённые комиссией** — 3-й слой diff-провенанса заблокирован для специалиста.

По статусу: {Новый, Требуется доп. информация} → правится по линзе наследования конверта;
все прочие статусы → полный read-only (гард полей P3-R10).

## 3. Статусная модель и пайплайн процесса

### 3.1 Перечень статусов заявки (`STATUS_META`, одна дорожка)

`step` (0..4) управляет полным таймлайном `renderLifecycle`; `badge` — цветом бейджа.

| # | Статус | Смысл | step | mode | Владелец |
|---|---|---|:--:|---|---|
| 1 | **Новый** | создаётся/заполняется, ещё не в комиссии (слиты «Черновик»+«Новый») | 0 | edit | кред. специалист |
| 2 | **На рассмотрении** | отправлен в комиссию | 1 | view | комиссия |
| 3 | **Требуется доп. информация** | комиссия вернула на дозаполнение (возврат, не решение) | 1 | edit | кред. специалист |
| 4 | **Одобрена** | решение комиссии = одобрить | 2 | view | — |
| 5 | **Отклонена** | решение комиссии = отклонить (терминал) | 1 | view | — |
| 6 | **Ожидает решения/программы** | *только поток B*: одобрено, кредита нет — ждёт привязку решения+программы | 2 | view | оператор |
| 7 | **На оформлении** | кредит/транши сформированы; договоры на нотар. регистрации | 3 | view | оператор |
| 8 | **Оформлена** | договоры зарегистрированы — терминал (успех) | 4 | view | — |
| 9 | **Отозвана** | отозвана до решения (терминал) | 1 | view | кред. специалист |

**Нормализация (`STATUS_ALIAS`):** legacy-значения приводятся к каноническим при загрузке
(`r.status = STATUS_ALIAS[r.status] || r.status`): `Одобрено → Одобрена`,
`Отклонено → Отклонена`. Правило: новые синонимы добавлять в `STATUS_ALIAS`, не в
бизнес-логику; сравнения по статусу — только по каноническим строкам. Инвариант касается и
статкарточек (§4.7): считать **после** алиаса, иначе статус разъедется на две плитки.

### 3.2 Граф переходов (спека Топик 4; реализация — мокап)

Каждый переход — **действием** (управляемый переход), не редактированием статуса.

| Из | Действие | В |
|---|---|---|
| Новый | Отправить в комиссию | На рассмотрении |
| Новый | Отозвать | Отозвана |
| На рассмотрении | Вернуть (комиссия) | Требуется доп. информация |
| На рассмотрении | Одобрить (комиссия) | Одобрена |
| На рассмотрении | Отклонить (комиссия) | Отклонена |
| На рассмотрении | Отозвать | Отозвана |
| Требуется доп. информация | Отправить в комиссию (без лимита циклов) | На рассмотрении |
| Требуется доп. информация | Отозвать | Отозвана |
| Одобрена (поток A) | Сформировать кредит | На оформлении |
| Одобрена (поток B) | Сформировать кредит (авто-ветвление) | Ожидает решения/программы |
| Ожидает решения/программы | Привязать решение + программу | На оформлении |
| На оформлении | Зарегистрировать договоры (нотариус) | Оформлена |
| Отклонена / Отозвана | Пересоздать (клон) | Новый (новая заявка) |
| Оформлена | — | терминал |

Правила: отзыв только из {На рассмотрении, Требуется доп. информация}
(`WITHDRAWABLE_STATUSES`); при отзыве запись комиссии уничтожается (`app._com = null`);
{Оформлена} — тупик, {Отклонена, Отозвана} — только клон. Ветвление A/B на «Сформировать
кредит» — по `app.flow` (`formCredit`).

### 3.3 Пять узлов пайплайна (`STAGE_LABELS`/`appStage`) — ортогональны статусу (P3-R42)

Статус = юридическая веха; пайплайн = где физически стоит комплектация. Показывается на
«Общей информации» (`_renderStages`) и колонкой списка (`_stageCell`).

| Узел | Подпись | Короткая | Условие достижения (`appStage`) |
|:--:|---|---|---|
| 1 | Комплектация | Комплектация | собирают/проверяют документы (`!accMet`) |
| 2 | Подтверждение ГФ | Подтверждение | ждут подтверждения головного филиала (`!confirmedMet`) |
| 3 | Заключения отделов | Заключения | ждут заключений отделов (`!conclusionsReady`); `ready` когда всё готово |
| 4 | Комиссия (предв.) | Комиссия | «На рассмотрении», фаза `prelim` |
| 5 | Комитет · решение | Решение | фаза `session` / решённые статусы; `decided: approved/rejected` |

Флаги узлов: `sb-ready` — готова к отправке (`sendReady`); `sb-neg` — есть отрицательное
заключение отдела (`_conclNegDepts`).

**Инвариант «готова к отправке»:** признак обязан стоять на `sendReady(app)`, **не** на
`appStage().ready` — последний не знает про гейт группы (`_groupGate`) и покрытие залогом,
пометил бы «готовой» заявку, которую кнопка отправить не даёт. Список, колонка «Стадия» и
кнопка считаются по одному правилу.

> **Расхождение со спекой:** закрытый Топик 4 знает 9 статусов, но узлы «Подтверждение ГФ»
> и «Заключения отделов» — надстройка мокапа (спека 2026-07-10 §6). При переносе в live
> согласовать: узлы 2–3 — под-состояния статуса «Новый», а не отдельные статусы.

### 3.4 Действия жизненного цикла (`TOOLBAR`/`renderToolbar`)

Все кнопки тулбара **заявки** жмёт кредитный специалист (`can(app).toolbar === (_role==='spec')`).
Решения комиссии — в тулбаре вкладки «Комиссия», не здесь.

| Статус | Действия тулбара (`TOOLBAR`) |
|---|---|
| Новый | Изменить · Отправить в комиссию · Удалить |
| На рассмотрении | Отозвать |
| Требуется доп. информация | Изменить · Отправить в комиссию · Отозвать |
| Одобрена | Сформировать кредит *(A)* / авто-ветвление в ожидание *(B)* |
| Ожидает решения/программы | Привязать решение + программу |
| На оформлении | Зарегистрировать договоры |
| Отклонена / Отозвана | Пересоздать |
| Оформлена | — (терминал) |

Гейты и аудит:
- **Удалить** — только «Новый» (`UNDELETABLE_STATUSES` защищает «Одобрена»); удаление лишь на
  черновике (P3-R7); ролевой гард — только `spec`.
- **Отправить в комиссию** — честный гейт `sendReady(app)` (см. §4.8). Не прошёл → кнопка
  `disabled`, `title = sendGateReason(app)`.
- **В режиме правки** тулбар заменяется «Сохранить»/«Отмена» (`enterEdit`/`saveEdit`/`cancelEdit`);
  «Сохранить» гейтится `_validateCondFields` + `_validateAppModel`.
- **Каждое действие пишет аудит** в Историю `_audit(app,user,event,from,to,key)`; `key`
  связывает реальную запись с синтетической вехой (§8).
- Гард по статусу — тот же класс, что P3-R7/R10: действие проверяет и роль, и статус, иначе
  `showToast(...,'warn')`.

## 4. Список заявок и форма создания

### 4.1 Форма «Новая заявка» — одноэкранная (P3-R28, P3-R5, P3-R6)

Открывается кнопкой грида «Создать» (`openCreateModal`) как отдельная страница-редактор
(`#view-create`), **не пошаговый мастер**. Все поля видны сразу, тремя секциями:
«Заявитель», «Условия кредита», «Дополнительная информация». Как есть на live — мастер с
«Далее» (до 6 шагов); целевое — один экран.

| Поле | Тип | Обяз. | Источник | Правило |
|---|---|:--:|---|---|
| Субъект | lookup (пикер реестра субъектов) | **Да** | Реестр заёмщиков | Метка `«<ИНН/ПИН> - <Наименование>»` (обычный дефис); при подаче разбор `^\s*(\d+)\s*[-–—]\s*(.+)$` → `inn`+`name`; выбор снимает `.invalid` |
| Адрес | text | Нет | автозаполнение из субъекта | readonly (`ro`), плейсхолдер «Заполнится из субъекта» |
| Номер телефона | tel | Нет | — | Маска `+996 XXX XXX XXX` (`maskPhone`/`formatPhone`, P3-R25); валиден при 0 или ровно 9 цифрах |
| Кредитная программа | lookup (пикер программ) | **Да** | Реестр программ | Предзаполняет условия; **никогда не блокирует** ввод (спека 2026-07-05); хинт «Не выбрана — все условия вручную» |
| Условия кредита | блок виджетов | частично | копия виджетов условий выбранной программы (`renderCond`) | Сумма>0 и ∈ конверта (`validateReqAmount`); срок>0 и ∈ конверта (`validateReqTerm`); день платежа 1–31; обязательные селекты программы (`validateProgSelects`) |
| Комментарий к заявке | textarea | Нет | — | → поле `info` |

- **Обязательность «Субъект»+«Кредитная программа» (P3-R5 🔴):** оба — ядро заявки; на live
  `required=false` на сервере (P3-05), заявку можно провести без заёмщика и программы —
  бессмысленна, засоряет список. Маркер `*` + inline «обязательно» **и серверная валидация**
  (UI-гейт защитой не является). Валидаторы `validateSubject`/`validateProgram`.
- **Ленивая валидация (P3-R6 🟡):** ошибки обязательности **не на рендере**, только по `blur`
  или «Сохранить». Валидатор «Сохранить» (`validateCreate`) прогоняет все field-валидаторы
  **без short-circuit** — разом подсвечиваются все невалидные поля.
- **Кнопка коммита «Сохранить», не «Далее» (P3-R28):** на одном экране «Далее» врёт (шага
  нет). Подпись «Сохранить» + иконка (`submitCreate`); по успеху push строки (`status:'Новый'`)
  → `renderGrid()` → `gotoDetail(newNum)`. Роли разведены: grid-кнопка «Создать» открывает
  форму, кнопка формы «Сохранить» фиксирует запись.
- **P3-R1 (визуальная семантика полей):** общий компонент ДС — обязательное → белый+рамка+`*`,
  опциональное → белый+рамка, readonly → серый (на live все в сером disabled-стиле, P3-01).

### 4.2 Колонки списка (P3-R8, P3-R26)

Грид `#appGrid`, слева направо: Номер документа (`num`) · Заемщик (`name`) · Стадия (`stage`,
производное, сортировка по номеру узла) · Статус заявки (`status`) · Кредитная программа
(`program`) · Запрашиваемая сумма (`amount`, **числом**, `NUMERIC_COLS`) · Дата создания

### 4.8 Отправка в комиссию — единая (P3-R26)

Одна кнопка «Отправить в комиссию» (`openSendModal('credit')`). Гейт `_sendBlockReason`: заявка
выбрана → роль `spec` → статус ∈ `SENDABLE_STATUSES` → `sendReady(app)`; иначе тост.

Число комиссий-адресатов — из программы (`app.commissions`): **1** → отправка сразу (`_doSend`);
**2+** → модалка `#modal-send` с выбором из этих комиссий (обязателен). `_doSend` → статус «На
рассмотрении», пишет `_commission`, сбрасывает голоса на новый круг (`_comStateFresh`),
аудит-запись → История, тост.

**`sendReady(app)` — единый гейт готовности** (то же правило, что бейдж «готова» и колонка
«Стадия»): комплект обязательных документов собран (`_docStats.met`) → при группе `_groupGate.ok`
→ при залоге `coverageOf` даёт `hasItems && covOk` → подтверждено ГФ (`confirmedMet`) → заключения
готовы и без отрицательных (`conclusionsReady`). `sendGateReason` — человекочитаемая причина.

### 4.9 Структурный номер заявки `З-2026-000123` (P3-R27 🟡)

Как есть — голый счётчик «Заявка - 60», не сортируется между годами.
- **Формат:** `З` + год (4 цифры) + `-` + порядковый (**6 цифр**, ведущие нули); без кода
  программы/органа. Пример `З-2026-000123`.
- **Годовой сброс:** серия с `000001` с 1 января, ключ — год.
- **Присвоение при подаче** («Подать»/«Сохранить»), **не в черновике** (черновик — временный
  техн. id). Номер неизменяем и не переиспользуется.
- **Реализация:** атомарный `sequence` БД по ключу года; в Jmix — `Sequences` / idGenerator
  `Numbers`. **Запрет `max(id)+1`** и клиентского счётчика (гонка → дубль). В мокапе
  `submitCreate` считает через `max(...)+2` только для демо формата — в проде номер выдаёт
  sequence при подаче.

## 5. Карточка заявки: «Общая информация» и «Условия кредита»

Общие правила редактируемости — §2 (`condPhase`, `can(app)`). Модель правок (persist, P3-R29):
«Изменить» (`enterEdit`) снимает снимок (`_editSnapshot`), включает `_editMode`; поля пишутся

### 5.2 Вкладка «Условия кредита» (P3-R18 / R29 / R15)

Рендер `renderConditions(app)` (`tab-cond`). Аккордеон-секции повторяют вкладки программы:
① Основные данные · ② Сумма и срок · ③ Процентные ставки · ④ Штрафы · ⑤ Льготный период ·
⑥ Платежи и расчёты · ⑦ Решение комиссии.

**3-слойная провенанс-модель (P3-R18):** программа (базовое) → заёмщик (запрошенное) → комиссия
(одобренное). `condField`/`condDiff`:
- diff «было→стало» на КАЖДОМ условии — цепочка со стрелкой `→` (прежние зачёркнуты `cf-prior`)
  + тег источника у каждого слоя;
- где единственный вариант / не менялось / нет прежнего — **одно значение без стрелки** (не
  фабриковать изменение);
- «было» из `app.orig[prop]` либо дефолта программы; «стало» из `app[prop]`; демо-провенанс —
  `DEMO_PROV` (заявки 084/089/103).

**3 класса affordance редактируемости (P3-R29):**

*Класс 1 — readonly 🔒 (единичное значение программы):* Кредитная программа, Валюта, Тип основной

*Класс 2 — ограничено программой (select из опций / число в min–max, `condRefField`/`condSelect`/

| Условие | Контрол | Ограничение |

- единый `editCond = editReq || editOp`: специалист правит до «На рассмотрении», комиссия — на

### 6.1 Вкладка «Члены группы» (P3-R19)

Рендер `renderMembers()` (`tab-3`). Модель — зонтичный batch: 1 кредит / N траншей, члены
независимы, joint-liability и лидера группы нет.

| Поле строки | Тип | Правила | Функция |
|---|---|---|---|
| Субъект (реестр заёмщиков) | ПИН+ФИО, пикер | только через пикер; дубль ПИН запрещён | `addMemberOpen`→`addGroupMember` |
| Сумма транша | число, инлайн | `>0`; пересчитывает Σ/доли/гейт | `setMemberSum` |
| Планируемая дата освоения | нативный календарь | хранение ru, виджет ISO | `setMemberDate` |
| доля % (read-only) | бейдж | `sum/Σ×100`, `.bad` при `>MAX_SHARE` | inline |

- **Инвариант суммы (снизу вверх):** «Сумма кредита» = Σ траншей — read-only итог внизу; тело
  для сверки — `approvedAmount || amount`.
- **Гейт `_groupGate(app)`** (единый для вкладки и `sendReady`/`sendGateReason`): `sizeOk`
  (`GROUP_MIN ≤ n ≤ GROUP_MAX`, демо 3–10) · `shareOk` (доля ≤ `MAX_SHARE`+0.5, демо 40%) ·
  `sumOk` (`|Σтраншей − сумма| < 0.01`). Баннеры: открыт → «Группа укомплектована…»; закрыт →
  «Отправка заблокирована: <причины: размер / доля / расхождение Σ>».
- **Тулбар** (при `editReq`): «Добавить члена» (disabled при `n≥GROUP_MAX`) / «Удалить» (по
  выбранной строке). `delMember`: **запрещено после одобрения** (транш в работе → тост); вместе
  с членом каскадом уходят его документы (`mem:<pin>::…`, связь P3-R37).
- Производные группы: график — §6.2 (P3-R30), документы — §7 (P3-R37).

### 6.2 Вкладка «График погашения» (P3-R30 / R11 / R13)

Диспетчер `renderSchedule(app)` (`tab-1`).

**10 колонок (`SCHED_HEAD`/`_computeRows`):** № · Дата начала · Дата платежа · Дней · Ставка % ·
Остаток на начало · Проценты · Погашение осн. долга · ПЛАТЁЖ ИТОГО · Остаток на конец. Колонка
«Надбавка» **убрана** (P3-R13). Один формат у члена, свода и индивидуального кредита.

- **Индивидуальный кредит (`!isGroup`):** один график; поле **«Дата выдачи»** (req, только
  не-групповые, `condDate`+`setIssueDate`) — от неё считается 1-я «Дата начала».
- **Групповая заявка (`isGroup`):** дропдаун «График по: <член·сумма> / ▦ Вся группа» (`schedView`);
  дефолт «Вся группа»; у члена свой график от его транша (`m.sum`) и даты освоения (`m.date`,
  контекст `_schedCtx`); «Вся группа» — **read-only свод** (`_aggInner`): слияние по-членных строк
  по датам (`Map`), Проценты/Тело/ИТОГО = Σ по членам, Дней/Остатки — на уровне группы; правка
  только на вкладке члена (в своде `_schedCtx=null`, хендлеры выходят).
- **P3-R11 — проценты по ставке за период:** на live баг «Проценты»=0 при 12%. Считать: ставка
  за период `T=rate/100×fEff/12`; **аннуитет** `PMT=P·T/(1−(1+T)^(−n))` (при 0% → `P/n`);
  начисление `accrued=openBal×rate/100×days/base` (`base=365`); дифференцированный → `P/bodyRows`;
  поддержаны льготы (`graceMain`/`graceAccrual`/`graceInterest`); дата платежа зажимается по длине
  месяца (`_payDate`, 31→28/29/30, не «перетекает»), опц. перенос выходных.
- **P3-R13 — правка под гейтом «Изменить»:** тулбар строк («Построить график»/«Добавить»/
  «Изменить»/«Удалить строку») виден только при `editSched` (edit-mode); право = spec@draft **или**
  com@locked; вне окна read-only авто-рендер (`genSchedule`, сохраняет льготу); дата платежа —
  нативный календарь (`type=date`); валидация строки (тело ≥0, дата строго позже предыдущей и не
  позже выдача+срок, ручные помечаются `man:true` ✎); баланс-инвариант (остаток на конец=0, иначе
  баннер); аудит `_schedAudit`.

### 7.1 Жизненный цикл документа (P3-R35, `DOC_STATUS`)

Хранимых статусов 8 (`review` — производный **вид**, не хранится: `uploaded`-документ у заявки в
комиссии показывается «На проверке» через `_dispSt`/`_atCommission`).

| Ключ | Label | Когда | Владелец перехода |
|---|---|---|---|
| `required` | Требуется | файла нет | — |
| `requested` | Запрошено | запрошен из госсистемы, ждём ответа | спец (P3-R36) |
| `uploaded` | Загружен | файл есть, заявка не в комиссии | спец |
| `review` | На проверке | вид для `uploaded` при заявке в комиссии | — (производный) |
| `accepted` | Принят | спец «Проверил» (1-й уровень) | спец |
| `confirmed` | Подтверждён | ГФ «Подтвердить» (2-й уровень) | ГФ |
| `rejected` | Отклонён 🔴 | вернули на перезагрузку | спец/комиссия |
| `expired` | Просрочен 🔴 | срок истёк (мета `valid`) | система |
| `waived` | Не требуется | освобождён спецом | спец |

**Два уровня подтверждения:** «Проверил» (спец, `docAccept`→`accepted`) → «Подтвердить»/«Отклонить»
(роль ГФ, `confirmDoc`→`confirmed`); две галочки «✓спец/○ГФ». Подтверждённый ГФ документ спец не
трогает (`gfLocked = st==='confirmed'`) — сначала ГФ снимает подтверждение.

**Действия строки** (`docRow`): `required`→«Загрузить»; `rejected`/`expired`→«Перезагрузить»;
`uploaded`→«Проверил»/«Отклонить»; `waived`→«Вернуть»; печатные (`kind:'print'`)→«Печать»→«Загрузить
скан»; договорные (`kind:'contract'`)→«Сгенерировать»; `requested`→«Отменить запрос»/«Ответ получен».

**Гейты-предикаты (`_docStats`):** `OPEN_BLOCKS = required|rejected|expired|requested`;
`met = reqClosed===reqTot` (собран); `accMet = reqAccepted===reqTot`; `confirmedMet =
reqConfirmed===reqTot` (→ открываются заключения). `requested` — открытый блок, но **не** blocker.

### 7.2 Секции документов (`DOC_SECTIONS`)

| key | Название | Обязательность | Особенности |
|---|---|---|---|
| `ident` | Идентификация заёмщика | обяз./опц. по типу (`ft`) | физ: паспорт/ИНН/согласия; юр: устав/ЕГРЮЛ/решение/приказ |
| `print` | Печатные формы | обяз./опц. | `kind:'print'`; бланк→печать→подпись→скан |
| `fin` | Платёжеспособность | обяз./опц. | физ: справка о доходах; юр: отчётность/декларации; `cbr` — кредитный отчёт (опц.) |
| `proj` | Целевые / проектные | все опц. | по решению специалиста (бизнес-план, ТЭО, финмодель) |
| `coll` | Залоговые | обяз. по программе | `collateral:true`; под-панели по предмету, комплект из `COLL_DOC_DEFAULTS` по виду; **источника интеграции не имеют** |
| `contract` | Договорные | обяз. | `stage:'contract'`; шаблоны с merge-полями, залочены до одобрения |

Программа сужает `ident/print/fin/proj` до `docPack` (`_applyProgramPack`); `coll`/`contract` — как
есть.

### 7.3 Обязательные документы как гейт одобрения (P3-R12)

«Отправить в комиссию» доступно только при полном обязательном комплекте: `sendReady = met &&
confirmedMet && conclusionsReady && (при залоге) covOk && (при группе) _groupGate.ok`. Причина
блокировки — баннером над прогресс-баром (`sendGateReason`). Договорные и залоговые исключены из
`_docStats` (этап после одобрения / гейт по покрытию); загрузку после этапа блокирует `gfLocked` /
статус.

### 7.4 Запрос из внешних госсистем (P3-R36, `DOC_SOURCES`)

Маппинг `docId → система`: `p1/p2`→ГРС (Тундук); `inn`→ГНС; `reg/egr`→Минюст; `inc`→Соцфонд;
`tax/nod`→ГНС; `cbr`→Кредбюро («Кредитный отчёт (кредбюро)», новый док, `req:false`).
- Кнопка «Запросить из «X»» — при `DOC_SOURCES[docId]` && роль спец в правке && обычная секция.
- Флоу: `required`→(«Запросить»)→`requested` («Запрошено из «X» · дата»)→(«Ответ получен»)→
  `uploaded` («Получено из «X» · автоматически»)→ обычная проверка. «Отменить запрос»→`required`.
- Гейт ждёт ответа: «Ожидается ответ интеграции по N док.».
- **Инвариант:** источник по `docId`, а не по сырому ключу строки (`_docSrc`): у составных ключей id
  пересекаются (`reg` залогового ТС ≠ `DOC_SOURCES.reg` Минюста); разворачивать только ключи члена
  (`mem:…`), залоговые (`::`) источника не имеют никогда.

### 7.5 Документы по членам группы (P3-R37)

- При `isGroup` секции `ident`/`fin` — на уровень **члена** (`print`/`proj`/`contract` остаются по
  заявке). Ключ `mem:<pin>::<docId>`; стор плоский (`_memberDocs`); тип комплекта — по субъекту
  члена (физ → паспорт/ИНН; юр → устав/ЕГРЮЛ).
- Селектор «Документы по: [член · N/M ▾ | ▦ Вся группа]» (`renderMemberDocs`), тот же паттерн, что
  на «Графике»; свод «Вся группа» — сводка N/M + замечания по членам; `_docStats` суммирует по

- Гейт называет членов **поимённо** (`_memberGateList`): «У 2 чл. (ОРОЗБЕКОВА А.М. — 0/6,
  БЕКТУРГАНОВ Э.Н. — 5/6) комплект не собран».

### 7.6 Командный центр вкладки «Документы» (P3-R46, `renderDocs` — 3 блока над аккордеонами)

1. **Сегментный прогресс** (`docprog`): три накопительных порога Собрано (`reqClosed`) → Принято
   спецом (`reqAccepted`) → Подтверждено ГФ (`reqConfirmed`); крупный headline-% = первая
   незавершённая стадия.
2. **Конвейер стадий-фильтров** (`docstage-rail`): чипы по стадиям ЖЦ со счётчиком (`_docStageCounts`),
   клик = фильтр (несовпавшие строки **скрываются**); первым — синтетический чип «Надо загрузить»
   (`DOC_UPLOAD_PENDING = required|rejected|expired`).
3. **Очередь «Требуют вашего действия»** (`docq`, `_docActionQueue`): по роли — спец:
   Загрузить/Перезагрузить/Проверить/Ответ получен; ГФ: Подтвердить; клик → раскрыть+подсветить.
- **Инвариант:** очередь строится на праве роль×фаза (`canEditReq`/`confirmDoc`), **не** на
  транзиентном `_editMode` — обзор работает и в просмотре. Охват доменов повторяет `_docStats`
  (статические + члены + залог по предметам); договорные исключены.

### 7.7 Вкладка «Заключения» (P3-R41)

**Справочник `DEPT_DIR`** (7 отделов): рисков (дефолтный), кредитный (дефолтный), залога (авто при
залоге), юридический, аналитики, безопасности, мониторинга.

**Назначение состава** (спец, `draft`, `assignDepts`): `risk`+`credit` несъёмные (🔒); `coll` авто
по наличию залога (`_conclSyncColl`, несъёмный пока залог есть); остальные — добор из справочника
(`conclAssign`), снятие через `conclUnassign`. Панель назначения (`_conclAssignPanel`): чипы, 🔒 на
несъёмных (тултип-причина), `✕` снимает; снятие непустого — через модалку подтверждения
(`modal-concl-unassign`). Dropdown добора = `DEPT_DIR` минус назначенные и минус авто-отделы. Роль
«Отдел» правит **только свой** блок (`editConcl(dept)` = роль отдел && `_deptKey===dept` && draft &&
`confirmedMet`); чужие — read-only.

**ЖЦ заключения** (`_conclItem`: status/verdict/text/conds/files/author/date/log): `pending → draft →
submitted`. «Внести» (`conclSubmit`), «Сохранить черновик» (`conclSaveDraft`, вердикт снимается),
«Отозвать» (`conclWithdraw`→draft, содержимое цело). Возврат комиссией (`_conclResetByCommission`)→
`submitted`→`draft`. Вердикты: Положительное / С условиями / Отрицательное. Условия — список (≥1 при
«С условиями»), вложения (`files`).

**Валидация «Внести»:** нет вердикта → «Выберите вердикт»; текст пуст → «Заполните текст»; `cond` &&
0 условий → «Добавьте пункт»; `neg` && текст <20 симв. → «Обоснуйте отрицательное заключение».

**Гейты:** `conclusionsReady = assigned.length>0 && все submitted && ни одного neg`. Отрицательное
заключение блокирует отправку: узел пайплайна краснеет (`.stg.rej`), причина называет отдел поимённо
(«Отрицательное заключение: Отдел рисков — устраните замечания, отдел отзовёт и внесёт новое»;
эскалации «вопреки» нет). Порядок причин баннера (`_conclGateBanner`): не подтверждён ГФ → отделы не
назначены → есть neg → внесено N из M → все внесены. **Одно заключение на отдел на заявку** (и при
группе, и при N предметах залога).

**Инварианты переноса:** (1) `_conclCard` рисуется в двух панелях (`tab-concl` — редактор своего
отдела; `tab-6` — read-only свод у комиссии), различаются по `ctx`/`_conclCardId`; редактор — только
в `tab-concl`. (2) Снятие формы перед перерисовкой (`_conclReadForm` в `conclSubmit`) — иначе правки
теряются. (3) Запертость через `_conclLocked(app,key)`, **не** через поле `locked` (было мёртвым
always-false). Вкладка **безусловна** (назначение с черновика).

## 8. Вкладка «История» (P3-R16)

На live История пуста для заявки Подача→Одобрено (аудит-трейла нет, P3-16). Целевой аудит юридически
значим.

**Механизм (мокап: `_audit` → `buildHistory` → `renderHistory`):**
- **Запись:** `_audit(app,user,event,from,to,key)` кладёт реальные действия в `app.audit`; `key`
  (`send`/`return`/`approve`/`reject`/`waitB`/`credit`/`notary`/`withdraw`) связывает запись с
  синтетической вехой того же смысла.
- **Сборка:** `buildHistory(app)` = синтетические вехи, **выведенные** из текущей модели (на пустой
  заявке не появляются) + реальные `app.audit` и `app.schedEdits`; `pushK(key,...)` печатает
  синтетику только если реальной записи с этим ключом нет (защита от задвоения); сортировка по
  времени убыв.
- **Формат грида (`renderHistory`):** Дата-время · Пользователь · Событие · Было → Стало (readonly).

**Что логируется:**
- *Статусные вехи:* Заявка создана · Отправлена в комиссию · Возврат из комиссии · Решение
  Одобрить/Отклонить · Ожидание решения/программы (поток B) · Сформирован кредит · Договоры
  зарегистрированы · Заявка отозвана.
- *Field-level («было→стало», юридически значимо, `COND_AUDIT_FIELDS`/`_captureCondEdits`):* сумма,
  срок, годовая ставка, назначение, источник финансирования, вид кредита, кредитная линия, вид счёта
  погашения, 3 льготных периода. Автор — `[specialist]` в draft / `[commission]` в review; «было»
  пишется в `app.orig[prop]` один раз. Правки комиссии (`[chair]`), добавление члена группы, загрузка
  обязательного комплекта (`N из M`), правка графика после одобрения (`app.schedEdits`, P3-R13).
- *Не логируем детально* (только факт правки, без диффа): телефон и прочие не-денежные поля.

| **A. Предварительное изучение** | карточка заявки, вкладка «Комиссия» (`renderCommission`, `prelim`) | список обяз. документов, «Запросить замену», «Вынести на заседание комитета» (тулбар) | да (переход фаз + возврат документа) |
| **B. Заседание комитета** | модуль `commission.html` | голосование · кворум · решение · протокол · скан | да — **единственный** |
| B (зеркало) | карточка заявки, та же вкладка, `session` | read-only: статус-решение · голоса+комментарии · подписи · скан · прогресс; баннер направляет в модуль | **нет** |

**Инварианты:** функции `comDecide`/`comVote`/`comSign`/`comUploadProtocol`/`comBackPrelim` в
карточке заявки **удалены** (иначе два движка с расходящимися правилами; раньше tab-6 имел свой
`comDecide` без кворума); `comRequest`/`comAdvance` целы. **Runtime-связь (на прод):** `commission.html`
держит заявку строкой без id — реальный модуль должен линковать `З-…` ↔ `К-…` (структурные номера,
P3-R27), чтобы зеркало читало итог из записи комиссии. **Одна комиссия судит кредит+залог** (P3-R4
#4).

### 9.2 Кворум-гейт и единая карточка (P3-R38, ядро 🔴)

| Функция | Правило |
|---|---|
| `quorumOf(rec)` | Кворум = `⌊n/2⌋+1` от числа членов |
| `tally(rec)` | `approve/reject/abstain/novote`; `voted`, `quorum`, `quorumMet = voted ≥ quorum` |
| `decisionGateReason(rec)` | человекочитаемая причина блокировки; пусто = гейт пройден |
| `decisionReady(rec)` | `decisionGateReason === ''` |

**Гейт итогового решения** (порядок = порядок устранения): 1) статус ≠ «На рассмотрении» → «решение
уже вынесено»; 2) кворум не набран → «проголосовало N из M, требуется K»; 3) нет номера протокола;
4) нет даты заседания.

**Единая карточка** (замена двух экранов): одна карточка read-only + зоны действий (голос · решение ·
документы · протокол); прогресс **над** блоком решения; «Проголосовать» — в строке **своего** члена
(повторный голос заблокирован); пустая секция «Комментарии членов» удалена (решение+комментарий — в
гриде «Состав комиссии»); «OK/Отмена» и правка сущности убраны.

**Права `can(rec)` (роль × статус):** `vote` — член состава по нерешённой комиссии, ещё не
голосовавший; `decide` — **только** председатель при `pending` (под гейтом кворума); `protocol` —
председатель/секретарь при `pending`; `manage` — секретарь.

**Инвариант (критично):** UI-гейт без серверной проверки кворума — **не защита**. Отдельной задачей
подтвердить, что сервер валидирует кворум перед сменой «Тип решения».

**Эволюция правила (P3-R4 #3 ↔ P3-R38):** as-is (P3-R4 #3, live 2026-06-19) — кворум **без порога**
(председатель закрывает в любой момент, голоса справочно). To-be ядро P3-R38 **вводит** кворум-гейт
`⌊n/2⌋+1`. Осознанное ужесточение — при реализации согласовать каноническое правило с заказчиком.

### 9.3 Правила комиссии (P3-R4)

| # | Правило |
|---|---|
| #1 | **Финал ручной + аудит** 🟠: UI ручное (кнопки Одобрить/Отклонить/Запросить), не менять; добавить аудит-запись при сохранении финала (кто/когда/решение+причина, особенно при расхождении с тэлли — вероятно backend) |
| #2 | **4-eyes предупреждение** 🟠: автор/податель = член/председатель по своей заявке → не блокировать, предупредить + пометить в аудите (в UI сейчас нет) |
| #3 | **Кворум без порога** — no-op (задокументировать); **расходится с P3-R38** — см. §9.2 |
| #4 | **Одна комиссия кредит+залог** 🟠 (реверс, 2026-07-06) — см. §9.1 |
| #5 | **«Крайний срок» члена** 🟠: дедлайн принуждается **уведомлением** члену+председателю; голос авто **не** меняется (без авто-«воздержался»); нужен таймер/планировщик + нотификации |

### 9.4 Список комиссий: рабочая очередь (P3-R39 🟠)

| Область | Стало (to-be) | Функция |
|---|---|---|
| Рабочая очередь | плитка + фильтр «Ждут моего голоса»; тег «ваш голос» в строке | `awaitsMyVote`, `tileFilter`, `renderTiles` |
| «Крайний срок» | колонка-чип: «Просрочен N дней» / «Сегодня» / «дата · N дн.»; только пока «На рассмотрении» | `deadlineCell` |
| «Тип решения» | статус-чип `--status-*` | `STATUS_BADGE` |
| «Голоса» | мини-полоса сегментов + «N из M» | `miniVote` |
| Балласт | убрать константную «Тип документа» + всегда-пустой «Комментарий» | — |
| Обрезка | `title` на обрезанные ячейки | — |
| Номер | структурный `К-2026-000138` (единый с `З-…`, P3-R27) | — |

**Статкарточки (поглощает P3-R32):** «Всего» = Σ статусных плиток; кликабельны как фильтр (повторный
снимает); рамка серая при 0, статусная при >0; фильтр — «Применить/Сбросить» + чипы активных условий
(`renderChips`/`applyFilter`/`resetFilter`/`clearOne`), свёрнут по умолчанию.

**P3-R14 🟡 (дубль колонки «Сотрудник»):** в живом гриде состава две колонки «Сотрудник» (логин
`[admin]` / ФИО) → развести «Логин» / «ФИО» (или «Должность»). *В `commission.html` состав уже несёт
единую колонку «Сотрудник» = ФИО.*

### 9.5 Прогресс, диалоги, пустые состояния (P3-R40 🟡)

- **Прогресс:** стек-сегменты по 4 категориям (Одобрили/Отклонили/Воздержались/Не проголосовали,
  `renderVote`/`LEG`) + «Проголосовало N из M · кворум K набран/не набран». Цвета из палитры,
  контраст WCAG ≥ 4.5:1 (as-is «Отклонили» = `rgb(255,123,123)` ≈ 2.3:1 — провал AA;
  «Не проголосовали» — чёрная плашка).
- **Диалоги:** Escape + клик по подложке закрывают (`askConfirm`/`tryClose`/стек `_stack`); при
  несохранённом вводе — подтверждение потери (`voteDirty`); одна пара подписей «Да/Отмена»; текст
  называет действие **и запись** («Одобрить заявку «Заявка - 97» по комиссии К-2026-000138? Протокол
  №… Решение необратимо»); деструктив — красная кнопка (`.btn-danger`); диалог «Ваш отзыв» — 560px,
  «Решение» обязательно, «Риск» — реальный справочник.
- **Пустые состояния:** empty state с текстом и действием вместо `.empty-rows`; в «Документах» —
  кнопка «Загрузить»; «Протокол» — не тупик («Сформировать протокол»/«Загрузить скан» под тем же
  гейтом, `renderProt`); «История» — таймлайн (`renderHist`); read-only значения без имитации
  контролов.
- **Комментарий члена (P3-R34):** текст в ячейке + «Показать полностью» (обрезка меряется по факту
  `scrollWidth>clientWidth`, не по длине строки).

### 9.6 Вкладка «Комиссия» карточки заявки (`renderCommission`, tab-6)

Строка-пайплайн стадий над контентом: «1. Предварительное изучение → 2. Заседание комитета · решение ·
протокол». Вход: `commissionFor(app)==null` (Новый/Отозвана) → баннер «Заявка ещё не отправлена в
комиссию»; `_comStateOf(app)` держит проекцию (`phase`·`votes`·`comments`·`final`·`scan`·`open`).

- **Фаза A «Предварительное изучение» (движок в карточке):** список обяз. документов (`_reqDocList`:
  осн. секции + документы членов + залоговые) со статус-бейджем; «Запросить замену» (`comRequest` →
  возврат документа специалисту → заявка «Требуется доп. информация»; только роль «Комиссия»);
  «Вынести на заседание комитета» (`comAdvance`, **в общем тулбаре**, решение 2026-07-10; гейт =
  комплект подтверждён ГФ **И** все заключения внесены). Гарды продублированы в модели (`_comActor`) —
  UI прячет кнопки, функции не верят рендеру.
- **Фаза B = read-only зеркало:** движка нет; показывает прогресс, тэлли-чипы, грид состава
  (read-only), «Финальное решение» (бейдж), скан протокола; баннер направляет в модуль. По-членный
  чек-лист подписей убран (итог = голосование + один скан).
- **Расхождение зеркала (хвост):** прогресс-полоса фазы B в карточке одноцветная, без кворум-строки —
  в отличие от `commission.html` (стек-сегменты + кворум). При реализации зеркало читает те же данные
  из записи комиссии; эталон прогресса — `commission.html` (P3-R39/40).

## 11. Открытые вопросы и расхождения (на согласование при переносе в live)

1. **Роли `gf`/`dept` и узлы «Подтверждение ГФ»/«Заключения отделов»** — есть в мокапе (спека
   2026-07-10 §6), но НЕ в закрытом графе Топика 4. Решить: под-состояния статуса «Новый» или
   отдельные шаги. **Не сверено на live.**
2. **Кворум комиссии** — as-is «без порога» (P3-R4 #3) ↔ to-be кворум-гейт `⌊n/2⌋+1` (P3-R38).
   Согласовать каноническое правило; серверная валидация кворума обязательна.
3. **Статкарточки заявок** — плитки считать по выборке или по всему реестру? (на комиссиях — по
   выборке).
4. **`STATUS_META['Отклонена'].step=1`** — на полном таймлайне отклонение отматывается на веху 2
   через `LC_OFF`; проверить, что бейдж/степпер не показывают «Отклонена» как ранний шаг.
5. **Состав вкладок vs «макс 7»** — добавлены «Условия кредита» и «Заключения»; переучёт лимита —
   хвост в мокапе (`TABS`).
6. **Серверные гейты** — все UI-гейты (обязательность полей, кворум, «нет отрицательных заключений»,
   комплект подтверждён ГФ, покрытие залогом) без серверной валидации защитой не являются —
   подтвердить на live отдельными задачами.
7. **Ключи прода** — район по id (не имени); связь `commission.html` заявка↔запись по структурным
   номерам (не строке); залоговые/составные ключи документов не путать с источниками интеграции.
