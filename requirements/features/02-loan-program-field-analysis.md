> Route: `/loan-programs/new` · Осмотр живого тест-стенда `https://fkftest.okmot.kg/`
> (сессия `admin`) — 2026-06-18. Линза анализа: **полный продукт** (не MVP).
> Дополняет структурный документ [`02-loan-program.md`](02-loan-program.md),
> не заменяет его. Здесь — смысл, нужность и правило обязательности каждого поля.

## Метод

- Пройдены все 9 вкладок визарда, сняты скриншоты, программно собран инвентарь
  полей (типы, метки, флаг `required`, `readonly`) и **реальные наборы значений
  всех `select`-полей**.
- Маркеры обязательности **не показываются** при открытии формы — обязательность
  определена поведением валидации (7 полей Tab 1) + анализом downstream-логики.
- **Содержимое справочников lookup-полей (•••) захвачено** (2026-06-19, скрипт
  `scripts/inspect/lookups-all.mjs`): 12 из 14. Значения и находки — в разделе
  [«Захваченные справочники»](#захваченные-справочники-2026-06-19). Не захвачены 2
  условных: виды залога (за залоговым гейтом) и справочник ставок (появляется после
  выбора «Фиксированная» на вкл.3).

## Легенда вердиктов «Нужно?»

| Вердикт | Значение |
|---|---|
| **Оставить** | Поле необходимо, название и поведение корректны |
| **Переименовать** | Поле нужно, но название/формулировка/раздел вводят в заблуждение |
| **Объединить** | Дублирует/перекрывает другое поле — свести воедино |
| **Условное** | Показывать только при определённом выборе (зависимое поле) |
| **Убрать** | Бизнес-ценность не подтверждается, на уровне программы лишнее |

Колонка «Обязательность»: **текущее состояние формы → предлагаемое**
(Обязательное / Необязательное / Условное / Авторасчёт).

> ✅ Таблицы по вкладкам **приведены в соответствие** с решениями владельца
> (2026-06-19). Раздел [«Решения обсуждения»](#решения-обсуждения-2026-06-18-с-владельцем)
> сохранён как датированный лог обсуждения 2026-06-18.

- Всего редактируемых полей/контролов (без вкладки 9 «Предпросмотр»): **~46**.
- Вкладка 9 — read-only представление (25 полей), 1:1 дублирует данные вкладок
  1–8; в пополевой разбор не входит как самостоятельные поля.

| Вердикт | Кол-во (после обсуждения) |
|---|---|
| Оставить | ~36 |
| Условное | ~10 |
| Переименовать | ~2 (Способ погашения; заголовок «Ограничение штрафа») |
| Объединить | ~3 (Валюта-зеркало вкл.2; Выбранные шаблоны; Метод расчета дней + Тип расчета процентов) |
| Убрать | **0** (заявители оставлены как поимённый whitelist) |

**Ключевой системный риск (подтверждён, решение принято):** обязательны только **7**
полей вкладки 1. Всё финансовое ядро — сумма, срок, ставка, метод/периодичность
погашения — опционально → можно создать «пустую» программу. **Решение:**
обязательность финансового ядра **к отправке** (валидация на «Отправить на
подтверждение»), черновик остаётся свободным. +3 поля вкл.1 становятся обязательными
сразу (кырг. наименование, источник финансирования, валюта).

финансовые параметры, условия погашения, требуемые документы, настройки проверки.

### Находки по содержимому (для владельца)

- **A. «Назначение кредита» — не классификатор, а свободные цели.** Диалог
  «Цели кредита» содержит целые абзацы конкретных проектов («…строительство завода

  моделью «3 чистые оси». **Вопрос:** на уровне программы нужен короткий классификатор
  целей (пополнение оборотных / инвестиции / строительство), а не свободные тексты?
- **B. «Вид кредита» смешивает 3 понятия:** продукт (Бюджетная ссуда/кредит) +

  активы). Доноры тут **дублируют** уровень «Кредитная линия». **Вопрос:** почистить
  справочник «Вид кредита» до продуктовых типов, доноров увести в «Кредитную линию»?
- **C. «Кредитная линия» = АРР, ФРР** — не совпадает с названными АБР/МАР/ВБ.
  Тест-данные скудны. **Вопрос:** дать боевой перечень доноров/линий.
- **D. «Тип расчёта процентов» содержит «Аннуитет»** — но аннуитет это **метод
  погашения** (есть отдельное поле «Метод погашения» = Аннуитетный/Дифференц.).
  «Аннуитет» в расчёте процентов — вероятно ошибочно заполненное тест-значение.
