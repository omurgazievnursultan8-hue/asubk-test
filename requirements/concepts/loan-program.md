# Кредитная программа — что это и зачем

## 1. Зачем она нужна

**Шаблон кредитного продукта.** Кредитная программа — это «заготовка» кредита:
один раз заданные условия (ставки, сроки, суммы, график, документы, требования к
залогу), по которым потом штампуются конкретные заявки и кредиты. Не нужно
описывать условия заново для каждого заёмщика — он подаёт заявку **по программе**,
и она наследует всю рамку.

**Точка, где задаются условия кредитования.** Решение правительства разрешает
*направление* (на каком основании вообще можно кредитовать). Программа превращает
это разрешение в **конкретный продукт**: под какой процент, на какой срок, какими
траншами, под какой залог. Из программы «вырастают» заявки, а из заявок — кредиты.

**Прослеживаемость (аудит).** Любой выданный кредит разматывается обратно:
кредит → **кредитная программа** → решение правительства. Программа — это среднее
звено цепочки происхождения средств: она привязывает рыночные условия к правовому
основанию.

**Без программы** нельзя создать заявку — заявке не на что сослаться за условиями,
и кредит не из чего «собрать».

---

## 2. Что это и из чего состоит

Программу **создаёт сотрудник Холдинга** (в отличие от решения, которое приходит
извне) — но только **на основе уже действующего решения правительства**. Программа
ссылается ровно на одно решение, и только в статусе **«Действует»** (серверная
проверка — P2-R9). Нет действующего решения — программу создать нельзя.

Программа собирает в одном месте всё, что нужно для выдачи кредита:

| Блок | Что задаёт |
|---|---|
| Правовое основание | ссылка на решение правительства (статус «Действует») |
| Назначение и вид | цель кредита, тип продукта, отрасль, валюта, источник финансирования; признак «Групповой кредит»; комиссия по заявкам |
| Срок действия | «Дата с» / «Дата по» — окно, когда по программе можно работать |
| Сумма и срок | фиксированный набор значений **или** диапазон (мин/макс) |
| Ставки | основная процентная ставка (фикс / плавающая), ставка за неполное освоение (резерв), штрафы за просрочку |
| Льготы | льготные периоды по основному долгу / начислению / уплате процентов |
| График | периодичность, день платежа, метод погашения, очередь, day-count |
| Документы | кредитные документы, печатные формы, шаблоны договоров; залоговые документы — по видам залога |
| Залог | требуется ли, виды залога, требования к оформлению |

> Полный пополевой разбор (≈40 полей, 9 вкладок) — в техническом спеке
> `requirements/tz/02-kreditnaya-programma.md` и мокапе
> `mockups/loan-program/loan-program.html`.

## 3. Место в кредитном процессе

Программа — **второе звено** цепочки, сразу после решения. Она берёт правовое
разрешение и оборачивает его в продаваемый продукт:

```
Решение правительства
        │  (разрешает направление кредитования)
        ▼
Кредитная программа          ← ссылается на действующее решение
        │  (условия: ставки, сроки, суммы, график, документы, залог)
        ▼
Заявка (+ рассмотрение комиссией)   ← подаётся по программе
        ▼
Заёмщик
        ▼
Кредит (выдача)
        ▼
Освоение (транши) → Мониторинг кредита (платежи) → Мониторинг залога
        ▼
(при необходимости) Взыскание задолженности
```

Ключевые связи: **одна программа ← одно решение** (N:1) и **одна программа → много
заявок/кредитов** (1:N). Поэтому программу, на которую уже опираются заявки или
кредиты, нельзя бесконтрольно удалять (защита ссылок — P2-R7); удаление возможно
только в статусе «Черновик».

## 4. Жизненный цикл записи (целевая модель)

Программа проходит согласование по принципу «создатель ≠ утверждающий»
(maker/checker, 4 глаза — P2-R5). Целевая модель — **6 статусов** и роли «Автор»,
«Согласующий», «Администратор». Полная матрица прав — в спеке §7.

```
Черновик ──Отправить──► На рассмотрении ──Одобрить──► Действует ──Закрыть──► Закрыта
   ▲                        │     │
   └──Отозвать──────────────┘     ├──Отклонить──► Отклонена
                                  └──Вернуть──► На доработке ──Отправить повторно──┐
                                        ▲                                          │
                                        └──────────────────────────────────────────┘
        админ-возврат: Закрыта / Отклонена ──► На рассмотрении (с причиной в журнал)
```

Что означают статусы:

- **Черновик** — создаётся автором, ещё не подана; редактируется свободно.
- **На рассмотрении** — подана Согласующему на проверку.
- **На доработке** — возвращена автору с замечаниями; снова редактируется.
- **Действует** — утверждена; **только по ней** можно создавать заявки и кредиты.
- **Отклонена** — Согласующий отказал в согласовании (отдельный статус, **не**
  «Закрыта»).
- **Закрыта** — ранее действовавшая программа отменена/утратила силу (достижима
  только из «Действует»).

Правила целевой модели:

- Редактирование — только в «Черновик» / «На доработке».
- Удаление — только «Черновик» (soft-delete + запись в журнал).
- «Действует» — можно лишь закрыть.
- Самосогласование запрещено: отправивший не может сам одобрить — нужен **другой**
  пользователь.
- «Закрыта» и «Отклонена» **не терминальны**: администратор возвращает их «На
  рассмотрении» с обязательной причиной в журнал.
- Тулбар: кнопки **скрываются по роли**, **гасятся по статусу**.

> Модель унифицирована с решением правительства (Фаза 1 R7) с поправкой на род
> терминов программы («Отклонена» / «Закрыта»).

## 5. Что хранит карточка программы — поле за полем

Карточка — **9-вкладочный мастер** (≈50 редактируемых полей). Ниже — каждое поле:
**что это** и **зачем нужно**. Обозначения обязательности: **Обяз.** — нужно для
отправки на согласование; **Усл.** — обязательно при включении соответствующего
блока (диапазон / льгота / залог); **Опц.** — необязательное; **Авто** — выводится
системой. Технический тип каждого поля и валидация — в спеке
`requirements/tz/02-kreditnaya-programma.md`.

### Вкладка 1 — Основные данные

*Группа «Общая информация» — кто этот продукт и на каком основании.*

- **Наименование на русском** (Обяз.) — название продукта; по нему программу узнают
  в списках, отчётах и связанных сущностях. Базовая идентификация.
- **Наименование на кыргызском / на английском** (скрыто) — поля **временно скрыты**
  до появления мультиязычности интерфейса; модель данных под них сохранена, при
  включении i18n вернутся (кырг. — гос. язык для офиц. документов, англ. — для
  международной отчётности). Сейчас в форме не показываются и в обязательные не входят.
- **Краткое описание** (Обяз.) — короткая выжимка условий программы; показывается в
  списке, чтобы отличать программы без открытия карточки.
- **Решение правительства** (Обяз.) — ссылка на правовое основание (только статус
  «Действует»). Ядро прослеживаемости: связывает продукт с государственным актом.

*Группа «Параметры программы» — три независимые оси «зачем / какой / кто» + рамка
действия, источник и условия возврата.*

- **Назначение кредита** (Обяз.) — **зачем** берётся кредит, цель трат (пополнение
  оборотных, покупка техники, строительство). Ось «цель».
- **Вид кредита** (Обяз.) — **какой** это продукт по источнику/механизму:
  **бюджетный кредит**, **бюджетная ссуда**, **иностранный кредит**. Ось «тип продукта».
- **Отрасль** (Обяз.) — **кто** заёмщик: сектор экономики (агропромышленный комплекс,
  энергетика, машиностроение, лёгкая промышленность, банки и т. д. — справочник из 15
  отраслей). Ось «сектор».
- **Групповой кредит** (Опц., справочный флаг) — программа для группового продукта
  (один кредит → N траншей на членов группы) или индивидуального. Дефолт — выкл.
  Логику формы **не меняет** (не гейтит поля, не обязателен): только сохраняется и
  показывается в «Просмотр»/предпросмотр. Потребитель — стадия заявки: `true` включает
  условную вкладку «Члены группы» (P2-R24).
- **Дата с** (Обяз.) — начало действия программы; нижняя граница — дата связанного
  решения (нельзя начать раньше правового основания).
- **Дата по** (Опц.) — конец действия программы; если задана — не раньше «Даты с»
  (`Дата по ≥ Дата с`). Может быть бессрочной (R12).
- **Источник финансирования** (Обяз.) — откуда деньги: бюджет / донор / собственные.
  Задаёт категорию средств для отчётности и трассировки происхождения денег.
- **Валюта** (Обяз.) — валюта программы. Единственный источник валюты: от неё зависят
  сумма и срок, на вкладке 2 она наследуется только для чтения. Без валюты сумма
  бессмысленна.
- **Кредитная линия** (Усл.) — конкретный донор / линия финансирования (АБР, МАР, ВБ)
  внутри выбранного источника. Не дубль «Источника»: источник — категория, линия —
  конкретный донор; показывается при источнике-доноре.
- **Вид счёта погашения** → «Способ погашения» (Опц.) — куда/как возвращаются
  средства: **накопительный счёт** или **бюджет**. Подлежит переименованию: текущее
  название путает со счётом бухгалтерским.

*Группа «Дополнительные параметры» — адресность и ответственность.*

- **Список заявителей (физ. лица)** (Опц.) — поимённый whitelist допустимых
  заявителей-физлиц для адресных программ (когда круг получателей заранее ограничен).
- **Список заявителей (юр. лица)** (Опц.) — то же для юрлиц.
- **Ответственные сотрудники** (Обяз.) — список **кураторов**, которые будут
  назначены кредитам по программе (наследуются на кредит при выдаче). Задаёт пул
  ответственных за сопровождение, а не просто «владельца программы».
- **Комиссия по заявкам** (Опц.) — коллегиальный орган, который будет рассматривать
  заявки по программе. Одиночный выбор из справочника «Комиссии», отфильтрованного
  по типу **«Заявки»** (тип «Поручительства» не показывается), только активные.
  Потребитель — стадия заявки: поле **предзаполняется** из программы (оператор может
  переопределить); пусто на программе = выбор комиссии вручную на заявке. Отдельная
  роль от «Ответственных сотрудников» (P2-R25).
- **Комментарий** (Опц.) — свободный служебный текст.

> **Итого обязательных на вкл. 1 — 10** (мокап R12): наименование рус, краткое
> описание, решение, назначение, вид, отрасль, источник, валюта, дата с, ответственные
> сотрудники. Дата по — опциональна; кырг./англ. наименования скрыты до мультиязычности.

### Вкладка 2 — Сумма и срок

*Сколько и на какой срок можно занять по программе. Финансовое ядро.*

- **Ограничение суммы** (Обяз.) — как задаётся сумма: **Фиксированная** (готовый
  перечень) или **Диапазон** (мин/макс). Базовый выбор, от него зависят поля ниже.
  Ранее «Тип суммы кредита» (переименовано — P2-R15).
- **Фиксированные суммы** (Усл.) — перечень допустимых сумм при типе «Фиксированная»;
  требуется ≥1 значение; запрет дублей, авто-сортировка ↑.
- **Минимальная / Максимальная сумма** (Усл.) — границы при типе «Диапазон» (грид
  сумм скрывается); правило `0 < Мин ≤ Макс`, мин==макс → мягкий warning.
- **Ограничение срока** (Обяз.) — аналогично сумме: фиксированный набор сроков или
  диапазон. Ранее «Тип срока кредита» (P2-R15).
- **Фиксированные сроки** (Усл.) — перечень допустимых сроков (единица — **месяцы**);
  требуется ≥1 значение.
- **Минимальный / Максимальный срок** (Усл.) — границы срока при «Диапазон»; `Макс ≥ Мин`.
- **Валюта** (Авто) — зеркало валюты вкладки 1, только для чтения; чтобы суммы
  читались в правильной валюте.

### Вкладка 3 — Процентные ставки

*Под какой процент даётся кредит.*

- **Тип основной ставки** (Обяз.) — фиксированная ставка или диапазон.
- **Значение фиксированной ставки** (Усл.) — ставка из **регламентированного
  справочника ставок** (ввод списком — по дизайну, не свободное число); при типе
  «Фиксированная» нужно ≥1 значение.
- **Значение ставки с / по** (Усл.) — границы при типе «Диапазон»; `0 < с ≤ по`,
  с==по → мягкий warning.
- **Использовать плавающую ставку** (Опц.) — чекбокс-модель: плавающая ставка =
  **надбавка к основной** (база/бенчмарк + маржа). При включении — **Тип плавающей
  ставки** (пикер из справочника бенчмарков; хранится тип/индекс, значение бенчмарка
  берётся на дату расчёта). **«Маржа к плавающей ставке»** — read-only зеркало
  основной ставки, **не** отдельный ввод (P2-R16).

*Блок «Ставка за неполное освоение (резервная сумма)» — программный дефолт платы за
неосвоенный (резервный) остаток; «неполное освоение» и «резерв» — один и тот же
процент (P2-R26).*

- **Использовать неустойку за неполное освоение** (Опц., гейт) — дефолт выкл: блок
  ставки скрыт, плата за резерв по программе не взимается. Вкл — показывается блок
  ставки.
- **Ставка за неполное освоение** (Усл., при включённом гейте) — **точная копия
  основной ставки**: тип Фиксированная/Диапазон, значение из справочника ставок,
  чекбокс «Использовать плавающую ставку» + тип плавающей + маржа-зеркало. При
  включённом гейте — обязательна (Фикс → ≥1 значение; Диапазон → оба поля, `с ≤ по`;
  плавающая вкл → тип обязателен). Потребитель — стадия кредита: поле «Процент за
  резерв» предзаполняется из программы (оператор может переопределить).

### Вкладка 4 — Штрафы

*Что грозит за просрочку. Защищает интересы кредитора при неплатеже.*

- **Тип штрафной ставки за основную сумму** (Обяз.) — как задан штраф по основному
  долгу: **Фиксированная** / **Плавающая** (ранее «Диапазон» — переименовано, P2-R17).
- **Значение штрафной ставки за основную сумму** (Обяз.) — размер штрафа по основному
  долгу из справочника штрафов (не свободный ввод), единица «% годовых». При
  «Плавающей» — пикер бенчмарка + **маржа-зеркало** (read-only копия фикс. штрафной);
  итог = бенчмарк + фикс. ставка.
- **Тип штрафной ставки за проценты** / **Значение** (Обяз.) — блок штрафа по
  просроченным процентам — **зеркало** блока по основной сумме (та же логика
  Фикс/Плавающая).
- **Максимальный размер штрафа** (Обяз., дефолт 20) — потолок **всего** штрафа
  (тело + проценты), «% от первоначальной суммы»; отдельная полоса «Ограничения
  штрафа». Убран дубль-заголовок «Штраф за просрочку процентов»; структура таба:
  [Осн. сумма | Проценты] + полоса [Ограничения штрафа].

### Вкладка 5 — Льготный период

*Можно ли дать заёмщику отсрочку на старте. Три независимых блока, по дизайну
длительность задаётся вне формы — здесь только включение опции.*

- **Использовать льготный период по основной сумме** (Опц.) — отсрочка платежей по
  телу долга (заёмщик первое время не гасит основную сумму).
- **Использовать льготный период по начислению процентов** (Опц.) — отсрочка самого
  **начисления** процентов (проценты не капают). **Не** то же, что блок ниже.
- **Использовать льготный период по процентам** (Опц.) — отсрочка **уплаты**
  процентов (начисляются, но платятся позже). Начисление ≠ уплата — это разные вещи.
- **Тип / продолжительность льготы** (Усл.) — при включённом блоке требуются тип
  (Фиксированный/Диапазон) и длительность отсрочки (целые месяцы; из справочника,
  запрет дублей). **«Условия предоставления»** — опц. textarea (maxlen 1000), в
  Просмотре не показывается.
- **Период распределения отсроченных %** (Опц., в блоке «по процентам») — «с периода
  / по период»; пусто = дефолт (всё накопленное — единой суммой в первый платёж после
  льготы). Задаёт, куда распределять накопленные-неоплаченные проценты (возникают,
  когда льгота «по процентам» ON и «по начислению %» OFF: проценты начисляются, но не
  платятся → копятся). Гасятся **распределением**, не капитализацией (P2-R18).

### Вкладка 6 — Платежи и расчёты

*Как считается и гасится долг. Параметры графика и формулы процентов.*

- **Способ задания графика** (Обяз., гейт) — как строится расписание платежей:
  **«Стандартная периодичность»** или **«По конкретным месяцам»**. Это две
  **взаимоисключающие** альтернативы — ровно один способ (мокап R19).
- **Периодичность платежей** (Усл.) — при стандартном способе: ежемесячно /
  ежеквартально / **раз в полгода** (f=6) / ежегодно. Каркас регулярного графика.
- **Конкретные месяцы платежей** (Усл.) — при способе «по конкретным месяцам»: любой
  набор месяцев календарного года, в которые вносятся платежи (для сезонных графиков).
- **Привязка последнего платежа** (Обяз., дефолт «По дате первого платежа») — от чего
  считается последний платёж: **«По дате первого платежа»** (последний =
  EDATE(1-й, (n−1)×f), фактический срок может ≠ n мес.) или **«По дате выдачи»**
  (последний = EDATE(выдача, n), срок ровно n мес.). Промежуточные даты всегда от
  1-го платежа (сверка с калькулятором `loan_calculator_v10.xlsx`, P2-R19).
- **День месяца для платежа** (Обяз.) — число месяца платежа (1–31; короткий месяц →
  последний день).
- **Обработка нерабочих дней** (Опц.) — что делать, если платёж попал на выходной/
  праздник: не переносить / на следующий / на предыдущий рабочий день (учитывает
  производственный календарь КР).
- **Метод расчёта дней в периоде (числитель)** (Обяз.) — фактические или условные
  (Финансовый, 30) дни периода (мокап R19 разбил day-count на две части).
- **База дней в году (знаменатель)** (Обяз.) — 360 / 365 / фактическая. Числитель +
  знаменатель вместе задают day-count-конвенцию.
- **Метод погашения кредита** (Обяз.) — аннуитетный (равные платежи) или
  дифференцированный (убывающие). **При графике «по конкретным месяцам» — авто
  «Индивидуальный»** (поле перекрыто, только чтение).
- **Очередь погашения** (Обяз.) — в каком порядке платёж гасит составляющие долга
  (тело → проценты → штрафы → комиссии).
- **Мин. / Макс. кол-во дней до первого платежа** (Опц.) — окно между освоением и
  первым платежом; гибкое, окончательно фиксируется на этапе кредита.

### Вкладка 7 — Документы

*Какие бумаги нужны и кто их проверяет.*

- **Список обязательных кредитных документов** (Опц., рекоменд. ≥1) — обязательный
  пакет документов по кредиту.
- **Список опциональных кредитных документов** (Опц.) — дополнительные документы.
- **Печатные формы** (Опц.) — бланки для выдачи клиенту на стадии заявки
  (заявка-анкета, согласия на обработку ПДн и запрос кредитной истории); **печатаются
  пустыми**. Отдельный блок и отдельный жизненный цикл от шаблонов договоров (мокап).
- **Шаблоны договоров** (Опц.) — шаблоны кредитных договоров; в отличие от печатных
  форм **генерируются после одобрения** с подстановкой данных программы/заёмщика.
- **Залоговые документы — по видам залога** (Усл.) — пакет залоговых документов
  больше **не плоский** список: он формируется **по каждому выбранному виду залога**
  (вкл. 8, «вариант C»). При выборе вида типовые документы **авто-наследуются** из
  админ-справочника связей «вид залога → документы» (`COLL_DOC_DEFAULTS`), оператор
  донастраивает под программу; у каждого вида свой набор **обязательных/опциональных**
  документов и своё поле **«Требования к оформлению залогового договора»**. Удаление
  вида залога каскадно снимает его документы. Вся секция — только при включённом залоге.
- **Уровень проверки документов** (Опц.) — глубина проверки пакета.
- **Срок проверки документов (в днях)** (Опц.) — SLA на проверку (рабочие дни).
- **Ответственный за проверку документов** (Опц.) — кто проверяет документы;
  отдельная роль от «Ответственных сотрудников» вкладки 1.

### Вкладка 8 — Залоговое обеспечение

*Нужен ли залог и какой. Чекбокс-гейт управляет всей залоговой частью (вкл. 7 + 8).*

- **Требуется залоговое обеспечение** (Опц., гейт) — нужен ли залог. Выключен — вся
  залоговая секция скрыта и исключена из валидации; включён — поля ниже обязательны.
- **Список видов залога** (Усл.) — допустимые виды залога (недвижимость, техника,
  товары…); при включённом гейте нужен ≥1 вид.
- **Требования к залогу (ру)** (Усл.) — требования к залогу на русском; обязательно при залоге.
- **Талаптар (кырг)** (Усл.) — требования на кыргызском; обязательно при залоге (языковая политика).
- **Requirements (eng)** (Опц.) — требования на английском; для международной отчётности.

### Вкладка 9 — Предпросмотр

Сводка всей программы **только для чтения**: данные 1:1 подтягиваются из вкладок
1–8 (это представление, а не самостоятельные поля). Нужна, чтобы автор увидел
программу целиком перед отправкой. Внизу — кнопка **«Отправить на согласование»**,
запускающая жизненный цикл (§4).

> Предпросмотр строится **тем же рендером, что и страница «Просмотр»** (единый
> `buildDetailSections` — мокап): один источник правды, поэтому предпросмотр перед
> отправкой и просмотр действующей программы показывают одно и то же, без расхождений.

> **Плюс на странице деталей** (вне мастера): вкладка **«Заявки»** — downstream-ссылки
> на заявки по программе; вкладка **«История»** — журнал аудита (кто, когда,
> действие, причина по каждому переходу статуса).

**Валидация двухуровневая** (P2-R10): «Сохранить» проверяет только вкладки до
текущей (черновик можно оставить неполным); «Отправить на согласование» валидирует
**все** вкладки. Прогрессивный гейт навигации не пускает «вперёд» с незаполненными
обязательными полями.
