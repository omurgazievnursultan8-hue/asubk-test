# Решение правительства — что это и зачем

## 1. Зачем оно нужно

**Правовое основание.** Государственный финансовый холдинг — государственная
организация. Он не может выдавать средства «просто так»: под каждый кредит должно
лежать **законное основание** — решение уполномоченного органа. Запись о решении
и есть это основание, зафиксированное в системе.

**Точка происхождения средств и условий.** Решение задаёт рамку: какое
направление кредитуется, в рамках какого государственного акта. Из решения
«вырастают» кредитные программы, а из программ — конкретные кредиты.

**Прослеживаемость (аудит).** Любой выданный кредит можно «размотать» обратно:
кредит → программа → **решение правительства**. На вопрос «на каком основании
выданы эти средства?» система отвечает ссылкой на конкретный государственный акт
и приложенный документ-первоисточник.

**Без записи о решении** нельзя корректно создать кредитную программу — программе
не на что сослаться, и цепочка происхождения средств рвётся.

---

## 2. Откуда берётся и какие бывают

Решение приходит **извне** — его принимает государственный орган, а сотрудник
Холдинга **регистрирует** его в системе: заводит карточку и **прикладывает
документ-первоисточник** (PDF акта).

Виды решений (поле «Вид решения»):

| Вид | Что это |
|---|---|
| Постановление | Постановление (как правило — Кабинета Министров) |
| Распоряжение | Распоряжение |
| Основное | Базовое/основное решение |
| РКМКР · РПКР · ПКМКР | Ведомственные типы актов |

> 🟡 **На подтверждение:** полный перечень видов и их официальные расшифровки —
> уточнить у домен-эксперта (список собран из стенда, не из нормативной базы).

## 3. Место в кредитном процессе

Решение — **самое начало** цепочки. Это первый раздел домена **«Бюджетное
кредитование»**. Всё остальное идёт следом:

```
Решение правительства
        │  (разрешает направление кредитования)
        ▼
Кредитная программа          ← ссылается на решение
        │  (условия: ставки, сроки, лимиты)
        ▼
Заявка (+ рассмотрение комиссией)
        ▼
Заёмщик
        ▼
Кредит (выдача)
        ▼
Освоение (транши) → Мониторинг кредита (платежи) → Мониторинг залога
        ▼
(при необходимости) Взыскание задолженности
```

Ключевая связь: **одна кредитная программа ссылается на одно решение**. Поэтому
запись о решении нельзя бесконтрольно удалять или менять, если на неё уже
опираются программы (защита ссылок — задача R4).

## 4. Жизненный цикл записи (целевая модель)

Решение проходит согласование по принципу «создатель ≠ утверждающий»
(maker/checker). Целевая модель — **6 статусов** и роли «Автор», «Согласующий»,
«Наблюдатель» (просмотр без действий). Закрытие действующего решения и
админ-возвраты выполняет **администратор**. Полная матрица прав — в
`decision-workflow.html`.

```
Черновик ──Отправить──► На рассмотрении ──Одобрить──► Действует ──Закрыть──► Закрыт
   ▲                        │     │
   └──Отозвать──────────────┘     ├──Отклонить──► Отклонён
                                  └──Вернуть──► На доработке ──Отправить повторно──┐
                                        ▲                                          │
                                        └──────────────────────────────────────────┘
        админ-возврат: Закрыт / Отклонён ──► На рассмотрении (с причиной в журнал)
```

Что означают статусы:

- **Черновик** — запись создаётся, ещё не подана; редактируется свободно.
- **На рассмотрении** — подана Согласующему на проверку.
- **На доработке** — возвращена автору с замечаниями; снова редактируется.
- **Действует** — утверждена; **только на её основе** можно строить программы.
- **Отклонён** — Согласующий отказал в согласовании (отдельный статус, **не**
  «Закрыт»).
- **Закрыт** — ранее действовавшее решение отменено (достижим только из
  «Действует»).

Правила целевой модели:

- Редактирование — только в «Черновик» / «На доработке».
- Удаление — только «Черновик».
- «Действует» — можно лишь закрыть.
- «Закрыт» и «Отклонён» **не терминальны**: администратор возвращает их «На
  рассмотрении» с обязательной причиной в журнал (мис-клик не хоронит юр.
  основание).
- Тулбар: кнопки **скрываются по роли**, **гасятся по статусу** (с подсказкой,
  почему недоступны).

## 5. Что хранит карточка решения

Целевой набор (детали и валидация — в техническом спеке и мокапе `decisions.html`,
страница в режиме «Просмотр + действия», вкладки: Общая информация / Кредитные
программы / Документы / История):

- **Вид решения** — тип акта (см. §2)
- **Номер решения** и **Дата решения** — реквизиты исходного акта (дата не позже
  сегодня — R5)
- **Наименование** и **краткое наименование**
- **Код** — **генерируется системой** в формате `{ТИП}-{ГГГГ}-{NNNN}`
  (пример `ПОСТ-2026-0007`), вручную не вводится (R3)
- **Статус** — этап жизненного цикла (см. §4)
- **Примечание** — свободный текст
- **Документы** — вложения PDF: основной акт + приложения/изменения, ровно один
  помечен «основным»; основной документ **обязателен** для перехода в «Действует»
  (R1)
- **История** — таймлайн действий и переходов статуса: кто, когда, действие,
  причина (единый журнал аудита — R2/R6)
