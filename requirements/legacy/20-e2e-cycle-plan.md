# E2E-прогон полного цикла кредита (legacy) — план

> **Что это.** План сквозного *операционного* прогона на **старой** тестовой системе
> `http://85.113.29.29:8080/`: создать заёмщика, оформить и выдать кредит, **навесить
> залог**, вести график/погашения, запустить **взыскание** (претензия → иск → решение
> → исп.лист), и довести до **полного погашения** (закрытие кредита).
>
> **Отличие от P1–P12.** P1–P12 — статическое описание экранов (что где лежит). Здесь —
> **реальный прогон**: мы САМИ создаём и двигаем записи через весь жизненный цикл, чтобы
> увидеть переходы, пересчёты, статусы и связи «в динамике». Это то, что нельзя снять
> с готового экрана.
>
> **Зачем.** База для мокапов новой системы: увидеть реальный поток (какие поля обязательны,
> какой порядок шагов, что пересчитывается автоматически, где ручной ввод), и поймать
> дефекты, видимые только при прохождении цикла.
>
> **Доступ.** Браузер авторизован (Сламкулов А.О., `/user/1/details`). Хост `85.113.29.29:8080`.

## ⚠️ Осторожно — это МУТИРУЮЩИЙ прогон

Создаём/меняем реальные записи в тест-базе. По `CLAUDE.md` тестовые данные не жалко
(сервер ждёт сброса), НО:
- Необратимые кнопки (**Сохранить · Зарегистрировать · Удалить · Отменить фазу ·
  Изменить статус**) — сначала смотрим диалог, потом жмём **осознанно**, фиксируем результат.
- НЕ трогаем чужие реальные записи (debtor-140 и пр.) — работаем только со своими
  свежесозданными (запоминаем их id).
- Каждый созданный id пишем в «Журнал прогона» (ниже) — чтобы потом найти/добить/сверить.

## Предпосылки цикла (что от чего зависит)

Из P1/P2/P4/P5 порядок зависимостей:
1. **Субъект** (физлицо/организация, P8) → нужен как заёмщик/залогодатель/поручитель.
2. **Заёмщик** (`debtor`, P1) — карточка-агрегатор; кредиты висят на нём.
3. **Решение/Оформление** (`order`, P6 — *ранее вне скоупа, здесь ОБЯЗАТЕЛЬНО*) —
   «Основание выдачи кредита» в карточке кредита ссылается на `order-N`.
4. **Кредит** (`loan`, P2) — создаётся из карточки заёмщика,
   `/manage/debtor/{debtorId}/loan/{typeId=1}/0/save`.
5. **Условия кредита** (ставка/штрафы/метод дней, P2 вкл.9) — версия условий.
6. **График** (P2 вкл.4, «+ Добавить график») — план погашения.
7. **Залог** (P4) — предметы залога + поручители + акт обследования + (снятие с ареста).
8. **Освоение/выдача** — освоение суммы (появляется в Расчётах/Детрасчёте).
9. **Погашения** (P3) — расчёт → приём платежа, частями во времени.
10. **Взыскание** (P5) — при просрочке: процедура → фазы (Претензия→Иск→Решение→Исп.лист)
    → события (заседания).
11. **Полное погашение** — закрыть остаток; статус кредита → «Нулевая сверка»/погашен;
    группа платежесп. → «погашенные долги».

## Метод на каждой стадии

1. Открыть форму → снять **все поля** (обязательные/опциональные, дефолты, справочники).
2. Заполнить минимально валидно → **Сохранить** → зафиксировать: новый id, редиректы,
   что появилось/пересчиталось, ошибки валидации.
3. После каждой мутации — вернуться в карточку и глянуть, **что изменилось** (остаток,
   статус, график, расчёт).
4. Скрины ключевых переходов → `screenshots/legacy/e2e/`.
5. Дефекты по ходу → секция «Дефекты прогона» ниже (потом свод в `notes/qa-findings.md`).
6. **Для каждой стадии фиксируем ЛОГИКУ и АЛГОРИТМ** (раздел ниже): не только поля/id, а
   *что делаем по шагам* + *как система устроена внутри* (модель данных, привязки, что
   пересчитывается, бизнес-правило). Это ядро для проектирования новой системы.

## Ход прогона — ЛОГИКА и АЛГОРИТМ (по стадиям)

> Формат каждой стадии: **Цель** · **Алгоритм** (пошагово, что жмём) ·
> **Логика системы** (модель/привязки/пересчёты) · **Находки**.

### Стадия 0 — Создание субъекта (физлицо)
- **Цель.** Завести субъект-носитель ПД (адрес, идент.документ, контакты), к которому потом
  привяжутся роли (заёмщик, залогодатель, поручитель).
- **Алгоритм.**
  1. `GET /person/add` → модалка «Добавить физ.лицо» (3 секции-вкладки).
  2. Заполнить обяз.поля: ФИО, Область/Район, Адрес+Адрес проживания, Вид документа,
     Кем выдано, Номер, Дата, ИНН.
  3. `Сохранить` → сервер создаёт `person`, редирект `GET /person/{id}/details`.
- **Логика системы.** Субъект — самостоятельная сущность. У него **ДВА разных id**:
  «карточка физлица» = **person-63545** (в URL), внутренний **subject-id = 58542**
  (то, чем субъект привязывается как `owner`). Адрес хранится отдельным `address_id`.
  Гео-иерархия: Область → Район → А.Окмоту → Село (каскадные справочники).
- **Находки.** Село по умолчанию = `title` (мусор-плейсхолдер) → пишется в новую запись
  (E2E-02). Сохранение синхронное, id присваивается сразу.

### Стадия 1 — Создание заёмщика (обёртка над субъектом)
- **Цель.** Превратить субъект в **заёмщика** — сущность, к которой цепляются кредиты,
  залог, взыскание.
- **Алгоритм.**
  1. `GET /manage/debtor/list1` → кнопка «+» = `GET /manage/debtor/0/save` (id=0=создание).
  2. Поле ЗАЕМЩИК — начать печатать ФИО → AJAX-автокомплит выдаёт `[subjId] ФИО (тип)`.
  3. **Кликнуть подсказку мышью** (важно: это заполняет скрытый `owner.id`).
  4. Выбрать ОТРАСЛЬ (`workSector.id`).
  5. `Сохранить` → создаётся `debtor`, редирект `GET /manage/debtor/{id}/view`.
- **Логика системы.** Заёмщик = **тонкая обёртка**: своих ПД-полей НЕТ, только
  `owner.id` (ссылка на subject-id субъекта) + `workSector.id` (отрасль). Всё остальное
  (адрес/идент/контакты) наследуется от субъекта. Значит модель: `person/organization`
  (субъект) — 1 — `debtor` (роль-заёмщик). Один субъект теоретически = несколько ролей.
- **Находки.** Скрытая привязка `owner.id`: клавиатурный выбор автокомплита её НЕ
  проставляет (E2E-01) — только мышь. Отрасль — фиксированный справочник из 15 значений.

### Стадия 2 — Создание решения на выдачу кредита (order, основание)
- **Цель.** Завести «основание выдачи кредита» — распорядительный документ (решение), на
  который сошлётся кредит («Основание выдачи кредита» в карточке кредита, P2 вкл.1).
- **Алгоритм.**
  1. `GET /manage/order/list` («Решении на выдачу кредита») → «+» = `GET /manage/order/0/save`.
  2. Форма: **ВИД** (`creditOrderType.id`: ППКР / РПКР / Другие) · **НОМЕР** (`regNumber`) ·
     **ДАТА** (`regDate`) · **ПРИМЕЧАНИЕ** (`description`). Ни одно поле не помечено `required`.
  3. `Сохранить` → создаётся `order`, редирект `GET /manage/order/{id}/view`.
- **Логика системы.** Решение — **контейнер с воркфлоу-статусом**: сразу после создания
  статус = **«в Ожидании подтверждения»** (значит есть переход подтверждения перед
  использованием). Карточка решения = **3 вкладки**:
  - **ПАКЕТ ДОКУМЕНТАЦИИ** (кнопка «Добавить пакет») — приложенные документы решения;
  - **СПИСКИ ПОЛУЧАТЕЛЕЙ** — перечень заёмщиков-получателей по этому решению;
  - **КРЕДИТНЫЕ УСЛОВИЯ** — условия кредитования, заданные на уровне решения.
  ⭐ **Гипотеза модели:** выдача кредита идёт *через* решение
  (решение → список получателей → заёмщик → кредит), а «Основание выдачи» в кредите —
  это ссылка на `order`. Кредитные условия могут наследоваться из решения. Проверить на
  стадии 3 (создание кредита): требуется ли подтверждённое решение и как он привязывается.
- **Находки.** Минимальный order создаётся без обязательных полей (все `required=false`).
  Тип ВИД = справочник {ППКР, РПКР, Другие} — вид распорядительного акта (Постановление/
  Распоряжение Правительства КР / Другое). Статус после создания — черновик, не активен.
  Открытый вопрос: нужно ли **подтверждать** решение перед выдачей кредита.

### Стадия 3 — Создание кредита (loan)
- **Цель.** Выдать кредит заёмщику: сумма, валюта, вид, куратор, основание (решение).
- **Алгоритм.**
  1. `GET /manage/debtor/{debtorId}/loan/{typeId}/0/save` (typeId=1=кредит, из карточки
     заёмщика вкл. «Кредитная информация»). URL уже несёт `debtorId` в скрытом поле.
  2. Форма (обяз. помечены \*): **РЕГ.НОМЕР\*** (`regNumber`) · **ДАТА ДОГОВОРА**
     (`regDate`, преднаполнена сегодня) · **СУММА\*** (`amount`) · **ВАЛЮТА\***
     (`currency.id`, деф. KGS) · **ВИД КРЕДИТА\*** (`loanType.id`) · **КУРАТОР\***
     (typeahead `#staff_typeahead` → скрытый `supervisorId`) · **ОСНОВАНИЕ ВЫДАЧИ\***
     (typeahead `#order_typeahead` → скрытый `creditOrder.id`).
  3. Куратор/основание — **twitter-typeahead**: печать → AJAX-подсказки `[id] Название`;
     выбор мышью проставляет скрытый id.
  4. `POST /manage/debtor/{d}/loan/1/save` → создаётся `loan`, редирект на карточку
     `/manage/debtor/{d}/loan/{loanId}/view`.
- **Логика системы.** ⭐ Ключевые связи кредита:
  - `debtor` (владелец, в URL) — кредит всегда принадлежит заёмщику (нет плоского реестра);
  - `creditOrder.id` — **прямая ссылка на решение** (order-734). Значит модель:
    решение —1:N— кредиты (одно решение = основание для нескольких кредитов). Выдача НЕ
    идёт через «список получателей» решения — привязка прямая по FK.
  - `supervisorId` — куратор (staff-запись, здесь id 8128 = текущий юзер).
  - Заголовок карточки склеивает **основание + рег.номер + дата + сумма+валюта**:
    «E2E-001 E2E-LOAN-001 от 2026-07-12 (100 000,00 KGS)».
  - Кредит создаётся с **освоением = 0** — сумма договора задана, но деньги ещё не выданы
    (освоение — отдельная операция, стадия 9).
  - Справочник ВАЛЮТА содержит мусор из P11 (тонн, USD(АО …)) — виден прямо в форме выдачи.
- **Находки.**
  - ✅ **Решение НЕ требует подтверждения** для создания кредита: order в статусе
    «в Ожидании подтверждения» принялся как основание без ошибок.
  - ✅ **Точка приёма платежа = вкладка ПЛАТЕЖИ → «Добавить платеж»** в карточке кредита
    (снимает открытый вопрос стадии 10).
  - Форма кредита — «шапка» договора (сумма/вид/куратор/основание). Ставка, штрафы, счёт
    погашения, метод дней — НЕ здесь: задаются отдельно (условия кредита — стадия 4,
    счёт погашения — кнопки-диалоги карточки, P2 вкл.1).
  - Автоматизация: submit кнопкой «Сохранить» не сработал после JS-заполнения полей —
    потребовался прямой `form POST /…/loan/1/save`. При живом выборе typeahead мышью
    кнопка submit'ит штатно (артефакт скрипта, не дефект продукта).

### Стадия 4 — Условия кредита (term, версия начисления)
- **Цель.** Задать параметры начисления: ставка, штрафы, лимит штрафов, очередь погашения,
  метод дней. Без условий кредит не считается (график/начисление опираются на них).
- **Алгоритм.**
  1. Карточка кредита → вкладка **УСЛОВИЯ КРЕДИТА** (`#creditTerms`) →
     «Добавить условия кредита» = `/manage/debtor/{d}/loan/{l}/term/0/save`.
  2. Форма (обяз. \*): **НАЧАЛО ДЕЙСТВИЯ** (`startDate`, преднапол.) · **ПРОЦ.СТАВКА\***
     (`interestRateValue`) · **ПЕРИОД НАЧИСЛЕНИЯ\*** (`ratePeriod.id`: Год/Полугод/Кв/Мес/
     Нед/День) · **ВИД ПЛАВАЮЩИХ СТАВОК\*** (`floatingRateType.id`) · **ШТРАФ ПО ОСН.С.\***
     (`penaltyOnPrincipleOverdueRateValue`)+**ВИД\*** · **ШТРАФ ПО ПРОЦ.\***
     (`penaltyOnInterestOverdueRateValue`)+**ВИД\*** · **ПРЕДЕЛЬНЫЙ ЛИМИТ ШТРАФОВ\***
     (`penaltyLimitPercent`, деф.20%) + **ДАТА** · **ОЧЕРЕДЬ ПОГАШЕНИЯ\***
     (`transactionOrder.id`: осн→%→штраф / %→штраф→осн / Другое) · **МЕТОД ДНЕЙ В ПЕРИОД\***
     + **В ГОД\*** (`daysInMonthMethod.id`/`daysInYearMethod.id`: Календарный/Финансовый/
     Календарный 366) · **СТАТУС**.
  3. `POST /…/loan/{l}/term/save` → запись условий, редирект на карточку кредита; условие
     появляется в таблице со статусом **Действует**.
- **Логика системы.** Условия — **версионируемая запись** (ключ = `startDate` «начало
  действия»): у кредита может быть история условий, действует последняя. Структура ставки:
  **числовое значение (`interestRateValue`) + база (`floatingRateType.id`)** — т.е.
  ставка = фикс.спред поверх плавающей базы (или «--» = без базы для фикс.ставки).
  Штрафы устроены так же (значение + вид-база). Лимит штрафов в % ограничивает капитализацию
  пени. Очередь погашения и метод дней здесь же (не в шапке кредита) — значит начисление
  привязано к версии условий, а не к договору целиком.
- **Находки.** ⚠️ Дропдаун **ВИД ПЛАВАЮЩИХ СТАВОК** (и оба «вид штрафа») содержит мусор
  из справочника floatingRate (P11/P12): «Ставка АБР ГГГГГГГГ», «тест апрапр олпрлр…»,
  «--», «-» — **прямо в рабочей форме условий кредита** (E2E-03). Метод дней задаётся
  раздельно для месяца и года (две настройки).

### Стадия 5 — График погашения (paymentschedule = «срочные обязательства»)
- **Цель.** Задать план: когда осваивается сумма и когда/сколько гасится (осн.сумма +
  проценты). Без графика начислять/сверять нечего.
- **Алгоритм.**
  1. Карточка кредита → вкладка **ГРАФИК** (`#paymentSchedules`) → «Добавить график»
     = `/…/loan/{l}/paymentschedule/0/save`. Форма — **одна строка** («Срочное
     обязательство»), добавляется по одной.
  2. Поля (обяз. \*): **ДАТА** (`expectedDate`) · **ОСВОЕНИЕ/СНЯТИЕ\*** (`disbursement`) ·
     **ОСН.СУММА\*** (`principalPayment`) · **ПРОЦЕНТЫ\*** (`interestPayment`) ·
     **НАК.ПРОЦЕНТЫ\*** (`collectedInterestPayment`) · **НАК.ШТРАФЫ\***
     (`collectedPenaltyPayment`) · **СОСТОЯНИЕ РАССРОЧКИ\*** (`installmentState.id`:
     Не подтвержден / Не наступивший / Наступивший / Непогашенный / Погашенный) ·
     чекбокс `enabled` · загрузка файла.
  3. `POST /…/paymentschedule/save` → строка в таблице графика, редирект `#paymentSchedules`.
  4. Повторить для каждой строки (освоение, затем платежи).
- **Логика системы.** ⭐ График = **набор строк, вводимых ВРУЧНУЮ** (нет авто-генератора
  аннуитета/дифференцир.). Одна таблица содержит **и освоение, и погашения**: строка с
  `disbursement>0` = выдача (drawdown), строки с `principalPayment/interestPayment>0` =
  плановые платежи. `installmentState` описывает жизненный цикл позиции
  (Не наступивший→Наступивший→Погашенный/Непогашенный). Значит **освоение (стадия 9) —
  это строка графика**, а не отдельная операция ⇒ стадия 9 фактически покрыта здесь.
  Колонки таблицы: Дата · Освоение · Осн.сумма · Проценты · Нак.проценты · Нак.штр · Статус.
  Экспорт PDF/Excel (кнопки `label.getPdf`/`label.getExcel` — непереведены).
- **Находки.**
  - Ручной ввод графика — трудоёмко и ошибкоопасно; в новой системе **авто-генерация
    графика** из суммы/срока/ставки/частоты обязательна.
  - Освоение вводится как строка графика (нет отдельной «выдачи») — модель «план+факт»
    в одной сущности.
  - ⚠️ Непереведённые ключи прямо на форме: **LABEL.ADD.PAYMANTSCHEDULE**,
    **LABEL.UPLOAD.FILE** (+ опечатка PAYMAN**T**SCHEDULE); кнопки экспорта
    `label.getPdf`/`label.getExcel` (E2E-04).

### Стадия 6 — Залог (договор залога + предмет)
- **Цель.** Обеспечить кредит: оформить договор залога и завести предмет (что заложено,
  сколько, по какой стоимости).
- **Алгоритм.** ⭐ Двухшаговый, но связанный:
  1. Карточка кредита → вкладка **ЗАЛОГ** (`#agreements`) → «Добавить залог»
     = `/…/loan/{l}/collateralagreement/0/save`.
  2. **Форма договора залога:** КРЕДИТЫ\* (`loanses`, мультиселект — какие кредиты
     обеспечивает, преднабран текущий) · № НОТАР.РЕГ\* (`notaryOfficeRegNumber`) + дата ·
     **НОТАРИУС\*** (typeahead → `notary.id`) · № НАЛОЖ.ЗАПРЕТА\* (`arrestRegNumber`) +
     ДАТА АРЕСТА · № ЗАЛ.РЕГ.СВИД + ДАТА РЕГ. + скрытый `owner.id` (залогодатель).
  3. `POST /…/collateralagreement/save` → создаётся договор и **авто-редирект на форму
     предмета** `/…/collateralagreement/{agrId}/collateralitem/0/save`.
  4. **Форма предмета залога:** НАИМЕНОВАНИЕ\* (`name`) · ПРИМЕЧАНИЕ · **ЗАЛОГОДАТЕЛЬ\***
     (typeahead → `owner.id`, **преднаполнен субъектом заёмщика**) · **ВИД ЗАЛОГА\***
     (`itemType.id`: Автотранспорт/Готовая продукция/Недвижимость/Оборудование/Сельхозтехника/
     Скот/Участок/Прочие/Кредитный портфель) · КОЛ-ВО\* (`quantity`) · **ЕД.ИЗМ\***
     (`quantityType.id`: шт/кг/тонн/кв.м/куб.м/метр/ед) · **ОЦЕНОЧНАЯ СТОИМОСТЬ\***
     (`estimatedValue`) · **ЗАЛОГОВАЯ СТОИМОСТЬ\*** (`collateralValue`) · МЕСТО АРЕСТА +
     **условные «Детали» по виду залога** (для Автотранспорта: МАРКА, ГОС НОМЕР\*,
     № двигателя/шасси/кузова, даты выпуска/ввода; для др. видов — свой набор).
  5. `POST /…/collateralitem/save` → предмет создан, редирект на карточку договора залога.
- **Логика системы.** ⭐ Модель залога **двухуровневая**:
  `collateralagreement` (договор: нотариус, арест, рег.номера, ссылки на кредиты `loanses`)
  — **1:N** — `collateralitem` (предметы: вид/кол-во/стоимость). Договор **M:N с кредитами**
  (мультиселект `loanses` — один договор может обеспечивать несколько кредитов). Стоимость
  предмета в **двух оценках**: оценочная (рыночная) и залоговая (с дисконтом). Набор полей
  предмета **динамически зависит от вида залога** (JS: «make fields required/not according
  to itemType») — для транспорта обязателен ГОС НОМЕР и т.п. Карточка договора = 6 вкладок:
  Договор · Предмет залога · Кредиты · Доп.соглашения · (обзор) · Информация.
- **Находки.**
  - Залогодатель предмета **авто-подставляется** из субъекта заёмщика (удобно), но скрытый
    `owner.id` требует явной установки (та же болезнь typeahead, E2E-01).
  - Условные поля по виду залога — сильная сторона старой системы (транспорт/недвижимость/
    скот — разные атрибуты); сохранить в новой.
  - ⚠️ Непереведённая вкладка договора **LABEL.COLLATERALAGREEMENT.TAB.AGREEMENTREVIEW**;
    на вкладке ЗАЛОГ кредита — `LABEL.QUANTITY/ESTIMATED_VALUE/COLLATERAL_VALUE`,
    `label.collateral_item.not_active` (дубль P2). Кнопка печати «Форма извещения (тест)»
    с «тест» в имени (E2E-06).

### Стадия 7 — Поручитель (договор поручительства)
- **Цель.** Добавить личное обеспечение — поручителя по кредиту.
- **Алгоритм.**
  1. Карточка кредита → вкладка **ДОГ. ПОРУЧИТЕЛЬСТВА** (`#guarantorAgreement`) →
     «Добавить поручителя» = `/…/loan/{l}/guarantoragreement/0/save`.
  2. Форма (обяз. \*): **ПОРУЧИТЕЛЬ\*** (typeahead `#guarantor_typeahead` → `owner.id`;
     рядом ссылка «Добавить физ.лицо» — создать субъект на лету) · **КРЕДИТЫ\***
     (`loanses`, мультиселект — тут «Nothing selected», надо выбрать вручную) ·
     № НОТАР.РЕГ\* (`notaryOfficeRegNumber`) · ДАТА НОТАР.РЕГ\* (`notaryOfficeRegDate`,
     преднапол.) · **НОТАРИУС\*** (typeahead `#owner_typeahead` → `notary.id`).
  3. `POST /…/guarantoragreement/save` → договор поручительства, редирект на карточку кредита.
- **Логика системы.** Структура почти идентична договору залога: обёртка с нотариальной
  регистрацией + M:N-привязка к кредитам (`loanses`). Отличие: вместо предмета — субъект-
  **поручитель** (`owner.id`). Значит и залог, и поручительство — это **обеспечительные
  договоры** над кредитом с общим паттерном (нотариус + рег.номер + кредиты). Возможность
  завести поручителя-физлицо прямо из формы («Добавить физ.лицо»).
- **Находки.** КРЕДИТЫ здесь НЕ преднабраны (в залоге — преднабраны) — несогласованность
  UX между двумя почти одинаковыми формами. Скрытые `owner.id`/`notary.id` — та же
  typeahead-болезнь (E2E-01).

### Стадия 8 — Акт обследования залога (insresult)
- **Цель.** Зафиксировать факт проверки залога и его состояние.
- **Алгоритм.**
  1. Карточка предмета залога (`/…/collateralitem/{i}/view`) → вкладка **АКТЫ ОБСЛЕДОВАНИЯ**
     (`#tab2`) → «Добавить акт обследования» = `/…/collateralitem/{i}/insresult/0/save`.
  2. Форма: **ДАТА** (`onDate`, преднапол.) · **ДЕТАЛИ\*** (`details`) · **ВИД РЕЗУЛЬТАТА
     ОБСЛЕДОВАНИЯ\*** (`inspectionResultType.id`: Хорошее / Удовлетворительное /
     Неудовл.(утраченное/повреждённое/аварийное/треб.ремонт/треб.замена) /
     Удовлетворительное(треб.ремонт) / «-»).
  3. `POST /…/insresult/save` → акт создан, редирект на карточку предмета.
- **Логика системы.** Акт обследования (`insresult`) принадлежит предмету залога
  (N актов на предмет во времени — история осмотров). Периодичность задаётся на предмете
  полем **Состояние** (`conditiontype`: «Обследование раз в 3/2 года / ежегодно / не
  подлежит») — кнопка `…/conditiontype/change`. Результат — из справочника градаций
  состояния (влияет на переоценку/решения по залогу).
- **Находки.** ⚠️ Заголовок поля «**B**ИД РЕЗУЛЬТАТА» — латинская «B» (дубль дефекта P12
  resulttype, E2E-07). Справочник результата содержит «-» (мусорная градация).

### Стадия 10 — Погашение (payment) + логика расчёта остатка
- **Цель.** Принять платёж и увидеть, как пересчитывается остаток долга.
- **Алгоритм.**
  1. Карточка кредита → вкладка **ПЛАТЕЖИ** → «Добавить платеж»
     = `/…/loan/{l}/payment/0/save`.
  2. Форма: **ДАТА** (`paymentDate`) · **ОБЩАЯ СУММА\*** (`totalAmount`) · **ОСН.СУММА\***
     (`principal`) · **ПРОЦЕНТЫ\*** (`interest`) · **ШТРАФЫ\*** (`penalty`) ·
     **ПЛАТ.НОМЕР\*** (`number`, деф. «б/н») · **ВИД ПЛАТЕЖА\*** (`paymentType.id`:
     денежными средствами / переводом долга / **прод. зерном** / приговором суда / др. /
     **принятие имущества в счёт погашения** / разница между договором и приёмкой /
     расходы Фонда) · скрытые `fee`, `exchange_rate`, `in_loan_currency` (валютные кредиты).
  3. `POST /…/payment/save` → платёж в таблице ПЛАТЕЖИ (Действует), редирект на карточку.
  4. Пересчёт остатка — вкладка **РАСЧЕТЫ**: кнопки **Расчет** (`btn_show_loanSummary`,
     показать снапшоты), **Обновить данные** (`bnt_init_dailyCalculator`, daily-калькулятор),
     **Акт сверки** (`bnt_init_manualCalculator`), **Претензия**, **Реестр погашений**,
     **Уведомление**.
- **Логика системы.** ⭐⭐ **Платёж и расчёт остатка ДЕКУПЛЕНЫ.** Платёж пишется в журнал
  ПЛАТЕЖИ сразу (итог 40 000 осн.сумма), но **остаток считается отдельным снапшотным
  калькулятором**: РАСЧЕТЫ хранит ряд снапшотов «на дату» (`/…/summary/{snapId}/view`),
  которые генерируются калькулятором (daily/manual), а не в момент ввода платежа. Ввод
  суммы платежа разбит по назначению (**осн.сумма / проценты / штрафы** — оператор сам
  раскладывает платёж), плюс поддержка **натуральных видов** (зерно, имущество).
  Валютные поля (`exchange_rate`, `in_loan_currency`) — для переоценки валютных кредитов.
- **Находки.**
  - ⚠️ **E2E-09:** свежий расчёт-снапшот (после ввода платежа 40 000 в дату освоения
    12.07) показал **остаток 100 000 / погашено 0** — платёж того же дня не отразился
    немедленно. Остаток = батч-снапшот, не realtime. Риск расхождения «журнал платежей ↔
    расчётный остаток». В новой системе — **realtime-пересчёт** остатка при вводе платежа.
  - Оператор вручную раскладывает платёж на осн/%/штраф — в новой системе логичнее
    авто-распределение по «очереди погашения» (transactionOrder из условий).
  - Дефолтный номер платежа «б/н» (без номера) — тот же мусор, что в справочниках P12.

### Стадия 11 — Просрочка и начисление (daily-калькулятор)
- **Цель.** Довести кредит до просрочки, увидеть механику начисления процентов и штрафов.
- **Алгоритм.**
  1. Карточка кредита → вкладка **РАСЧЕТЫ** → задать дату расчёта в скрытом `generatesDate`
     (+ `phaseInitDate`) на будущую дату (после срока платежа) → «Обновить данные»
     (`bnt_init_dailyCalculator`).
  2. Калькулятор проходит **по дням** от даты освоения до заданной, генерируя снапшоты
     «на дату» и применяя платежи/начисления.
  3. «Расчет» (`btn_show_loanSummary`) — показать таблицу снапшотов.
- **Логика системы.** ⭐ Ядро расчётного механизма:
  - **Проценты капают ежедневно** от остатка осн.суммы по ставке условий (12%/год): к
    сроку 12.08 накоплено 611,51; к 20.08 — 769,32.
  - После **срока платежа** (12.08 — плановая строка графика) весь непогашенный остаток
    переходит в **просрочку** («Всего просрочено» = полный остаток), и начинает капать
    **штраф** (0,1) — к 20.08 = 1,33.
  - Платёж применяется калькулятором к своей дате: снапшот 12.07 показывает погашено
    40 000, остаток осн.с 60 000. Т.е. **остаток = f(график, платежи, ставка, штраф, дата)**,
    пересчитывается снапшотами, а не в реальном времени.
  - Снапшоты накапливаются (каждый прогон создаёт серию `/…/summary/{id}/view`) —
    это и есть «592 записи расчётов» из P2 на реальном кредите.
- **Находки.** Модель «план + факт + ежедневное начисление» мощная, но **снапшотная и
  ручная** (оператор жмёт «Обновить данные»). В новой системе — фоновый ежедневный
  пересчёт + realtime при платеже. Штраф начисляется автоматически по факту просрочки.

### Стадии 12–13 — Вход во взыскание: процедура + фаза «Претензия» (одним действием)
- **Цель.** Запустить взыскание по просроченному кредиту: создать процедуру взыскания и
  первую фазу «Претензия» на сумму просрочки.
- **Алгоритм.**
  1. Карточка кредита → вкладка **РАСЧЕТЫ** (`#loanSummary`) → кнопка **«Претензия»**
     (`#bnt_init_phase`). **ВАЖНО: только ЖИВОЙ клик мышью** — программный `.click()`
     не срабатывает (E2E-11).
  2. Внизу разворачивается блок **расчёта суммы претензии** = просроченная сумма на
     дату (подтягивается из последнего снапшота РАСЧЕТЫ). Оператор задаёт **дату
     претензии** (взяли 28.08.2026, совпадает с последним расчётом).
  3. **«Сохранить»** → сервер за одно действие создаёт: **процедуру взыскания** +
     **фазу «Претензия»** внутри неё.
- **Логика системы.** ⭐⭐ Вход во взыскание — НЕ через ручную форму
  `/…/collectionprocedure/0/save` (та бажная, E2E-10), а через кнопку «Претензия» из
  расчёта. Одно действие создаёт **два уровня** иерархии:
  - **Процедура** `collectionprocedure-68252` — контейнер взыскания; тип
    **«Взыскание задолженности»**, статус **«В работе отр. отд.»**, дата инициирования
    28.08.2026. Живёт на заёмщике (`/manage/debtor/{d}/collectionprocedure/{id}`), НЕ на
    кредите напрямую.
  - **Фаза** `collectionphase-84374` — 1-е юр-действие внутри процедуры; вид **«Претензия»**,
    статус **«На исполнении»**, **НА СУММУ = 60 929,79** (просрочка = снята из расчёта, не
    вводится вручную). Путь `/manage/debtor/{d}/collectionprocedure/{proc}/collectionphase/{id}/view`.
  - ⭐ **Модель взыскания трёхуровневая:** заёмщик —1:N— **процедура** —1:N— **фаза**
    (—1:N— событие/доп.информация). Сумма фазы **вычисляется из снапшота расчёта**, а не
    задаётся — привязка взыскания к расчётному ядру (стадия 11).
  - Карточка фазы = 5 вкладок (Общ.информация / Фаза взыскания / Кредит / Доп.информация /
    Информация). Кнопки: **Редактировать · Изменить статус · Претензия · Доо талап ·
    Доо талап (Нарын) · Отменить фазу**.
    - «**Претензия**», «**Доо талап**», «**Доо талап (Нарын)**» — **печатные формы**
      (претензия-документ; «Доо талап» = кыр. «Исковое заявление» → это генерация иска).
    - «**Изменить статус**» — двигает фазу по воркфлоу-статусам (~70 статусов из P12).
    - «**Отменить фазу**» — отмена (необратимо, не жмём без нужды).
- **Находки.**

### Стадия 17 — Полное погашение остатка (в ноль) + сирота-округление
- **Цель.** Погасить весь остаток (осн.сумма + проценты + штраф) до нуля.
- **Алгоритм.**
  1. Взять **точный остаток из снапшота РАСЧЕТЫ** на дату платежа (28.08.2026:
     осн 60 000 + проц 927,12 + штраф 2,66).
  2. ПЛАТЕЖИ → «Добавить платеж»: totalAmount + разбивка `principal`/`interest`/`penalty`
     (десятичная **точка**, не запятая), дата, вид. `Сохранить` (PAY-E2E-002).
  3. Пересчёт: РАСЧЕТЫ → задать `#generatesDate` = дата ПОСЛЕ платежа (15.09.2026) →
     запустить daily-калькулятор. **⚠️ Кнопку «Обновить данные» надо триггерить
     `jQuery('#bnt_init_dailyCalculator').trigger('click')`** — нативный/мышиный клик
     через автоматизацию не срабатывал; jQuery-триггер сработал.
  4. «Расчет» → снапшот на 15.09.2026: погашено 100 929,78, **остаток осн/проц/штраф =
     0,00 каждый**.
- **Логика системы.** ⭐ Полное погашение = осн.сумма 100 000 внесена двумя платежами
  (40 000 + 60 000), проценты и штраф — отдельными компонентами того же платежа.
  Калькулятор применяет платёж к остаткам **по компонентам** (осн→проц→штраф). Пересчёт
  снапшотов — по-прежнему **ручной батч** (jQuery-триггер), не realtime.
- **Находки.**
  - ⚠️ **E2E-13** (новый, 🟠): **сирота-округление 0,01**. Снапшот «Всего остаток» /
    «Всего просрочено» = 60 929,**79**, а сумма компонентов осн 60 000 + проц 927,12 +
    штраф 2,66 = 60 929,**78** (расхождение 0,01). После полной оплаты компонентов
    осн/проц/штраф = 0,00, но агрегат «Всего остаток» застревает на **0,01** —
    **непогашаемая** копейка (платить не во что, все компоненты уже 0). Доп.платёж
    PAY-E2E-003 (0,01 в проценты) **не применился** калькулятором (остаток процентов уже
    0) → погашено осталось 100 929,78. ⇒ агрегатный остаток ≠ Σ компонентов — дефект
    округления/агрегации; мешает «чистому нулю» и, вероятно, закрытию по нулевой сверке.
  - ⚠️ **Снапшоты нестабильны:** повторные прогоны daily-калькулятора то показывают
    итог на 15.09 (погашено 100 929,78), то откатывают серию к 28.08 (погашено 40 000) —
    зависит от того, взялась ли `generatesDate` до триггера (E2E-09, усугубление).
    Нужно проверять, что дата встала, перед триггером.
  - Экономически кредит **погашен полностью** (осн.долг 100 000 = 100 000 освоения,
    проценты/штраф закрыты); мешает закрытию лишь 0,01-сирота.

     НОМЕР ДОКУМЕНТА (`resultDocNumber`) · **суммы завершения** (осн/проц/штраф —

     дата закрытия всей процедуры) · СТАТУС ПРОЦЕДУРЫ (`collectionProcedure.procedureStatus.id`).

  2. **Кредит:** карточка кредита → вкладка ОБЩ.ИНФОРМАЦИЯ → кнопка «Статус кредита»

  - Фаза закрывается сменой статуса на «Полное погашение» + `closeDate` + фиксация
    **суммы завершения** (сколько погашено по фазе = 60 929,78). Все 4 фазы закрыты.
  - Процедура **не имеет статуса «закрыто»** в enum (только «в работе/передано …») —
    её «закрытость» = наличие `closeDate`. Заголовок процедуры остаётся «В работе отр.
    отд.» даже после проставления даты закрытия (статус ≠ закрытие).
  - Статус кредита = **отдельный ручной справочник состояния** (`loanState`, 4 значения),

- **Находки.**
  - ⚠️ **E2E-14** (новый, 🟡): статус кредита «Нулевая сверка» выставился **несмотря на
    остаток 0,01** (сирота E2E-13) — смена статуса не проверяет фактический ноль. Кредит
    «закрыт» с ненулевым агрегатным остатком → риск «висящих копеек» на закрытых кредитах.
  - ⚠️ Процедура взыскания не переходит в явный статус «Завершена» (нет такого enum) —
    закрытые процедуры визуально неотличимы от активных по статусу (только по `closeDate`

    взыскания) — сохранить в новой модели.

## Дефекты прогона (→ потом в `notes/qa-findings.md`)

- **E2E-01** 🟡 Автокомплит «ЗАЕМЩИК» на форме `/manage/debtor/0/save`: клавиатурный
  выбор подсказки (↓+Enter) заполняет видимый текст, но НЕ записывает скрытый `owner.id`
  (остаётся 0). Требуется мышиный клик. Риск: сохранение заёмщика без привязки субъекта
  (owner.id=0), если оператор выбрал клавиатурой.
- **E2E-02** 🔵 Село=`title` (мусор-плейсхолдер) — дефолт и в новой записи физлица
  (`/person/add`), не только в старых (подтверждает P8-дефект на этапе создания).
- **E2E-03** 🟠 Форма «Условия кредита» (`/…/term/0/save`): дропдауны ВИД ПЛАВАЮЩИХ СТАВОК
  и ВИД ШТРАФА содержат мусорные записи справочника floatingRate («Ставка АБР ГГГГГГГГ»,
  «тест апрапр олпрлр…», «--», «-») — оператор может выбрать битое значение как базу ставки
  прямо при оформлении кредита. Тот же справочник, что засорён в P11/P12.
- **E2E-04** 🟡 Форма графика (`/…/paymentschedule/0/save`): непереведённые ключи
  **LABEL.ADD.PAYMANTSCHEDULE** (+опечатка PAYMAN**T**SCHEDULE), **LABEL.UPLOAD.FILE**;
  кнопки экспорта графика `label.getPdf`/`label.getExcel` (дубль дефекта P2).
- **E2E-05** 🟡 График вводится **вручную построчно** — нет авто-генерации плана погашения
  из суммы/срока/ставки/частоты. Трудоёмко, риск ошибок. В новой системе — автогенерация.
- **E2E-06** 🟡 Залог: непереведённая вкладка договора **LABEL.COLLATERALAGREEMENT.TAB.
  AGREEMENTREVIEW**; ключи `LABEL.QUANTITY/ESTIMATED_VALUE/COLLATERAL_VALUE`,
  `label.collateral_item.not_active` на вкладке ЗАЛОГ кредита; кнопка печати
  «Форма извещения ( тест )» с «тест» в названии (дубль P2/P4).
- **E2E-07** 🔵 Форма акта обследования: заголовок «**B**ИД РЕЗУЛЬТАТА» — латинская «B»
  (дубль P12 resulttype); справочник результата содержит мусорную градацию «-».
- **E2E-08** 🔵 КРЕДИТЫ преднабраны в форме залога, но НЕ в форме поручительства —
  несогласованный UX двух почти идентичных обеспечительных форм.
- **E2E-09** 🟠 Расчёт остатка **декуплен от платежей**: платёж 40 000 (в дату освоения)
  не отразился в свежесгенерированном снапшоте (остаток 100 000 / погашено 0). Остаток —
  батч-снапшот, не realtime → риск расхождения «журнал платежей ↔ расчётный остаток».
  Требует ручного «Обновить данные»/«Акт сверки» и, возможно, платёж датой ПОСЛЕ освоения.
- **E2E-10** 🟠 Форма процедуры взыскания (`/…/collectionprocedure/0/save`): два разных
  поля — «ПОСЛЕДНЯЯ ФАЗА ПРОЦЕДУРЫ *» и «ПОСЛЕДНИЙ СТАТУС *» — имеют **одинаковый**
  `name=lastPhase` (дубль id/name) → «последний статус» не сохраняется.
  ✅ **ОБХОД:** в реальном потоке эта форма не нужна — процедура авто-создаётся кнопкой
  «Претензия» (стадии 12–13). Но бажный дубль `name` в форме остаётся дефектом.
- **E2E-11** 🟠→ℹ️ Кнопка «Претензия» в РАСЧЕТЫ (`bnt_init_phase`) инертна при
  **программном** `.click()`. ✅ **РАЗРЕШЕНО:** при **живом клике мышью** работает штатно —
  разворачивает расчёт суммы претензии, по «Сохранить» создаёт процедуру+фазу. Т.е. это
  артефакт автоматизации (jQuery-обработчик не ловит синтетический клик), НЕ дефект
  продукта. Для оператора-человека всё работает.
- **E2E-12** 🟠 Форма события фазы (`/…/collectionevent/0/save`): справочник **ВИД
  события** (`eventType.id`) содержит только 2 реальных значения («Извещение о назначении
  судебного процесса», «Судебный процесс»); позиции 3–40 = «**-**» (битый/пустой
  справочник) — прямо в рабочей форме. Плюс куратор/участник события = свободный текст
  (не ссылка на субъект/staff). Подтверждает P12-дефект «виды событий нечитаемы».
- **E2E-13** 🟠 **Сирота-округление 0,01 при полном погашении.** Снапшот расчёта:
  «Всего остаток»/«Всего просрочено» = 60 929,**79**, но Σ компонентов (осн 60 000 +
  проц 927,12 + штраф 2,66) = 60 929,**78**. После полной оплаты компонентов осн/проц/
  штраф = 0,00, а агрегатный остаток застревает на **0,01** — непогашаемая копейка (нет
  компонента-адресата). Доп.платёж 0,01 не применяется (остаток процентов уже 0). ⇒
  агрегат ≠ Σ компонентов: дефект округления/агрегации, мешает «чистому нулю» и,
  вероятно, закрытию кредита по нулевой сверке. В новой системе — согласовать округление
  агрегата с суммой компонентов (банковское округление на уровне итога).
- **E2E-14** 🟡 Смена статуса кредита на «Нулевая сверка» (`/loanState/{loan}/save`) —
  **ручной дропдаун без проверки фактического нуля**: статус выставился при остатке 0,01
  (сирота E2E-13). Кредит «закрыт» с ненулевым агрегатным остатком. Плюс процедура
  взыскания не имеет терминального статуса «Завершена» (закрытость — только по `closeDate`),
  закрытые процедуры визуально неотличимы от активных. В новой системе — валидировать
  нулевой остаток перед закрытием + явный терминальный статус процедуры.
