## 1. Назначение и место в конвейере

Реструктуризация (урегулирование) — управляемое изменение условий действующего кредита
по обращению заёмщика, проходящее регламентный конвейер согласования и завершающееся
новой **версией условий**, вступающей в силу доп. соглашением. Модуль живёт поверх двух
соседних:

| Соседний модуль | Связь |
|---|---|
| Кредит (`mockups/loan-credit/`) | версия условий = новая редакция графика/ставки/срока кредита; счётчик реструктуризаций на кредите |
| Взыскание (`mockups/collection/`) | на время рассмотрения заявки процесс взыскания ставится на **паузу** (шов автоматики, §12) |
| Залог (`mockups/collateral/`) | гейт обеспечения при оформлении меряет покрытие на новых условиях (§10) |

Модуль **не** редактирует кредит напрямую и **не** трогает оси заёмщика (группа
платёжеспособности, риск-категория) — они меняются только своими классификаторами (Р-14).

## 2. Единица работы и якорь (Р-1) — `[ВЫВЕДЕНО]`

**Единица работы — ЗАЯВКА на реструктуризацию**, не кредит и не платёж. Реестр показывает
заявки. Якорь заявки — **ИНН заёмщика**: одна заявка покрывает кредиты **строго одного ИНН**
(зеркало решения «единица = процесс, привязан к одному ИНН» из взыскания).

- Заявка ↔ кредит — многие-ко-многим внутри одного ИНН: одна заявка охватывает несколько
  кредитов; но один кредит не может одновременно состоять в **двух активных** заявках
  (валидация охвата, §3).
- Идентификатор заявки — сквозной `RS-NNNN`.
- Заявка несобственна по отношению к производным (дедлайны, статусы гейтов, покрытие) —
  хранит только owned-факты (даты, реквизиты, визы, выбранные виды, параметры версии),
  всё остальное вычисляется (Р-19, §14).

## 3. Охват заявки (Р-2)

Вкладка «Охват» — таблица кредитов заявки: `кредит · остаток · просрочка (дни/сумма) ·
процесс взыскания (если есть) · признак «есть другая активная заявка»`.

**Валидации охвата:**
1. Добавляемый кредит должен принадлежать **тому же ИНН**, что и заявка. Чужой ИНН —
   запрет, тост.
2. Кредит с **другой активной заявкой** добавить нельзя — запрет, тост (активная =
   стадия ≠ «Закрыта»).
3. Пустой охват блокирует переход со стадии «Анализ» дальше.

Остаток и просрочка — read-only снимки из кредита; **не** редактируются из заявки
(запрет «быстрых правок из зеркал»).

## 4. Справочник видов реструктуризации (Р-3)

Вид — конструктор правил, задающий, **что заявке этого вида разрешено менять** и **какой
пакет документов требуется**. Доступ к конструктору — роль Администратор.

**Поля вида:**
- `название`;
- `разрешённые параметры` — набор чекбоксов: срок · график/периодичность · ставка ·
  льготные периоды · капитализация % · капитализация штрафов · прощение санкций · метод
  расчёта дней · база начисления · обработка нерабочих дней · метод погашения;
- `требуемые документы` — редактируемый список;
- **`режим просрочки`** — обязательный селект, один из двух (§11):
  1. `в новый график + обнуление счётчика дней`,
  2. `просрочка сохраняется со своим счётчиком`;
- `ограничения` — свободный текст.

**Композиция на заявке.** Заявка может нести несколько видов. Тогда:
- **редактируемый набор параметров версии = объединение** разрешённых параметров выбранных
  видов (Р-3 ↔ Р-11). Поля вне объединения видимы, но read-only с подписью «вид не
  разрешает»;
- **требования к пакету документов = объединение** документов выбранных видов (§5, гейт
  стадии 1);
- если выбранные виды несут **разные режимы просрочки** — правило приоритета согласуется
  (открытый вопрос №1, §17; в демо конфликт исключён подбором данных).

Новый вид, созданный администратором в рантайме, немедленно доступен при создании заявки
(без перезагрузки).

**Демо-виды:** «Пролонгация», «Изменение графика», «Льготный период», «Капитализация +
прощение штрафов» (режимы просрочки заданы разными, чтобы показать оба).

## 5. Конвейер стадий (Р-4)

Шесть стадий, визуальная лента в карточке:

```
1 Регистрация обращения → 2 Анализ (30 к.д.) → 3 Комитет →
4 Направлено в Минфин (≤70 к.д.) → 5 Решение Минфина → 6 Оформление / Закрыта
```

Активная стадия подсвечена; у стадий с дедлайном — счётчик (§8). Стадия — **производная**
от owned-фактов (даты фиксаций, исход), не селектор. Смена стадии = регистрация факта
(регистрация обращения, старт анализа, фиксация позиции комитета, отправка письма,
фиксация решения Минфина, регистрация доп. соглашения) → запись в историю (Р-16).

**Гейты переходов** (жёсткие, блокирующие — в отличие от регламентных дедлайнов, §8):

| Переход | Гейт |
|---|---|
| 1 → 2 (в Анализ) | пакет документов полон (§5); иначе доступен только исход «возвращена без рассмотрения» или ожидание докомплекта |
| 2 → 3 (в Комитет) | все затребованные заключения завизированы (§6); охват непуст (§3) |
| 3 → 4 (в Минфин) | зафиксирована позиция комитета + зафиксирована отправка письма (§7) |
| 4 → 5 | зафиксировано решение Минфина с обязательным реквизитом письма (§7) |
| 5 → 6 (Оформление) | исход = одобрено (полностью / с изменёнными условиями) |
| 6 → Закрыта | зарегистрировано доп. соглашение; гейт обеспечения пройден или снят waiver-ом (§10) |

**Исходы** (терминальные для заявки): `возвращена без рассмотрения` · `отказ Минфина` ·
`одобрена` · `одобрена с изменёнными условиями`.

## 6. Заключения (Р-5)

По паттерну `loan-application.html`. По умолчанию два блока-заключанта: **СРМиК** и **ДПО**.
Каждый блок: `текст · виза · дата`. Куратор ОД может **добавить подразделение-заключанта**
на конкретной заявке — это добавляет новое требование к гейту перехода 2 → 3.

- Заключант правит и визирует **только своё** заключение (проверка роли, Р-17).
- Гейт «в Комитет»: `conclGate(app).ok` ⇔ все затребованные заключения имеют визу.
- Добавление третьего заключанта немедленно «размыкает» гейт, пока новое заключение не
  завизировано.

## 7. Шов комитет → Минфин (Р-6) — `[ВЫВЕДЕНО]`

Ключевое: **позиция комитета необязывающая, письмо в Минфин направляется всегда,
решение Минфина — канон.**

Последовательность (вкладка «Комитет / Минфин»):

1. **Позиция комитета** — фиксирует Секретарь комитета: `целесообразно` / `нецелесообразно`.
   Позиция — рекомендательная, **не** определяет исход.
2. **Авто-дедлайн письма** — 5 **рабочих** дней на письмо в Минфин, якорь = дата фиксации
   позиции (§8). Письмо направляется **при любой позиции** (в т.ч. «нецелесообразно»).
3. **Фиксация отправки письма** (Куратор ОД) → стадия «Направлено в Минфин»; включается
   автоматика паузы взыскания (§12) и дедлайн решения ≤70 к.д.
4. **Фиксация решения Минфина** (Куратор ОД) — **обязательный реквизит письма Минфина**
   (номер/дата); без реквизита кнопка не активна (`minfinGate`). Решение: `одобрено` /
   `одобрено с изменёнными условиями` / `отказ`.
   - `одобрено с изменёнными условиями` → при фиксации открывается редактирование условий
     версии с пометкой источника «условия Минфина» (то, что утвердил Минфин, а не то, что
     просил заёмщик);
   - `отказ` → исход «отказ Минфина», автоматика снятия паузы и перевод в высокий риск (§12).

## 8. Регламентные дедлайны (Р-7)

Дедлайны стадий — **производные**, считаются от даты-якоря стадии. Просрочка регламентного
срока = **красный флаг + запись в историю, НЕ блокировка**. (Регламентные сроки — мягкие;
жёсткие — только гейты переходов §5.)

| Дедлайн | Лимит | Якорь | Тип дней |
|---|---|---|---|
| Анализ | 30 | дата регистрации обращения | календарные |
| Письмо в Минфин | 5 | дата фиксации позиции комитета | рабочие |
| Решение Минфина | 70 | дата отправки письма | календарные |

`deadline(stage) = { limitDays, anchorDate, remaining, overdue }`; `remaining` и `overdue`
пересчитываются от «сегодня» демо. Реестр красит строку при `overdue`. При переходе стадии
факт просрочки (если был) фиксируется записью в историю.

## 9. Новые условия и diff (Р-11)

Вкладка «Новые условия» — параметры новой **версии** условий кредита:
`срок · ставка · периодичность · метод расчёта дней · база начисления · льготные периоды ·
обработка нерабочих дней · привязка последнего платежа · метод погашения (включая
"Индивидуальный")`.

- Редактируемые поля ограничены **объединением разрешённых параметров выбранных видов**
  (Р-3). Прочие поля видимы, но read-only, подпись «вид не разрешает».
- Кнопка «Рассчитать график» → черновик графика (демо: детерминированный расчёт из
  параметров, без автогенерации «магией» — показывается таблица платежей версии).
- **Diff-таблица**: старая версия и новая рядом; изменённые параметры подсвечены.
- Версия существует в состоянии `проект` до регистрации доп. соглашения (§10), затем
  `действует`. Ретроактивных пересчётов нет — **версия действует только вперёд**.

## 10. Оформление и гейт обеспечения (Р-8, Р-15)

Стадия «Оформление» работает с версией-проектом.

**Гейт обеспечения (Р-15).** Панель проверки залога на новых условиях:
`coverGate(credit, version) = { cover, required, coverOk, ageOk, insured, ok }`, где

- `coverOk` ⇔ покрытие (залоговая стоимость / остаток на новых условиях) ≥ `requiredCover`;
- `ageOk` ⇔ возраст техники + новый срок не выходит за предел (для движимого залога);
- `insured` ⇔ страховка действует на новый срок;
- `ok = coverOk && ageOk && insured`.

Недостаточность (`!ok`) → красный флаг; кнопка «Зарегистрировать доп. соглашение» **заблокирована**.
Разблокировка — модалка **«Решение комиссии по залогу»** с **обязательным основанием**;
waiver пишется в историю (Р-16) и снимает блокировку для этой заявки.

**Регистрация доп. соглашения (Р-8).** Реквизиты `номер + дата` — **оба обязательны**
(`dsGate`). При регистрации:
- версия → `действует`; дата вступления = дата доп. соглашения;
- выполняются операции на дату вступления (§11);
- счётчик реструктуризаций кредита +1 (Р-14);
- стадия → «Закрыта», исход зафиксирован; запись в историю.

## 11. Операции на дату вступления (Р-9, Р-10)

При активации версии, по настройке выбранных видов, выполняются **документированные**
операции (никаких ретро-пересчётов; всё — только вперёд):

1. **Капитализация (Р-9)** — операция «перенос начисленных %/штрафов в тело долга».
   Показывается сумма **до/после**. Отдельно капитализация % и капитализация штрафов
   (два независимых флага вида).
2. **Прощение санкций (Р-9)** — операция списания с **основанием**; реализуется
   **компенсирующей записью**, не удалением (Р-16).
3. **Режим просрочки из вида (Р-10)** — один из двух:
   - `(1)` включить в новый график + **обнулить счётчик дней просрочки**;
   - `(2)` просрочка живёт дальше со своим счётчиком.
   Эффект показывается в карточке явно (счётчик дней до/после).

## 12. Автоматика швов (Р-12, Р-13, Р-14)

**Постановка (Р-12).** Переход заявки в «Направлено в Минфин» → блок «Автоматика» в карточке:
- «Пауза поставлена в процессе взыскания №… (scope: **весь процесс**, дедлайн **70 к.д.**)»;
- «Оверлей подавления **181-го дня** поставлен на кредит …»;
- мини-зеркало процесса взыскания — read-only панель с баннером Паузы и ссылкой на
  заявку-источник (в макете — реальная внутристраничная навигация, не мёртвый href).

**Снятие (при отказе Минфина).** Фиксация отказа → события снятия «**на следующий
календарный день**»: «взыскание возобновлено с фазы …», «ночной пересчёт категории →
высокий» (перевод в высокий риск — следующим календарным днём).

**Пеня во время паузы (Р-13).** Info-plate: «Пени во время паузы **продолжают начисляться**.
Пауза останавливает **меры**, не **обязательства**.» Пауза не замораживает начисление.

**Оси заёмщика (Р-14).** Info-plate: «Оси заёмщика **не трогаем**: группа платёжеспособности
и риск-категория меняются только своим классификатором (ночной пересчёт). Модуль
реструктуризации меняет лишь **счётчик реструктуризаций на кредите (+1)**.»

## 13. История и аудит (Р-16) — `[ВЫВЕДЕНО]`

История заявки — **append-only** timeline. Записи **не удаляются и не переписываются**.
Фиксируются: смены стадий, просрочки регламентных дедлайнов, визы заключений, автоматика
паузы/снятия/оверлея, waiver комиссии по залогу, операции капитализации/прощения,
активация версии, реквизиты доп. соглашения и письма Минфина.

Отмена/корректировка выражается **новой компенсирующей записью**, а не правкой прошлой
(прощение санкций — частный случай этого правила, §11).

---

## 14. Зеркальная дисциплина (Р-19) — `[ВЫВЕДЕНО]`

**Ни одно производное значение не хранится** — всё вычисляется в рантайме из owned-состояния:
счётчики дедлайнов (`deadline`), статусы гейтов (`packetGate`, `conclGate`, `minfinGate`,
`coverGate`, `dsGate`), покрытие обеспечения, стадия (`stageOf`), исход-производные.
В данных заявки нет полей `stage`, `deadlineRemaining`, `gateOk`, `cover` — только факты,
из которых они считаются. Это гарантирует, что зеркала (мини-панель взыскания, diff,
реестр) не расходятся с истиной.

## 15. Роли и права (Р-17)

Переключатель ролей в шапке. Права проверяются `canX(role, action, ctx)`-функциями; чужое
действие заблокировано с тостом-объяснением.

| Роль | Может |
|---|---|
| **Куратор ОД** | ведёт заявку, охват, свод; добавляет заключанта; фиксирует отправку письма и решение Минфина; редактирует условия версии; оформляет доп. соглашение; инициирует waiver |
| **Заключант СРМиК** | правит и визирует только заключение СРМиК |
| **Заключант ДПО** | правит и визирует только заключение ДПО |
| **Секретарь комитета** | фиксирует позицию комитета |
| **Администратор** | ведёт справочник видов (конструктор) |
| **Наблюдатель** | только чтение; все действия заблокированы с тостом |

## 16. Демо-данные (Р-18)

Обязательные цепочки в стартовом состоянии:

1. **Полный путь.** Заявка по 2 кредитам одного ИНН, виды «Пролонгация + Капитализация»,
   прошла все стадии, версия активирована доп. соглашением. В истории — вся цепочка,
   включая автоматику паузы/снятия и операции капитализации; на одном из кредитов был
   красный гейт обеспечения, снятый waiver-ом комиссии.
2. **Отказ Минфина.** Заявка в исходе «отказ»; история показывает постановку и снятие
   паузы/оверлея, возобновление взыскания с сохранённой фазы, перевод в высокий риск
   следующим днём.
3. **Возврат без рассмотрения.** Неполный пакет, исход на первой стадии.
4. **Активная заявка в работе.** Стадия «Анализ», дедлайн 30 к.д. частично израсходован;
   одно заключение завизировано, второе — нет (гейт комитета виден вживую).
5. **Изменённые условия.** Кредит с исходом «одобрена с изменёнными условиями» в истории;
   счётчик реструктуризаций = 2 на одном из кредитов (лимита повторов нет — см. №17).
