# `users.html` — сопроводительная записка

Мокап модуля управления пользователями АСУБК. Один самодостаточный файл, vanilla JS,
без сети и без `localStorage` — состояние живёт в памяти на время сессии. Источник
истины по логике — `ASUBK-users-logika.md` (спека главнее промпта).

Открывается двойным кликом. В шапке переключается **пользователь** (не роль) и сетевой
контекст «в офисе / вне офиса» — роли, scope и сеть вычисляются из модели (§4.1).

## Реализованные экраны (§4)

| Экран | Что внутри |
|---|---|
| **Пользователи** | грид с фильтрами (подразделение/статус/роль/тип/поиск); тулбар Создать · Изменить · Заблокировать · Разблокировать · Отключить · Сбросить пароль · Завершить сессии — видимость/блокировка по `resolvePermission`. Карточка: **Общая информация · Роли · Замещения · Сессии · История** |
| **Роли** | грид (код · наименование · описание · globalScope · назначений · статус). Карточка: **Общая · Матрица разрешений** (дерево функций × `view/create/edit/delete/execute`, правки в память) **· Политики по подразделениям** (изъятия, сравнение базовых/эффективных прав) **· Пользователи с ролью** |
| **Замещения** | грид + **форма создания с живой валидацией V1–V7** |
| **Сетевые политики** | список политик (IP/CIDR, MAC) + привязка `функция×действие → политика`, реально меняет гейтинг |
| **Журнал аудита** | грид только для чтения, фильтры (время · ФИО/логин · отдел · действие). Ни одной кнопки правки/удаления |
| **Проверка прав** | «рентген» модели: эффективные роли/scope + дерево функций × действий с итогом `resolvePermission` (цветовая маркировка `reason`) + строчный доступ на реальных записях |

## Ядро

Один чистый вычислитель `resolvePermission(user, func, action, record?, ctx)` → `{allowed, reason, detail}`.
Порядок 4 независимых измерений строго по §7:
**функциональное** (роли ∪ роли замещения − изъятия политики; deny нет) →
**строчное** (globalScope ∨ подразделение+дерево ∨ кураторство свой+замещаемого ∨ владелец ∨ принадлежность объекта) →
**сетевое** (функция×действие → политика) →
**контроль** (§6, отложен, `/* SLOT */`, сейчас `true`).
Весь гейтинг UI идёт только через неё (`permBtn`, дерево «Проверки прав», карточки).

## 7 сценариев приёмки — «где нажать»

Панель **«Демо-сценарии»** вверху экранов «Пользователи» и «Проверка прав» — кнопка
«показать →» проигрывает каждый.

1. **Роль ≠ scope.** Проверка прав → переключить пользователя `a.asanov` (филиал) ↔ `b.bekova`
   (головной). Одна роль SPEC, разный scope; у головного `Заявка · Подтвердить комплект` = «Доступно»,
   у филиала = «Вне scope».
2. **Политика сужает.** Проверка прав под `a.asanov` (Ф-Север) vs `d.duishenov` (Ф-Юг): строка
   `Заявка · Отправить в комиссию` — «Доступно» у первого, «Изъято политикой» у второго.
3. **Замещение.** Переключиться на `s.sadykov` → в scope появляются кураторские кредиты Осмонова
   (rec1, rec2) по активному замещению S1; действие под замещением в аудите — «замещая Осмонова О.».
4. **Валидации V1–V7.** Замещения → «Создать замещения». Меняйте поля: одно подразделение (V1),
   пересечение периодов (V2/V3), роли не из набора замещаемого (V4), неактивная УЗ (V5), сам себя (V6)
   — блокируют; V7 («четыре глаза») — предупреждение, кнопка «Всё равно создать (продавить V7)»,
   продавливание пишется в аудит.
5. **Сетевой контроль.** Тумблер «Вне офиса» → на «Проверке прав» критичные execute (списание, снятие
   залога, назначение ролей) становятся «Блокировано сетью» — не исчезают.
6. **Жизненный цикл УЗ.** Карточка `n.nurlanov` (уже Заблокирована после 5 попыток) и `e.erkinov`
   (Отключена, уволен — «Разблокировать» неактивна). В любой карточке: «Имитация неудачного входа»
   (5× → автоблок), «Уволить сотрудника» (→ Отключена).
7. **Аудит append-only.** Журнал аудита под `admin` — кнопок правки/удаления нет; собственные действия
   администратора (создание УЗ, назначение роли, сброс пароля, продавливание V7 …) присутствуют.

## Итог smoke-теста (Node + jsdom, headless)

- **Логика (`smoke.mjs`): 24/24.** Контрольные `reason`: `no-role`, `denied-by-policy`, `out-of-scope`,
  `blocked-by-network`, `ok`; headOffice-, кураторство-по-замещению-, техническая-пустой-scope-,
  членство-в-комиссии-случаи; аудит пополняется и записи не мутируются (`Object.freeze`, включая сид-историю);
  автоблокировка залогирована от системы (`actorAccountId=null`).
- **Рендер (`smoke2.mjs`): 75/75.** Все 6 экранов под всеми 12 УЗ, все вкладки карточек УЗ и ролей,
  открытие форм, все демо-сценарии — без исключений.

## Adversarial-верификация (10 измерений спеки, скептик-проверка каждой находки)

10 параллельных ревьюеров били по спеке (чистота ядра, роль≠scope, политика, замещение, V1–V7,
сетевой гейтинг, аудит append-only, ЖЦ УЗ, scope-математика, вокабуляр/данные); каждая находка
проходила скептика-верификатора. Итог: **8 измерений чисто**, 2 подтверждённых дефекта (minor+cosmetic),
**оба исправлены и покрыты smoke-тестом**:
- автоблокировка (5 попыток) писалась актором залогиненного администратора, а не системы → `actorAccountId=null`;
- сид-записи журнала аудита не были заморожены (только рантайм) → заморожены при инициализации.

## Открытые вопросы (ОВ-1…ОВ-5)

Не влияют на работоспособность мокапа:
- **ОВ-1** (сколько уровней контроля) — слой контроля отложен (§6), ветка-заглушка `/* SLOT */`.
- **ОВ-2** (реализуемость MAC) — MAC учтён как демо-параметр политики; по HTTP не передаётся (§5.3).
- **ОВ-3** (2FA отложена) — вне модели входа мокапа.
- **ОВ-4** (где ведётся кураторство) — в мокапе `CURATORSHIPS` как отдельный реестр; модуль его читает.
- **ОВ-5** (значения политик паролей/попыток/таймаутов) — демо-параметры (порог попыток = 5).
