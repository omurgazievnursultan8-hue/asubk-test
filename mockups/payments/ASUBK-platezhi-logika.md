# АСУБК — логика учёта и сверки платежей (спецификация)

> Standalone-спек по образцу `ASUBK-zalog-logika.md` и `ASUBK-vzyskanie-logika.md`.
> Синхронизируется в мастер-документ `ASUBK-analiz-mokapov.md` как
> **секция 14. Учёт и сверка платежей**.
> Фиксирует решения, принятые по семи архитектурным швам платёжного блока.

---

## 1. Назначение и место в конвейере

Модуль отвечает за путь денег: **поступление → разнесение по долгу → сверка с
внешним источником истины → заморозка → акт сверки.**

Базовое требование ТЗ («Модуль учёта и сверки платежей»): сверка платежей · ввод
результатов сверки · корректировка платежей с указанием основания; при
положительном результате сверки платёж **фиксируется**; снятие фиксации — только
по спец. роли, с обязательной повторной сверкой.

Источник истины — **Центральное казначейство (ЦК)**, внешняя организация.
Это определяет всю архитектуру: система не является первичным регистратором денег,
она сопоставляет свой учёт с внешним реестром.

### Три канала поступления

| Канал | Как попадает в АСУБК | Подтверждение ЦК |
|---|---|---|
| **Ручной ввод** | Куратор вносит по списку от ЦК (почта) | Список ЦК **и есть** основание ввода |
| **Шлюз погашения** (новое) | Мобильный кошелёк → платёж появляется **мгновенно**, автоматически | Приходит **позже**, отдельно |
| **Импорт реестра ЦК** (перспектива) | Интеграция с ЦК → строки реестра грузятся в систему | Сам реестр |

Шлюз — источник главной сложности модуля: **деньги действуют до подтверждения**,
и при неподтверждении требуется откат с пересчётом задним числом.

## 2. Две сущности: ПЛАТЁЖ и ЗАПИСЬ РЕЕСТРА ЦК (шов №1)

**Решение: не схлопывать.** Платёж (наш учёт) и запись реестра ЦК (внешний факт) —
разные объекты. Сверка — это **сопоставление** между ними, и расхождение есть
состояние, при котором пары нет. Схлопывание в одну запись со статусом делает
исходы 3 и 4 (ниже) невыразимыми.

### ПЛАТЁЖ (наш учёт)

| Поле | Примечание |
|---|---|
| `id`, `number`, `date` | **Дата = дата фактического поступления денег**, не дата ввода |
| `bindDate` | Дата привязки к кредиту (может быть много позже `date` — см. §7) |
| `amount`, `currency`, `rate` | При валютном — курс |
| `source` | `manual` \| `gateway` \| `cbk_import` |
| `kind` | Вид погашения (справочник: денежные средства, имущество в счёт погашения, по решению суда, по графику и др.) |
| `creditId`, `trancheId` | Привязка |
| `gatewayAccount` | Для шлюза: счёт зачисления (ОД / % / штрафы) — **намерение заёмщика** |
| `allocation[]` | Разнесение (см. §5) — отдельная структура, не поля платежа |

### ЗАПИСЬ РЕЕСТРА ЦК (внешний факт)

Загружается (файл из почты либо интеграция), **не редактируется** — только
перезагружается и пересопоставляется. Поля: номер платёжного документа ЦК, дата,
сумма, плательщик, **примечание** (свободный текст).

**Идемпотентность повторной загрузки.** Реестр за один период может прийти
дважды (уточнённый, исправленный). Ключ дедупликации: **номер платёжного документа
ЦК + дата + сумма**. Без него уточнённый реестр задваивает платежи.

## 3. Сверка: четыре исхода, а не два (шов №2)

| # | Исход | Действие |
|---|---|---|
| 1 | **Совпало 1:1** | Платёж готов к заморозке |
| 2 | **Расхождение реквизитов** (сумма / дата / кредит) | Корректировка с указанием основания |
| 3 | **Есть у нас, нет в ЦК** | Ошибочно внесённый куратором платёж **либо** платёж по шлюзу, не дошедший до ЦК → ветка сторно (§4) |
| 4 | **Есть в ЦК, нет у нас** | **Невыясненное поступление** → буфер (§7) |

Исходы 3 и 4 не принадлежат ни платежу, ни записи ЦК по отдельности — это
состояния **сопоставления**. Отсюда три независимые оси (по образцу трёх осей
взыскания, §«Три оси» в `ASUBK-vzyskanie-logika.md`):

1. **Регистрация** — как платёж попал в систему;
2. **Сопоставление** — подтверждён ли ЦК (`ожидает` → `подтверждён` → `не подтверждён`);
3. **Заморозка** — можно ли редактировать (§8).

**Никогда не схлопывать в единый `status`.**

### Ключ сопоставления: его нет

**Констатация as-is:** формального ключа между платежом и кредитом не существует.
Разнесение идёт **по примечанию** платёжного документа, где обычно (но не всегда)
указаны ФИО и номер кредита.

**Следствие: автопривязка запрещена.** Система **предлагает**, куратор
**подтверждает**. Уровни уверенности:

| Что найдено в примечании | Поведение |
|---|---|
| Точное совпадение номера кредита | Предложение с подтверждением в один клик |
| Только ФИО (нечёткое, возможны однофамильцы) | Список кандидатов на выбор |
| Ничего не распознано | **Невыясненное поступление** (§7) |

Ошибочная автопривязка чужого платежа обходится дороже, чем ручная работа.

## 4. Шлюз погашения: платёж действует до подтверждения (шов №3)

**Решение: платёж по шлюзу считается сразу**, датой операции — уменьшает долг,
останавливает начисление. Но если ЦК не подтвердит — **сторнируется с полным
пересчётом**.

### 4.1. Сторно ≠ удаление

Платёж остаётся в системе со статусом `сторнирован` + основание + кто/когда.
Удаление недопустимо: заёмщик деньги отправил, они где-то есть, а следа нет.

### 4.2. Таймаут подтверждения

- Параметр **`payment.gateway.confirmTimeoutDays`** — настраиваемый (модуль
  управления объектами системы / параметры расчёта).
- **Хранится с историей изменений.** Платёж отменяется по значению,
  действовавшему **на дату операции**, а не по текущему. Иначе смена настройки
  задним числом переигрывает уже закрытые платежи.
- Отсчёт — **от даты операции**, не от даты загрузки реестра ЦК.
- По истечении — **система автоматически** переводит платёж в сторно, запускает
  каскад пересчёта, помечает спорную пеню (§4.4).

### 4.3. Уведомления — три точки, не одна

| Точка | Кому | Зачем |
|---|---|---|
| **Предупреждение до сторно** (за N дней до таймаута) | Куратору | Шанс разобраться с ЦК до автоотмены |
| **Факт сторно** | Куратору + бухгалтерии + заёмщику | Долг восстановлен, есть спорная пеня |
| **Позднее подтверждение** (ЦК подтвердил **после** автосторно) | Куратору | Требуется восстановление платежа |

**Позднее подтверждение — не гипотетика**, а прямое следствие почтового обмена с
ЦК. Значит статус платежа **не терминален**:

```
ожидает → сторнирован по таймауту → подтверждён (восстановлен)
```

Восстановление гоняет каскад пересчёта **в обратную сторону** и **автоматически
снимает спорную пеню** — комиссия не нужна. Комиссия нужна только когда платёж
так и не подтвердился.

### 4.4. Спорная пеня

Сторно задним числом ретроспективно проваливает заёмщика в просрочку. Но
неподтверждение может быть сбоем шлюза, банка или ЦК — не виной заёмщика.

**Механика:**

1. При сторно пеня за период `дата операции → дата отмены` **вычисляется, но
   помечается как спорная**.
2. Спорная пеня **не входит в требование**: не попадает в претензию, не влияет на
   фазу взыскания, не двигает категорию риска, не блокирует акт сверки.
   Видна в детальном расчёте отдельной строкой.
3. Вопрос уходит на **комиссию** (внешне, вне АСУБК).
4. Решение вносится куратором как **реквизит платежа**, не как отдельный процесс:

| Поле решения | |
|---|---|
| `date` | Дата решения |
| `outcome` | `начислить` \| `не начислять` \| `начислить частично` |
| `amount` | Сумма (для частичного) |
| `protocol` | Номер протокола — **текстом** |
| `reason` | Основание / примечание |
| `enteredBy` | Кто внёс |

5. После решения спорная пеня становится обычной начисленной либо аннулируется
   с основанием. **До решения долг заёмщика по пене не растёт.**

**Почему считаем пеню сразу, а не «не начисляем до решения»:** если комиссия
решит начислить, а расчёт не вёлся — придётся ретроспективно перестраивать всю
цепочку начислений второй раз. Считать и придерживать дешевле и прозрачнее.

**Сознательное упрощение:** повестка / голосование / кворум комиссии в АСУБК **не
моделируются**. Комиссия заседает вне системы, вносится только окончательный
результат. Привязка к `commission.html` **не делается** (см. §11, открытый вопрос).

## 5. Разнесение платежа: трёхуровневая матрица (шов №4)

Очередь погашения — **не один список**, а перебор ячеек матрицы. В `loan-credit.html`
она сейчас схлопнута в четыре колонки (ОД / проценты / пеня / комиссия) — этого
недостаточно.

### Уровень 1 — СЛОЙ (по решениям суда)

> **По дате решения, от старого к новому. Свободный остаток — последним.**

### Уровень 2 — КАТЕГОРИЯ (внутри слоя)

Комиссия · пеня · проценты · основной долг. **Последовательность — реквизит
договора**, не константа в коде (ТЗ: поле «Очередь погашения» в графике +
справочник видов платежей). Настройка на уровне кредитной программы с
возможностью переопределения в договоре.

### Уровень 3 — СРОЧНОСТЬ (внутри категории)

**Просроченное → накопленное → текущее.**

Иначе платёж закроет будущий период, а просрочка останется — и заёмщик
по-прежнему в просрочке, заплатив «по графику». «Ранее накопленные проценты» и
«начисленные проценты» — **разные уровни приоритета**, а не синонимы просроченного
(это уже отражено в колонках `DR_COLS` детального расчёта).

Платёж «стекает» по матрице, пока не исчерпается. Остаток → переплата (§6).

### 5.1. Приоритет исполнительного листа над дефолтом

Правило «по дате решения» — **дефолт, а не жёсткий закон.** Деньги от ПССИ
приходят с привязкой к конкретному ИЛ. Если исполнительное производство идёт по
решению №2, а по дате первым стоит решение №1 — автоматика уведёт деньги не туда,
и лист по №2 не закроется.

**Правило:** при наличии указания на конкретный ИЛ (назначение платежа, реестр
ПССИ) слой определяется **по нему**, с фиксацией отклонения от дефолта и
основания.

### 5.2. Ручное переразнесение

Автоматика **предлагает** — куратор вправе **переопределить**. Это ровно та
«корректировка платежа с указанием основания», которую требует ТЗ. Расхождение с
авторасчётом остаётся в следе (audit).

## 6. Слои долга: несколько решений суда на один кредит (шов №5)

Иски подаются **и на просроченную часть, и на всю сумму**; **повторные иски
бывают**. Значит: **один кредит → N решений суда**, каждое со своей суммой, своим
режимом и своим исполнительным листом. Это то же M:N, что уже принято во
взыскании, всплывшее на уровне начислений.

### 6.1. Режим начисления зависит от ДАТЫ решения

| Дата решения | Проценты | Штрафы / пеня |
|---|---|---|
| **до 05.09.2025** | начисляются | **приостановлены** |
| **с 05.09.2025** | **приостановлены** (на сумму решения) | **приостановлены** (на сумму решения) |

**Хранится таблицей режимов с датами вступления, не зашивается в расчёт.**
Появится следующая норма — добавится строка, а не правка кода. Режим определяется
датой решения и **«прилипает» к нему навсегда**: старые решения по старому режиму
не переигрываются.

### 6.2. Структура слоёв

Приостановка — **не флаг на кредите, а свойство части долга**. Формулировка «на
сумму решения» прямо это задаёт.

| Слой | Содержание | Начисление |
|---|---|---|
| **Слой решения №1** | Присуждённая сумма, свой ИЛ, своё исп. производство | По режиму даты решения №1 |
| **Слой решения №2** (повторный иск / увеличение требований) | Своя сумма, свой ИЛ | **По режиму даты решения №2 — потенциально другому** |
| **Свободный слой** | Остаток договорного долга, не втянутый ни в одно решение | Полностью по договору: проценты, график, просрочка, пеня |

> **Один кредит может иметь два разных режима начисления одновременно.**
> Решение от 2024 г. → проценты по его слою идут. Решение от 2026 г. → проценты по
> его слою стоят. Это **норма, а не патология** — расчётное ядро обязано выдержать.

**Начисление идёт по слоям, не по кредиту.** Ночной расчёт обходит каждый слой со
своим правилом. Сумма слоёв = долг.

### 6.3. Судебное требование — представление, а не второе обязательство

Платёж гасит присуждённое, но обязан списаться и из договорного долга — иначе долг
задвоится. Обратное тоже: гасить только договорной — исполнительный лист не
закроется. **Слой решения есть представление части договорного долга**, а не
независимое обязательство. Одна сумма, два ракурса.

### 6.4. Миграция между слоями

Подан иск на просрочку → часть свободного слоя **переезжает** в слой решения и там
замерзает. Заявление об увеличении требований → слой решения **дорастает**.
Это **событие** (дата, основание, сумма), а не молчаливый пересчёт.

### 6.5. Ловушка задвоения — обязательная развилка при вводе решения

Повторный иск может **поглотить** первый (иск на всю сумму после иска на
просрочку) — и одна сумма окажется в двух слоях. **При вводе нового решения
система обязана спросить:**

- это **новый слой** (на ранее не присуждённое), или
- **замещение / поглощение** прежнего решения?

Без этой развилки долг задваивается тихо.

### 6.6. Разрыв «присуждено vs фактически должен»

В режиме «до 05.09.2025» проценты идут, а пеня стоит → сумма решения и фактический
долг **расходятся во времени**. Разрыв должен быть **виден** — он и есть повод для
заявления об увеличении исковых требований (в ТЗ такое заявление предусмотрено).

## 7. Невыясненные поступления и переплата

### 7.1. Невыясненные — висят до опознания

Поступление из реестра ЦК без распознанной привязки → **буфер**. Деньги видны, но
долг **не гасят**. Висят **бессрочно, пока не найдётся хозяин платежа** (срока
автосписания нет).

**Ловушка, требующая явного правила.** Если хозяин найдётся через полгода:

> **Дата платежа = дата фактического поступления, а не дата опознания.**
> `bindDate` — отдельное поле.

Следствие — **обратный каскад пересчёта**, тот же, что при позднем подтверждении
ЦК, но больнее: проценты и пеня за эти полгода снимаются, категория риска
пересчитывается, а меры взыскания, принятые за период, оказываются
**необоснованными**. Опознание обязано запускать полный пересчёт **с уведомлением
куратора и подразделения взыскания**.

### 7.2. Переплата — ОБРАТНАЯ очередь

Остаток после закрытия всего долга не растворяется. Порядок:

> **ОД → проценты → штрафы → возврат заёмщику.**

⚠ **Это обратная очередь** по отношению к обычному разнесению (где сначала
пеня/проценты, потом ОД). Логика: раз переплатил — выгоднее гасить тело.
**Описывать как отдельное правило досрочного погашения**, иначе разработчик увидит
противоречие с §5.

**Границы правила:**
- работает **только по свободному слою**: в замороженный слой решения суда
  переплата уйти не может (там начисление стоит);
- **график не перестраивается.** Досрочное погашение **не меняет ни срок, ни
  размер платежа**. Тело уменьшилось — кредит просто закроется раньше по факту.
  Никакой перегенерации графика.

## 8. Заморозка, крыжение, акт сверки (шов №6)

Реализация требования ТЗ о фиксации платежа. **Крыжение бухгалтерии = заморозка.**

### 8.1. Роли и переходы

| Действие | Кто | Основание |
|---|---|---|
| **Заморозить** (крыжить) | **Бухгалтер** | Положительная сверка. **Однократно** — скрыжённый платёж в крыжение не возвращается |
| **Разморозить** | **Бухгалтер** | **Только по решению комиссии.** Реквизиты решения — обязательное поле; без них защёлка не снимается |
| **Править после разморозки** | **Куратор** | **Только перераспределение** между ОД / процентами / штрафами |
| **Повторно заморозить** | Бухгалтер | Повторное крыжение |

**Что куратор править НЕ может после разморозки:** сумму, дату, номер, кредит.

### 8.2. Почему это хорошая архитектура

Разморозка **не может изменить факт поступления — только его трактовку.** Значит
каскад пересчёта после разморозки **ограничен**: остаток ОД и начисления
пересчитываются (доля тела изменилась), но **дата платежа не двигается**, и
заёмщик **не проваливается задним числом в просрочку**. Обратный каскад из ветки
сторно здесь не срабатывает.

### 8.3. Ошибка в самом факте → сторно, не правка

Перераспределением не лечится: платёж, которого не было; сумма 50 000 вместо
5 000; платёж записан не тому заёмщику.

> **Такой платёж сторнируется целиком** (тем же механизмом, что неподтверждённый
> шлюзовый) **и заводится заново правильный.**
> Право сторно замороженного платежа: **бухгалтер по решению комиссии** — тот же
> маршрут, что и разморозка.

### 8.4. Акт сверки — по инициативе куратора

**Не автоформирование.** Инициирует **куратор**. Жизненный цикл документа:

```
сформирован → согласован бухгалтерией → подписан
```

Связка с заморозкой:
- акт можно инициировать, только если платежи периода **заморожены**;
- подписанный акт — сам является основанием заморозки всего, что в него вошло;
- нулевой акт сверки (при полном погашении) — частный случай, не отдельная
  процедура.

## 9. Шлюз погашения: три счёта и гибридное правило (шов №7)

**As-is задумка:** по каждому кредиту генерируются **три уникальных счёта** —
основная сумма · проценты · штрафы. Заёмщик через кошелёк платит на нужный счёт,
и система по счёту определяет кредит и категорию.

**Конфликт:** три счёта означают, что **заёмщик сам выбирает, что гасить**. Но
очередь погашения (§5) диктуется договором (сначала пеня/проценты), а решения суда
(§6) вообще перебивают всё. Заёмщик кидает на счёт ОД, имея непогашенную пеню —
договор требует иного.

### Принятое решение: ГИБРИД

**Счёт зачисления определяет разнесение только при «чистом» долге:**

- ✔ нет просроченной задолженности ни по одной категории;
- ✔ нет активных решений суда (нет замороженных слоёв);
- ✔ нет спорной пени, ожидающей решения комиссии;
- ✔ нет непогашенной комиссии (если она в очереди раньше).

**Иначе — очередь погашения перебивает счёт:** сумма разносится по матрице
(слой → категория → срочность), независимо от счёта поступления.

Счёт при этом **остаётся в записи платежа как намерение заёмщика**
(`gatewayAccount`): фиксируется, но не исполняется; расхождение видно в следе.

### Обязательное следствие — предупреждение ДО оплаты

Заёмщик в приложении обязан видеть **до** нажатия «оплатить»:

> «У вас просрочка / решение суда — платёж будет направлен на погашение в
> очерёдности по договору.»

Иначе он платит «в основную сумму», деньги уходят в пеню — и это **конфликт, а не
техническая деталь**.

## 10. Каскад пересчёта — сводка

Три события гоняют пересчёт **задним числом**. Все три обязаны быть видимыми
событиями, а не молчаливой перезаписью цифр.

| Событие | Направление | Что пересчитывается | Побочные эффекты |
|---|---|---|---|
| **Сторно платежа** (таймаут / ошибка факта) | Долг **растёт** | ОД восстанавливается, проценты доначисляются, пеня возникает задним числом | **Категория риска** (ночной расчёт) и **фаза взыскания** могут сдвинуться → породить меры, которых не было. Пеня периода → **спорная** |
| **Позднее подтверждение / восстановление** | Долг **падает** | Обратный пересчёт | **Спорная пеня снимается автоматически**, без комиссии |
| **Опознание невыясненного** | Долг **падает** | Пересчёт от `date` (а не от `bindDate`) | Проценты/пеня за период снимаются; **ранее принятые меры взыскания могут оказаться необоснованными** → уведомление взыскания |

**Конфликт с зафиксированной статистикой.** ТЗ прямо требует: правки после
фиксации статистических показателей **не отражаются в ранее зафиксированных
расчётах**. Значит пересчёт живёт в **текущем** состоянии, закрытые периоды
остаются как были, а расхождение между ними должно быть **видимым и объяснимым**
(запись корректировки со ссылкой на сторнированный/восстановленный платёж), а не
тихо расходиться.

## 11. Открытые вопросы

1. **Комиссия — какой орган?** Кредитная комиссия (`commission.html`, председатель
   решает единолично, голоса совещательные) или отдельный орган? Пока привязка
   **не делается**: решение хранится текстовыми реквизитами. Если окажется, что это
   кредитная комиссия — добавится ссылка на решение, структура не сломается.
2. **Погашение комиссии (fee).** В `loan-credit.html` колонка «Комиссия» есть, но
   через шлюз счетов только три (ОД / % / штрафы) — через кошелёк комиссию не
   заплатить. Гасится только через ЦК/ручной ввод, или автоматически закрывается
   из любого поступления при перебивании очереди? — **обсуждение отложено.**
3. **Порядок категорий в очереди погашения** — нормативный, должен быть взят из
   положения (комиссия → пеня → проценты → ОД?). В спеке зафиксирована
   *настраиваемость*, конкретный дефолт не назван.
4. **Гранулярность реестра ЦК:** одна строка = один платёж, или ЦК присылает сводно
   (одной суммой за день по нескольким заёмщикам) и расщепление — на нашей стороне?
5. **N дней предупреждения** до автосторно (отдельный параметр от
   `confirmTimeoutDays`).
6. **Неденежные погашения** (принятие имущества в счёт погашения) — вынесены в
   **модуль по имуществам**. Точка стыковки: имущество принято → порождает
   погашение, которое приходит в этот модуль **уже как сумма с готовым видом
   погашения**; оценка и решение — там.
