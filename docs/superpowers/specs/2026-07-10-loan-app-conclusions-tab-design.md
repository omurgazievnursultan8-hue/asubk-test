## 1. Зачем

Текущая вкладка моделирует упрощённый путь: 4 захардкоженных отдела (риск · залог · юр · кредит),
все обязательны всегда, любой пользователь с ролью «Отдел» правит все четыре блока, заключение —
плоская запись `{verdict, author, date, text}`, гейт отправки в комиссию проверяет лишь
«у всех четырёх есть вердикт». Отрицательное заключение ничего не блокирует, «С условиями» —
свободный текст, жизненного цикла, вложений и истории нет.

Реальный процесс богаче по четырём осям: **состав отделов назначает человек**, **у заключения есть
ЖЦ с отзывом**, **отрицательное блокирует комиссию**, **условия — структурированный список**.

## 2. Процесс (со слов заказчика)

1. **Назначение.** Кредитный специалист назначает отделы — начиная со стадии черновика заявки.
   Набор предзаполнен: отдел рисков и кредитный отдел — всегда; отдел залога — если по заявке есть
   залог. Дефолтные снять нельзя. Сверх них специалист добавляет **любой** отдел из справочника
   оргструктуры (аналитики, юридический, безопасности, мониторинга, …).
2. **Адресация.** Запрос уходит **на отдел**, не на сотрудника: заключение вносит любой сотрудник
   этого отдела. **Срока нет** — дедлайнов и SLA в UI не моделируем.
3. **Внесение.** Доступно только после того, как комплект документов **подтверждён головным
   филиалом** (`_docStats().confirmedMet`). Отдел при этом видит все вкладки заявки на чтение.
4. **ЖЦ заключения.** `Черновик → Внесено`. После «Внести» тело замораживается. Отдел может
   **отозвать** своё внесённое заключение (пока заявка не ушла в комиссию), доработать и внести заново.
5. **Вердикт.** Положительное · С условиями · Отрицательное.
   - `Отрицательное` **блокирует** отправку в комиссию. Выход один: устранить замечания и
     договориться с отделом — отдел отзывает заключение и вносит новое. Отклонения заявки
     «по заключению отдела» и эскалации «вынести вопреки» **нет**.
   - `С условиями` не блокирует: комиссия собирается и принимает окончательное решение,
     видя условия.
6. **Объём.** Одно заключение на отдел на заявку — **всегда**: и при групповом кредите (не по членам),
   и при нескольких предметах залога (не по предметам). К заключению цепляются файлы.
7. **Возврат комиссией.** Если комиссия вернула заявку («требуется доп. информация»), внесённые
   заключения **сбрасываются в черновик**: вердикт снимается, текст, условия и вложения остаются
   и редактируются заново.

## 3. Модель данных

Справочник отделов (плоский, «любой отдел» — в проде это оргструктура):

| key | Отдел | Тип |
|---|---|---|
| `risk` | Отдел рисков | дефолтный, несъёмный |
| `credit` | Кредитный отдел | дефолтный, несъёмный |
| `coll` | Отдел залога | авто по наличию залога; несъёмный, пока залог есть |
| `legal` | Юридический отдел | опциональный |
| `analytics` | Отдел аналитики | опциональный |
| `security` | Отдел безопасности | опциональный |
| `monitor` | Отдел мониторинга | опциональный |

У каждой записи справочника: `key`, `title`, демо-сотрудник `emp`, засев текста `seed`.

Состояние на объекте заявки:

```js
app._concl = {
  assigned: [ { dept, auto:Boolean, locked:Boolean, addedBy, addedAt } ],
  items: { [dept]: {
     status:  'pending' | 'draft' | 'submitted',
     verdict: '' | 'pos' | 'cond' | 'neg',
     text:    '',
     conds:   [ { id, text } ],       // структурированные пункты условий
     files:   [ { name, size } ],     // вложения (в макете — фейковые записи)
     author:  '', date: '',           // автор и дата последнего внесения
     log:     [ { ts, who, action, note } ]
  } }
}
```

- `pending` — отдел назначен, ничего не введено;
- `draft` — есть текст / условия / вложения, вердикт не внесён;
- `submitted` — вердикт внесён, тело read-only.

`status` хранится явно, а не выводится: отзыв и сброс возвратом комиссии переводят `submitted → draft`
с сохранением содержимого, а «пустой черновик» и «pending» различаются только историей.

Действия лога: `assigned`, `unassigned`, `drafted`, `submitted`, `withdrawn`, `reset_by_commission`.

## 4. Роли и права

Свитчер ролей в шапке макета: `🧩 Отдел (заключение)` + **второй селектор** «я из отдела: `[Рисков ▾]`»,
список которого = назначенные отделы текущей заявки. Правится только свой блок; чужие — read-only.

`can(app)`:

| право | условие |
|---|---|
| `assignDepts` | роль «кредитный специалист» && стадия `draft` |
| `editConcl(dept)` | роль «отдел» && `_deptKey === dept` && стадия `draft` && `_docStats(app).confirmedMet` |
| `withdrawConcl(dept)` | то же && `items[dept].status === 'submitted'` |

Роль «Отдел» видит остальные вкладки заявки (документы, залог, график) на чтение — отдельных
ограничений не вводим.

## 5. Раскладка вкладки

Выбран вариант **«сводка + панель назначения + аккордеон карточек»** (эволюция текущего).
Отвергнуты: грид + модалка редактора (чтение всех заключений перед комиссией требует открывать
модалку по каждому) и master-detail (третий уровень вложенности внутри вкладки заявки, ломает печать).

### 5.1 Гейт-баннер

Один баннер, печатается первая сработавшая причина:

| Условие | Баннер |
|---|---|
| комплект не подтверждён ГФ | ⓘ «Заключения вносятся после подтверждения комплектации головным филиалом» (назначать отделы уже можно) |
| `assigned` пуст | ⚠ «Отделы не назначены — назначьте хотя бы один» |
| есть `neg` | ✕ ошибка: «Отрицательное заключение: **Отдел рисков**. Отправка в комиссию заблокирована — устраните замечания, отдел отзовёт заключение и внесёт новое» |
| есть не-`submitted` | ⓘ «Внесено 3 из 5. Ожидаются: Аналитики, Юридический» |
| все `submitted`, без `neg` | ✓ успех: «Все заключения внесены (5/5). Заявку можно отправить на рассмотрение комиссии» + приписка «условий: N — комиссия рассмотрит», если есть `cond` |

### 5.2 Сводка вердиктов

Четыре чипа-счётчика: `✓ Положит. N` · `! С условиями N` · `✕ Отриц. N` · `○ Ожидают N`.
Клик по чипу скроллит к первой карточке с этим состоянием.

### 5.3 Панель назначения

```
Назначенные отделы (5)                       [+ Добавить отдел ▾]
[Рисков 🔒] [Кредитный 🔒] [Залога 🔒 авто] [Юридический ✕] [Аналитики ✕]
```

- 🔒 — снять нельзя; тултип называет причину («дефолтный отдел» / «по заявке есть залог»);
- `✕` снимает отдел. Если у отдела есть `draft`/`submitted` — диалог подтверждения
  «Снять «Отдел аналитики»? Внесённое заключение будет удалено», кнопка подтверждения в красной теме;
- `+ Добавить отдел ▾` — dropdown `DEPT_DIR` минус назначенные; выбор создаёт карточку в `pending`
  и пишет `assigned` в лог;
- отдел залога назначается/снимается **автоматически** при появлении/исчезновении предметов залога,
  с тостом «Отдел залога назначен автоматически — по заявке есть залог».

Кнопки панели видны только специалисту на стадии `draft`; остальным панель показывается read-only.

### 5.4 Карточка отдела

Свёрнутая — одна строка: `▸ Отдел рисков · ТУРСУНОВА Г. А. · 10.07.2026 · 📎2 · усл.3 · [✕ Отрицательное]`.
Для `pending`: `▸ Отдел аналитики · не запрошено · [○ Ожидает]`.
Свой отдел раскрыт по умолчанию. В шапке списка — переключатель `Развернуть все / Свернуть`
(чтение перед комиссией, печать).

Раскрытая read-only: текст (`white-space: pre-wrap`) · «Условия (N)» нумерованным списком ·
«Вложения (N)» (имя · размер · «Скачать») · свёрнутая «⟲ История (N)» со строками
`ДД.ММ ЧЧ:ММ · кто · действие`.

Раскрытая в режиме редактора (свой отдел, роль «Отдел», стадия `draft`, комплект подтверждён ГФ):

- `Вердикт` — селект (пусто / Положительное / С условиями / Отрицательное);
- `Текст заключения` — textarea;
- `Условия` — редактор списка: строка-инпут + `✕`, кнопка `+ Добавить пункт`. Секция видна всегда,
  при вердикте `cond` помечена как обязательная;
- `Вложения` — `+ Прикрепить файл` (в макете — запись из `prompt()` имени, размер фиктивный), `✕` снимает;
- кнопки: **`Внести заключение`** (primary) · `Сохранить черновик` (ghost) · `Очистить`.

Валидация «Внести заключение»:

| Правило | Сообщение |
|---|---|
| вердикт пуст | «Выберите вердикт» |
| текст пуст | «Заполните текст заключения» |
| `cond` && условий 0 | «Добавьте хотя бы один пункт условий» |
| `neg` && длина текста < 20 симв. | «Обоснуйте отрицательное заключение» |

После внесения — тело read-only, плашка `Заключение внесено 10.07.2026 · ТУРСУНОВА Г. А.` и кнопка
**`Отозвать`** (пока стадия `draft`). Отзыв: подтверждение «Отозвать заключение? Заявка перестанет быть
готовой к отправке в комиссию» → `verdict=''`, `status='draft'`, текст/условия/вложения сохраняются,
запись `withdrawn` в лог.

## 6. Стыки

**Гейт отправки.** `conclusionsReady(app)` = `assigned.length > 0` && все `items` в `submitted`
&& ни одного `verdict === 'neg'`.

Порядок причин в `sendGateReason()` — первая сработавшая:

1. обязательный комплект документов не собран;
2. комплект не подтверждён ГФ;
3. ожидается ответ интеграции;
4. **отделы не назначены**;
5. **отрицательное заключение: «Отдел рисков»**;
6. **внесено 3 из 5, ожидаются: Аналитики, Юридический**;
7. покрытие залога.

**Пайплайн стадий.** Узел «Заключения отделов» остаётся третьим (`appStage()` не переписываем).
Добавляется модификатор `stage-err`: при наличии `neg` узел красится в ошибочную тему —
сейчас узлы знают только «пройден / текущий / будущий».

**Видимость вкладки.** Раньше вкладка появлялась со стадии собранного комплекта. Так как назначение
идёт с черновика, вкладка **видна всегда** (`cond` снимается). Это добивает гайдлайн «макс 7 вкладок»
(`TABS`, спек §5) — фиксируется хвостом в TODO; сам лимит в этой задаче не пересматривается.

**Комиссия.** На фазе A («Предварительное изучение») добавляется read-only блок-аккордеон
«Заключения отделов (N)» — те же свёрнутые строки с бейджами; разворот показывает текст, условия
и вложения. Комиссия принимает решение, видя условия, не уходя на другую вкладку.

**Возврат комиссией.** `comRequest()` дополнительно проходит по `items`: каждому `submitted` ставит
`verdict=''`, `status='draft'`, пишет `reset_by_commission` в лог. Содержимое сохраняется.

## 7. Демо-сид

Заявка `З-2026-000105` (комплект подтверждён ГФ) — три состояния карточки сразу на экране:

| Отдел | Состояние |
|---|---|
| Рисков | `submitted`, «С условиями», 2 пункта условий, 1 вложение |
| Кредитный | `submitted`, «Положительное» |
| Залога | `draft` — текст есть, вердикт не внесён |
| Юридический | `pending` |

Заявка со статусом «Отклонена» сидится с `neg` у отдела рисков (демонстрирует красный гейт и
`stage-err`). Заявки за пределами черновика (у комиссии / решённые) сидятся полным набором `submitted`.
