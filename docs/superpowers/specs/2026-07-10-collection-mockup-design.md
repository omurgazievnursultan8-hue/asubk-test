# Мокап модуля «Взыскание задолженности» (to-be, с нуля)

> Дизайн-спек. Дата: 2026-07-10. Мокап: `mockups/collection/collection.html` (новый файл).

## Контекст

АСУБК ведёт жизненный цикл госкредита: решение правительства → кредитная программа →
заявка → комиссия → кредит → при просрочке → **взыскание**. Взыскание — последний
модуль; **экрана в живом стенде нет**, проектируем с нуля.

Это **to-be макет, не клон живого стенда** и не прод-код — образец для команды
разработки. Отсюда: `notes/qa-findings.md` не пополняется (дефектов нет — нет
экрана), рекомендации идут новой секцией «Фаза 9» в `TODO.md`.

**Источники логики:** регламент «Порядок мониторинга, взыскания и урегулирования
задолженности по бюджетным кредитам ОАО "Госфинхолдинг"» (99 пунктов, 17 глав) и
ТЗ, раздел 2.5 «Модуль взыскания задолженности». Оба вне репозитория. Поэтому
доменные решения дублируются здесь и в шапке-комментарии мокапа — ссылаться на
внешний файл нельзя.

**База:** черновик `collection.html`, собранный по тому же промпту (890 строк,
7 вкладок, 4 процесса). Стиль совместим с `mockups/loan-credit/`. Спек описывает,
что в нём укрепляется.

### Отменяющее решение: кардинальность кредит ↔ процесс

Ранняя версия проектной логики фиксировала «процесс привязан к кредиту, не более
одного активного процесса на кредит» (1:N). **Это решение отменено.** Действует:

- Процесс привязан к **одному заёмщику** (ИНН — якорь). Процессов по разным
  заёмщикам не бывает.
- Кредит ↔ процесс — **многие-ко-многим**. По одному кредиту могут идти несколько
  активных процессов одновременно, различающихся **предметом требования**
  (процесс №1 взыскивает просроченную сумму графика, процесс №2 — обращение на
  залог по тому же кредиту). Один процесс может охватывать несколько кредитов —
  но всегда одного заёмщика.
- Связь живёт в отдельной сущности **«Охват процесса»** (процесс ↔ кредит, с
  предметом требования и суммой у каждой пары).

Открытый вопрос ранней версии — «какая группа платежеспособности у заёмщика с
двумя кредитами» — этим снимается: **группа всегда атрибут заёмщика**, процесс её
только читает.

## 1. Три независимых измерения состояния

Ключевой принцип модуля: у процесса **не одно состояние, а три параллельных**.
Свести их в одно поле «статус» — системная ошибка.

| Измерение | Природа | Источник | Правится? |
|---|---|---|---|
| **Категория риска** | измеритель | вычислена из дней просрочки | нет |
| **Группа платежеспособности** | полка ЖЦ заёмщика | атрибут заёмщика | нет (в этом модуле) |
| **Фаза взыскания** | рабочий статус | документ-основание (мера-веха) | нет (только регистрацией меры) |
| **Стадия** | производная | вычислена из фазы | нет |

Категория ничего не отправляет и никого не двигает — она задаёт стадию работы и
служит триггером («на 6-й день начни досудебку», «на 181-й переходи к
принудительному»). Дни считаются от старейшего непогашенного платежа.

## 2. Вычисляемые правила — в коде, не в данных

Черновик хранит производные значения как факты (`cat:'high', catDays:214` парой;
`stage:'Принудительная'` рядом с `phase:'Иск'`). Разработчик видит результат, но
не формулу. Три правила переезжают в код:

```js
const catOf   = days  => days <= 5 ? 'norm' : days <= 180 ? 'mid' : 'high';
const stageOf = phase => PHASES_PRETRIAL.includes(phase) ? 'Досудебная' : 'Принудительная';
const overlaps = proc => /* другие активные процессы с пересечением пары (кредит, предмет требования) */;
```

Из данных процесса **удаляются** поля `cat` и `stage` — остаются только `catDays`
и `phase`. Это и есть «фаза двигается мерой, стадия производная», выраженное
кодом, а не комментарием.

Границы категории: `0–5 дн` норма (техзазор) · `6–180 дн` средний риск ·
`181+ дн` высокий риск.

### `overlaps()` питает два места

1. **Модалка «Добавить кредит в охват»** — предупреждение срабатывает по реальным
   данным, а не висит статичным текстом. Мягкое предупреждение, **не жёсткий
   запрет** (пересечение бывает законным).
2. **Вкладка «Охват»** — плашка над таблицей: «По дог. №56 идёт процесс
   В-2026-000151 (предмет: залог)». Номер — ссылка, открывает тот процесс.

Второе — главное, чего не хватало черновику: кейс M:N лежал в данных, но из
процесса 142 нельзя было узнать, что по тому же кредиту идёт 151.

## 4. Шапка карточки — панель-ИНДИКАТОР, а не форма

Четыре плитки: **Стадия · Фаза · Категория · Группа**. Под каждой — подпись
источника:

| Плитка | Подпись |
|---|---|
| Стадия | `производная от фазы` |
| Фаза | `по документу-основанию: ИСК-77` |
| Категория | `вычислено · 214 дн просрочки` |
| Группа | `атрибут заёмщика` |

Плитка фазы показывает **номер меры**, которая эту фазу установила. Это делает
правило «фаза не редактируется, а следует из журнала мер» наглядным, а не
декларативным. Категория — цветным бейджем (зелёный / жёлтый / красный) + число дней.

Ни одно из четырёх полей не редактируется. Внизу карточки — только «Закрыть»
(см. §10).

## 5. Машина фаз

**Досудебная стадия** (средний риск, 6–180 дн):

```
Претензия → Повторная претензия → Безакцептное списание
```

**Принудительная стадия** (высокий риск, 181+ дн):

```
Извещение → Иск → Решение суда → Исполнительный лист → На исполнении
```

Фаза двигается **только регистрацией меры-вехи**: нельзя перевести в «Иск» без
зарегистрированного искового заявления, в «Исполнительный лист» — без листа.
Апелляции, кассации, частные жалобы, уточнения исковых требований **не порождают
новых фаз** — это меры *внутри* фазы «Иск» / «Решение суда». Машина остаётся
линейной; судебная возня живёт в журнале.

Матрица переходов «что чем подтверждается» — конфигурация, не пользовательские
данные. Виды мер и результатов — пополняемые справочники.

Таймлайн фаз на вкладке «Журнал мер» показывает фазы **своей** стадии: пройденные
(✓), текущая (выделена), будущие (приглушены).

## 6. Не-фазовые механизмы — три РАЗНЫХ, не путать

| Тип | Что это | Фаза | Дедлайн | Цвет баннера |
|---|---|---|---|---|
| **Пауза** | заморозка (реструктуризация, гарантийное письмо) | сохраняется | ≤70 дн, автоснятие | янтарный |
| **Соглашение** | мировое / добровольное исполнение — новый график, наблюдение | сохраняется | нет, есть график | синий |
| **Событие** | смерть, недееспособность — увод вбок в спецпроцедуру | сохраняется | нет | красный |

- **Пауза:** счётчик дней **НЕ замораживается** — тикает дальше; пауза подавляет
  автоматические пороговые действия. Область (scope) обязательна: «весь процесс»
  (реструктуризация) vs «конкретная мера» (гарантийное письмо отлагает только
  безакцепт) — иначе узкая пауза заморозит больше, чем разрешает норма. Выход
  двойной: негативный (отказ / нарушение / истёк дедлайн) → размораживание с той
  же фазы; позитивный (одобрено) → новый график или закрытие.
- **Соглашение — не пауза.** У паузы есть дедлайн, который *пережидают*; здесь —
  новый график, который активно *мониторят* (до 5 лет). Нарушение графика →
  возврат к принудительному исполнению с сохранённой фазы.
- **Событие** — врывается на любой фазе (аналог граничного события BPMN).

Активный оверлей показывается **баннером над плитками шапки** — он меняет набор
доступных действий. Черновик знал пауза/соглашение; **событие как баннер
добавляется**.

## 7. Исходы (терминальные состояния)

| Исход | Условие | Кто решает | Группа |
|---|---|---|---|
| Полное погашение | взыскано деньгами | отдел проблемных кредитов | 5 |
| Принятие имущества | торги не состоялись | **Комитет** | → отчуждение активов |
| Банкротство | нечего взять (ИЛ вернулся пустым) | сектор активов | 3.1 → 3.2 |

Паттерн «серьёзное необратимое действие = решение комитета» повторяется (безакцепт,
принятие имущества, списание, внесудебное обращение) — заложить общим механизмом.

В списке терминальный процесс — приглушённая строка, колонка «Фаза» заменена
исходом.

### Модалки

- **Escape и клик по подложке закрывают** (стандарт репо, P3-R33). В черновике нет ни того, ни другого.
- «Зарегистрировать меру»: раздел · вид меры (~20) · дата · номер · способ
  направления/вручения · результат · назначение. Предупреждение о сдвиге фазы —
  **условное**: показывается, только если выбранный вид меры является вехой.
  Отдельно — предупреждение, что претензия/извещение без подтверждения вручения
  не считается исполненной.
- «Добавить кредит в охват»: пикер кредита + предмет требования. Предупреждение
  «только тот же заёмщик» — статично; предупреждение о пересечении — **по реальным
  данным через `overlaps()`**.

## 10. Решения по объёму (YAGNI)

- **Внизу карточки — только «Закрыть», без OK/Отмена.** Все поля карточки
  read-only (шапка-индикатор; фаза не редактируется; охват, меры, передачи
  правятся своими кнопками и модалками) — сохранять нечего. Единая ИА с
  `commission.html`, где эту пару убрали намеренно (P3-R29). Промпт требовал
  «OK / Отмена»; **это требование отменено осознанно**, иначе разработчик
  прочитает кнопки как «здесь есть режим правки».
- **Роль не меняет редактируемость полей** — редактируемых полей нет. Смена роли
  показывает тост.
- **Подпункты меню** («Досудебный порядок», «Судебный порядок», «Исполнительное
  производство», «Отчуждение активов») — тост «доступен раздел "Процессы
  взыскания"».
- **Кнопки «Поставить на паузу / Соглашение / Событие»** — тост «(демо)», без
  модалок. Три модалки ради тоста — расход без отдачи; форма паузы не несёт
  доменного знания, которого нет в §6.
- **Пересчёт суммы при извещении не выполняется в рантайме.** Показывается
  результат в данных (у процесса 151 сумма = полный остаток) + заметка о правиле:
  регистрация извещения влечёт досрочное наступление всего долга.

## 12. Технические требования

- Один самодостаточный файл: `<style>` в `<head>`, `<script>` перед `</body>`.
  Без фреймворков, CDN, внешних шрифтов. Ванильный JS. Открывается двойным кликом.
- Раскладка: слева боковое меню, сверху шапка-хлебные крошки с переключателем
  роли, в центре список → карточка с горизонтальными вкладками.
- Дизайн-система АСУБК (сверено с `loan-credit.html`): белые поверхности, бордеры
  1px `rgba` синего, сетка 4px, радиусы 4px, высота контролов 36px, системный
  шрифт, 16px тело / 14px мелкое. Синий `#006AF5`, зелёный `#158443`, красный
  `#E03A3A`, янтарный `#B8860B`. Только светлая тема.
- Поля трёх видов через helper-функции: редактируемое (белый фон), read-only
  (серая заливка), лукап-пикер (серая строка + кнопка `•••`). Даты — с иконкой
  календаря. В карточке процесса встречаются только read-only и пикеры (§10);
  редактируемые поля живут в модалках.
- Весь текст, данные и комментарии — на русском.
- Роли для переключателя: Куратор отраслевого департамента · Департамент правового
  обеспечения (ДПО) · Отдел проблемных кредитов · Сектор по работе с активами ·
  Наблюдатель.

- `P9-R2` — три независимых измерения состояния; запрет сливать в одно поле «статус».
- `P9-R3` — машина фаз с матрицей «фаза ← вид меры-основания» как конфигурацией.
- `P9-R4` — журнал мер с обязательным подтверждением вручения для претензий/извещений.
- `P9-R5` — передача дела с подтверждением/отклонением («четыре глаза»).
- `P9-R6` — три раздельных оверлея (пауза / соглашение / событие), фаза сохраняется.
- `P9-R7` — флаг `право_безакцепта` на кредите (проверить: вероятно, отсутствует).
- `P9-R8` — общий механизм «решение комитета санкционирует необратимое действие».
- `P9-R9` — досрочное наступление долга при регистрации извещения (пересчёт суммы).

Точные формулировки и приоритеты — при написании плана.

## 14. Открытые вопросы (вне объёма мокапа)

- **Пени во время паузы** — продолжают ли начисляться финансовые санкции при
  заморозке (реструктуризация / мировое). Бизнес-вопрос.
- **Расширение вкладки обеспечения** кредита на поручительства, гарантии, депозит —
  нужно для автогенерации списка требований обеспечителям (§5 регламента, шаг 2).
- **Флаг `право_безакцепта`** в мокапе кредита — вероятно, отсутствует; требует
  проверки.

Эти вопросы фиксируются в мокапе как заметки, но не решаются им.
