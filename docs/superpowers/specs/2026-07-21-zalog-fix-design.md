## Сквозные контракты (Фаза 1 — до фич)

Делаются первыми, чтобы фичи легли без переделок. Каждый — грепом по всем call-sites за один проход.

### К-1. `requiredCover` меняет сигнатуру (Р-13)

```js
// было: function requiredCover(creditId, cs) -> Number
// стало:
function requiredCover(creditId, cs) -> { req: Number, source: String }
```

- `credit.kmDecision` задан → `req = kmDecision.coverPct/100`, `source = 'решение КМ ' + no + ' от ' + date + ' (П1 §2.6)'`.
- иначе движимый неликвидный в составе → `req = COVER_MOVABLE`, `source = 'П1 §2.4 (движимый неликвидный залог)'`.
- иначе → `req = COVER_LIQUID`, `source = 'П1 §2.3 (ликвидный залог)'`.

**Все вызовы обновить на `.req`.** Известные call-sites: `requiredCover` (1361), `liquidShareApplies` (1367 — сравнение `=== COVER_MOVABLE` → `.req === COVER_MOVABLE`), `coverage` (~1382 `const req = requiredCover(...)`), плюс точки после фикса Д-1/Д-2 (`ncCredits`, `openAlloc`, таблица гейта, триггеры, карточка ДС/договора). **Полный список — грепом `requiredCover(` на старте фазы.** `source` выводится в UI рядом с числом порога везде, где порог показан.

### К-2. Матрица ролей ПОЛ §9.1 (Р-19)

Переименования: `canEdit() → canCurate()`, `canCommission() → canCommittee()`, добавить `canHeadOfDept()`.

| Роль | Полномочия |
|---|---|
| `Куратор отдела залогового обеспечения` | реестры, приём, договоры/ДС, акты, извещения, согласия |
| `Заведующий отделом залогового обеспечения` | + понижение залоговой стоимости, согласование снятия запрета, отказ в замене |
| `Начальник Департамента администрирования кредитов` | заключение о ликвидности при замене (П3 §3.5), подтверждение снятия запрета |
| `Комитет по администрированию бюджетных кредитов` | допуск ниже порога, допуск неравноценной замены, допуск техники сверх лимита §2.5 — только с обоснованием |
| `Наблюдатель` | чтение |

- `<option>` списка ролей (стр.617) переписать.
- Все гварды прав перевести на новую матрицу; `denyComm` и тексты «направлено в комиссию по залогу» → «направлено на рассмотрение комитета по администрированию бюджетных кредитов».
- **Проверка полномочий override:** понижать — заведующий; допуск ниже порога — комитет; куратор и комитет **не** правят залоговую стоимость.

### Фаза 2 — Расчётный слой
- **Р-11.** `guaranteeReq(credit)` (1.0 / 1.2 по валюте). Индекс обеспеченности в `coverage()`: `Σзалог/(amount×req) + Σгарантия/(amount×guaranteeReq) ≥ 1`. Модель `otherSecurity` (объект). Поручительство — вне индекса. Доля ликвида — только от залоговой части. Реализовать `COVER_GUARANTEE`/`COVER_GUARANTEE_FX` (снять TODO). Триггеры: гарантия истекла (high), истекает за 60 дн. (mid).
- **Р-12.** +5 видов в `KINDS` (кредитный портфель, имущественный комплекс, незавершённое строительство, ценные бумаги, право аренды) c `k/liquid/movable/liquidityRank/орган`. `KIND_FIELDS` для каждого. `KIND_MIGRATION` фикс. Удалить комментарий 782.
- **Р-22 (Ш-1).** `SURVEY_MATRIX` read-only (норматив П2 §2.3). Слой `RISK_TIGHTENING` + `SURVEY_MIN_MONTHS`. Эффективный интервал = `max(min, round(матрица × коэф[риск]))`, worst-of. Требует согласования с зампредом. UI — обе величины. Банкроты — без слоёв (Р-3 сохраняется).
- **Р-13-гейт.** Третий гейт регистрации: `borrowerCategory` заполнен + `kmDecision` пуст → блок. Разблокировка — ввод реквизитов решения КМ на карточке договора.

### Фаза 3 — Приём и валидации
- **Р-17.** Поле `securityRole` (основной/дополнительный). Возраст: дополнительный+сверх → жёсткий блок; основной+сверх → решение комитета. Чек-бокс исправного состояния (П1 §3.3) + реквизит акта осмотра.
- **Р-18.** Стоп-лист §3.5 как валидация в `saveNewItem` + на гейте регистрации: `landPurpose` (с/х → блок без права допуска), `wearPct`≥`WEAR_LIMIT`+моральное устаревание, три отдельных подтверждения для ТМЦ, общий чек-бокс оборотоспособности. `PROHIBITED_KINDS` — рядом с каждым пунктом реализующая валидация.
- **Р-15.** Модалка обследования: производные read-only (заёмщик/залогодатель/имущество), обязательное заключение, участники (мультиввод), односторонний акт + подтверждение наличия, фотофиксация (≥1, обязательна). Колонка «Акт» в реестре мониторинга.
- **Р-16.** Поле `custody` (обязательно при регистрации, ПОЛ §6.4 п.7). Право аренды: `KIND_FIELDS` со сроком аренды + валидация покрытия срока кредита (П1 §6.4 п.6). При `у залогодержателя` — место хранения + снятие интервала обследования.

### Фаза 4 — Процессы
- **Р-14.** Гейт замены: равноценность по стоимости, согласие отраслевого департамента (реквизит+чек-бокс), заключение ДАК, ранг ликвидности (Р-6) — четыре проверки. Действие «Отказать в замене» с причиной.
- **Р-20.** Ось запрета: наложение — залогодатель (ПБК п.2.1), мы контролируем и принимаем отметку. Модель `ban.markInOriginals` + `ban.confirmedBy`. Действие «Зарегистрировать наложение запрета залогодателем». `BAN_DEADLINE_DAYS` — пометка «внутренний». Снятие (Р-9) — реквизит акта сверки при «Полное погашение».
- **Р-19-хвост.** Override только вниз (`value ≤ calcPledge`, заведующий, обоснование обязательно). `isOverAbove`/`beyondDiscount` — только для legacy-отображения; П-005 → `legacy:true` + плашка + триггер mid. Маршрут документов ПОЛ §11 — блок на карточке + триггер незавершённого маршрута.
- **Р-23 (Ш-2).** `riskFactorCandidates()` отдельным потоком (не в `triggers()`). Панель «Факторы для комитета» в реестре мониторинга + действие «Направить фактор в комитет». Триггер страховки — строка действия. Шапка п.7 — формула Р-23.
- **Р-21.** Границы модуля — в шапке/справочнике как осознанная граница. Закрыть Р-10.

## Тест-харнесс

- **Стек:** Node + jsdom, внешний раннер. `npm i -D jsdom`. Файл — `mockups/collateral/tests/zalog.smoke.mjs`. Отдельный скрипт запуска (не ломая `playwright test`); проводка команды — по месту (например `node mockups/collateral/tests/zalog.smoke.mjs`).
- **Загрузка:** jsdom с `runScripts:'dangerously'`, `resources:'usable'`; ждать выполнения инлайн-скрипта.
- **Тест-шов (design decision).** Function-declarations (`requiredCover`, `coverage`, `liquidShare`, `guaranteeReq`, …) доступны как свойства `window`. Но `const`-хелперы (`calcPledge`, `pledgeValue`, `coefOf`, `effShare`, `allocatedIn`) и `let COVER_LIQUID`/`const KINDS`/`SURVEY_MATRIX` — **не** свойства window (лексические привязки classic-script). Поэтому в конце скрипта добавить:
  ```js
  if (typeof window !== 'undefined') window.__zt = {
    requiredCover, coverage, guaranteeReq, calcPledge, pledgeValue, coefOf,
    effShare, KINDS, COVER_LIQUID, COVER_MOVABLE, SURVEY_MATRIX, riskFactorCandidates,
    /* + прочее по мере надобности тестов */ };
  ```
  Шов **используется тестами** → не заглушка, правило §1.2 не нарушает. Держать минимальным — только то, что тесты реально дёргают.
- **DOM-тесты:** рендер-точки (нет литерала «120%» при пороге 150% — грепом по `document.body.innerHTML`; блокировки кнопок; предзаполнение полей) — через прямой рендер функций UI в jsdom.
- **Покрытие:** обязательный список — §18 промпта (46 сценариев). Цель ≥45.

## Риски

1. **`requiredCover`-каскад** — самый рисковый. Митигация: сквозной греп `requiredCover(` до и после, тест К-1 сразу.
2. **Роли** — множество гвардов и текстов. Митигация: греп `canEdit\|canCommission\|Комиссия по залогу`, единый проход, тест ролей (§18 п.31–34).
3. **Размер файла** — 294 КБ, длинные строки. Точечные Edit по якорям (номера строк из промпта — валидны на снапшоте, перепроверять грепом перед каждой правкой, т.к. номера плывут после вставок).
4. **jsdom-совместимость инлайн-скрипта** — если скрипт падает в jsdom (нет какого-то браузерного API), тесты не поднимутся. Митигация: ранний smoke «файл загрузился, `window.__zt` есть» до содержательных тестов.
5. **Демо-id коллизии** — таблица соответствия §17→факт в плане, греп существующих id перед вставкой.
