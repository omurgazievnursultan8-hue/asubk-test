## Два потока заявок (фундамент)

**Поток A — по программе (сверху вниз):**
Решение → Программа (конверт условий) → заявка привязана → комиссия одобряет → **сразу формируется кредит**.

**Поток B — индивидуальная / без привязки (снизу вверх):**
1. Вносим заявку, заполняем всё **свободно** (программы/решения ещё нет).
2. Отправляем в комиссию → комиссия одобряет.
3. **Кредит НЕ формируется** (в отличие от A).
4. По итогу выходит **новое решение + кредитная программа**.
5. Заявка **привязывается** к этому решению/программе.
6. Дальше — как поток A → формируется кредит.

Т.е. поток B — заявка сама рождает программу (индивидуальная сделка, которую потом узаконивают).

**Модель связи — гибрид:** программа = конверт; заявка наследует/предзаполняет из него,
но поля остаются в рамках конверта. Аварийный выход из конверта = поток B
(хочешь условия вне любой программы → индивидуальная заявка).

## Топик 1 — Связь «программа ↔ заявка» (закрыто)

### Тип потока определяется точкой входа (не полем)

| Поток | Вход | Гейт доступа | Программа |
|---|---|---|---|
| **A — по программе** | кнопка «Создать заявку» в списке/карточке программы | механизм прав → видны только доступные специалисту программы | зафиксирована входом, наследуется |
| **B — индивидуальная** | кнопка «Создать индивидуальную заявку» в списке заявок | видна только пользователям с соответствующей ролью | пусто на входе, привяжется позже |

- У кого нет роли на индивидуальную — в списке заявок кнопки создания **нет**; стартуют только от программы.
- Никакого запасного дропдауна, никакого неявного поля-драйвера. Ошибиться типом нельзя.
- Отраслевая норма: «product-first origination» (LOS nCino/Finastra/Temenos, Salesforce create-from-parent, Jira create-in-project).
- Права на программы: в системе **будет** механизм «кому что доступно» (подтверждено владельцем).

### Обобщённая линза наследования полей

Наследуемое поле программы бывает:
- **одно-значное → залочено read-only** в заявке;
- **много-значное → обязательный select только из разрешённого** программой.

(Открыто на источнике финансирования: программа может нести 1 или несколько источников.)

### Правка backlog
- **P3-R5 снимается** («сделать программу обязательной» — неверно для потока B).
  Замена: поток определяется входом; в A программа зафиксирована, в B отсутствует и гейтится ролью.
  Обязателен видимый признак типа (баннер/бейдж); в B обязателен Субъект.

## Топик 2 — Мастер «Новая заявка» (закрыто)

**Форма-фактор:** короткий **ядро-мастер, 2 шага** → приземление на детальную.
Документы, залог и прочее — на детальной. Отправка в комиссию гейтится обязательными документами.

**Шаг 1 — Заёмщик** (+ живой баннер режима A/B).
**Шаг 2 — Условия кредита** (ветвится по потоку).

### Карта полей

| Поле | Поток A (по программе) | Поток B (индивидуальная) |
|---|---|---|
| Кредитная программа | убрана из формы — задана входом, показ read-only сводкой конверта | нет |
| Субъект | обязат.; фильтр по whitelist программы | обязат.; свободный lookup |
| Адрес | readonly из карточки Субъекта | readonly из карточки |
| Телефон | предзаполняется, **редактируемый** (маска +996, валидация); в карточке телефона нет | то же |
| Валюта | наследуется из программы (read-only) | свободный select (нужна для суммы) |
| Источник финансирования | 1→read-only · 2+→обязат. select из разрешённых | свободный, обязат. |
| Цель кредита | 1→read-only · 2+→обязат. select из разрешённых | свободно |
| Сумма | Фикс→select разрешённых · Диапазон→[мин,макс]+подсказка «Допустимо: от…до…»; обязат. | свободный decimal >0; обязат. |
| Срок (мес.) | Фикс→select · Диапазон→[мин,макс]+подсказка; обязат. | свободное число >0; обязат. |
| Метод погашения | 1→read-only · 2+→обязат. select из разрешённых | свободный выбор |
| Льготные периоды ×3 | только разрешённые программой, опт-ин | все 3 свободно |
| Доп. информация | опц. textarea (и **убрать из столбцов таблицы списка** заявок) | опц. textarea |

- Вне конверта в A **нельзя** (жёсткий блок; выход = поток B).
- Единица срока — месяцы, подписать в UI.

### Сквозные правки мастера
- eager-валидация → **lazy** (ошибка на blur/«Далее») — закрывает P3-R6.
- границы сумма/срок из конверта — закрывает P3-R3.
- дизайн-система полей required/optional/readonly — закрывает P3-R1.
- живой баннер режима «По программе: <название>» / «Индивидуальная заявка».

## Открыто из сверки с мокапом программы (уехало на детальную/след. топики)

- **Залоговая ветка** — программа несёт флаг **«Требуется залоговое обеспечение»** + виды залога.
  Если ДА → заявка обязана иметь залог нужного вида (вид/оценка) + залоговые документы. Вводится **на детальной**.
  ⚠️ **Требование изменено 2026-07-04:** заявку рассматривает **одна** комиссия. Отдельной **залоговой
  комиссии нет**, второго параллельного статуса (Статус залога) **нет**. Залог как сущность остаётся,
  но проверяет его та же единственная комиссия. Снимает P3-R4(d) (параллельные комиссии).
- **Документы** — программа **задаёт** списки обязательных/опциональных кредитных и залоговых документов.
  Гейт «Отправить в комиссию» берёт обязательный список **из программы** (не хардкод). Закрывает P3-12.
- **Кураторы** (Ответственные сотрудники) — наследуются на **кредит** при выдаче, не в заявке.
- **Шаблоны договоров / печатные формы** — генерация документов заявки/кредита, поздняя стадия.
- Кредитная линия, Вид кредита — наследуются показом в сводке конверта (A).

## Топик 3 — Детальная страница (в работе)

### Групповые заявки (модель определена — 2026-07-04)
Заёмщик по заявке **один**, кредит на выходе **один**. Групповой кредит = один кредит с **несколькими
траншами**: каждый транш ← **член группы**; кредит и транши рождаются **после одобрения** (post-approval),
но заявка заранее несёт список членов (будущие транши) — иначе комиссии нечего рассматривать.
- **Признак групповой задаёт программа** (флаг «групповой кредит» в конверте). Заявка по такой программе —
  всегда групповая, блок «Члены группы» обязателен; по обычной программе блока нет.
- **По члену только: ФИО (субъект) · сумма · дата освоения.** График члена генерируется из даты освоения.
  Всё прочее (ставка, метод, срок, периодичность, льготные) — **общее из программы** на всю группу.
- **Сумма заявки = ∑ сумм членов** (read-only, вычисляется; каждая в рамках конверта).
- UI: **отдельная условная вкладка «Члены группы»** (появляется при групповой программе) — таблица
  ФИО · Сумма · Дата освоения. Вкладка **«График»** — **селектор члена** → ниже график выбранного члена
  (один график за раз). Условность аналогична вкладке «Залог».

### Режим страницы (закрыто)
Режим по умолчанию **зависит от статуса**: Черновик/Новый → сразу **edit**; отправлено/одобрено → **view**.
Кнопка «Редактировать» доступна только когда статус разрешает правку; на Одобрено правки нет вообще.
Закрывает P3-R2 (нет «Просмотра») + P3-10 (правка одобренной).

### Вкладки (закрыто) — было 9, стало макс 7 (условные скрыты)
`Общая информация · Члены группы* · График погашения · Залог* · Документы · Комиссия · История`
- Схлопнуто: 4 документные → одна «Документы»; 2 комиссии → одна «Комиссия» (требование изм. 2026-07-04).
- `*` условные: **Члены группы** — только при групповой программе; **Залог** — только при залоговой.
- Обычная одиночная незалоговая заявка → 5 вкладок (условные скрыты).
- **Шапка/степпер — один раз** над вкладками, не повторять в каждой (закрывает P3-17).

### Вкладка «Документы» (закрыто)
Секции-аккордеоны: **Заявителя / Кредитные / Залоговые* / Проектные**.
Каждая секция — список обяз/опц документов **из программы** + счётчик «3/5».
Секция Залоговые — только при залоговой программе. Гейт «Отправить в комиссию» берёт
обязательный список из программы (закрывает P3-12).

### Вкладка «Залог»* (закрыто)
Условная (залоговая программа). Предмет(ы) залога: **вид** (из разрешённых программой) · **оценочная
стоимость** · **описание**; допускает несколько предметов + коэффициент покрытия (сумма оценок ÷ сумма кредита).
Залоговые *документы* тут не дублируются — они в «Документах».

### Вкладка «Общая информация» (закрыто — детальный проход)

**Блок 1 «Данные заявителя»:**
- **Субъект** — read-only всегда (в группе = заёмщик/держатель; члены — на своей вкладке).
- **Номер телефона** — правка когда статус разрешает (маска +996, валидация).
- **Адрес** — read-only из карточки Субъекта всегда.

**Блок «Условия» (слияние прежних блоков 2+3, трёхслойный diff-провенанс):**
Один блок вместо «условия заёмщика» + «одобренные комиссией». Показываем **все** условия из программы.
Каждое поле — цепочка происхождения, вытесненные значения **зачёркнуты**:
1. база = **программа** (конверт);
2. **заёмщик изменил** (в рамках конверта) → ~~значение программы~~ + выбор заёмщика;
3. **комиссия изменила** → ~~выбор заёмщика~~ + одобренное комиссией.
- Неизменённое поле = одно значение (без зачёркивания). Изменённое видно: откуда пришло, кто менял.
- Пример «Ставка»: `~~12% программа~~ · 10% заёмщик · **9% комиссия**`.
- Поля: программа/валюта/сумма/срок/источник/цель/метод/периодичность/льготные + одобренные ставка/сумма/срок.
- Правка слоя «заёмщик» — по статусу (Черновик); слой «комиссия» пишется только комиссией.
- В группе: тот же diff на общих условиях; различия по членам (сумма/дата освоения) — на вкладке «Члены группы».
- Снимает нужду в отдельном «нулевом» блоке одобренных; **P3-15** (пустые селекты одобренных) закрывается
  показом их значений в слое «комиссия».

### Вкладка «График погашения» + модель графика (закрыто — сверка с калькулятором xlsx)

Эталон расчёта — `loan_calculator_v10.xlsx` (лист «График»). Проверено соответствие полям программы.

**Колонки графика (12):** № · Дата начала · Дата платежа · Дней · Ставка % · Остаток на начало ·
Проценты · Погашение осн. долга · Платёж (базовый) · Надбавка · ПЛАТЁЖ ИТОГО · Остаток на конец.

**Формула процентов (фикс P3-11):** `Остаток × (Ставка/100) × Дней / база`, база = 360 (финансовый) или
365/366 (календарный) — из «Метод расчёта дней» программы. Сейчас проценты = 0 при 12% — это баг рендера/расчёта.
Аннуитет: `PMT = P·T/(1−(1+T)^−m)`, `T = год.ставка·f/12/100`. Дифф: осн.долг = `P/m` фикс.

**Разрывы калькулятор↔программа — решения владельца 2026-07-04:**
- **Полугодовая периодичность (f=6)** — ДОБАВИТЬ в программу (сейчас Ежемес/Ежекв/Ежегодно; нет полугодия).
- **Привязка последнего платежа** (по 1-му платежу / по дате выдачи) — ДОБАВИТЬ в программу.
- **Ступенчатые ставки** и **распределение накопл.% как обяз. поле** — НЕ нужны (отброшены).
- Уровень кредита (не программа/заявка): Дата выдачи, Дата 1-го платежа — фиксируются при выдаче;
  программа даёт лишь окно (Период освоение→1-й платёж Мин/Макс дней).

**Льготный период — генерится из условий программы полностью (капитализации НЕТ):**
Программа (таб 5 мокапа) несёт 3 независимых блока (по осн.сумме / по начислению % / по процентам),
в каждом при включении: Тип (Фиксир./Диапазон) + **Продолжительность в мес** (Фикс→мультивыбор
1/3/6/12/18/24 · Диапазон→c..по) + Условия. Длительность наследуется в заявку линзой Фикс/Диапазон
(как сумма/срок). → разрыв «нет длительности» закрыт.
- Помесячный treatment из 3 флагов+длительностей (фазная льгота допустима, считается месяц за месяцем):
  тело отсрочено = флаг осн.сумма ON и k≤длит.; % не начисляются = флаг начисление ON и k≤длит.;
  % отложены к уплате = флаг проценты ON и k≤длит.
- **Накопленные-отсроченные % возникают когда:** «по процентам» ON И «по начислению %» OFF
  (% начисляются, но не платятся → копятся). Гасятся **распределением, НЕ капитализацией**.
- **Дефолт распределения:** всё накопленное — единой суммой в **первый платёж после льготы**.
- **Опц. «период распределения» (с период / по период)** — новое поле в блок льготы «по процентам»
  **программы**; пусто = дефолт (первый платёж). Наследуется в заявку.
- «Тип расчёта процентов» (простой/сложный) settlement НЕ определяет — снято.

### Вкладка «История» (закрыто — аудит, P3-16)
Сейчас грид пустой. Наполнить журналом аудита.

**События (что логируем):**
- Смена статуса (старый → новый).
- Отправка в комиссию / возврат из комиссии.
- Решение комиссии: одобрено / отклонено / на доработку.
- **Изменение условий и денег** — field-level (было → стало): суммы, ставки, срок, льгота,
  члены группы (добавление/удаление, правка суммы/даты освоения), документы (добавление/удаление).
- Прочие поля (телефон и т.п.) — детально НЕ логируем (только сам факт «редактирование», без диффа).

**Формат грида:** `Дата-время · Пользователь · Событие · Было → Стало`. Только чтение, сорт по времени убыв.

**Гранулярность:** field-level для условий/денег/членов/документов (юридически значимо);
статусные вехи для остального. → закрывает **P3-16**.

## Топик 4 — Статусы и переходы (закрыто)

**Одна дорожка статуса** (статус залога убит — требование 2026-07-04). Набор:

| # | Статус | Смысл | Режим | Владелец |
|---|--------|-------|-------|----------|
| 1 | **Новый** | создаётся/заполняется, ещё не в комиссии (слиты «Черновик»+«Новый») | edit | кред. специалист |
| 2 | **На рассмотрении** | отправлен в комиссию | view | комиссия |
| 3 | **Требуется доп. информация** | комиссия **вернула** (не решение) специалисту на дозаполнение | edit | кред. специалист |
| 4 | **Одобрена** | решение комиссии = одобрить | view | — |
| 5 | **Отклонена** | решение комиссии = отклонить (терминал) | view | — |
| 6 | **Ожидает решения/программы** | *только поток B*: одобрено, кредита ещё нет — ждём привязку решения+программы | view | оператор |
| 7 | **На оформлении** | кредит(ы)/транши сформированы; договоры (кредитный + залоговый*) на нотариальной регистрации | view | оператор |
| 8 | **Оформлена** | договоры зарегистрированы у нотариуса — терминал успех | view | — |
| 9 | **Отозвана** | заявка отозвана до решения (терминал) | view | кред. специалист |

\* залоговый договор — только если у заявки есть залог.

Решения a–d (2026-07-04):
- **a.** «Черновик» и «Новый» слиты в один статус **«Новый»** (до отправки, редактируем).
- **b.** «Требуется доп. информация» = **возврат** комиссии (не решение). Решения комиссии = только
  {одобрить, отклонить}. Данные правит **кред. специалист** (не заявитель, не комиссия), затем повторно
  отправляет → «На рассмотрении».
- **c.** «Ожидает решения/программы» — **отдельный** статус (не подсостояние «Одобрена»).
- **d.** «Отозвана» — **нужен** отдельный терминальный статус.

### Граф переходов (закрыто)

```
Новый ──отправить──────────► На рассмотрении
Новый ──отозвать───────────► Отозвана
На рассмотрении ──вернуть──► Требуется доп. информация
На рассмотрении ──одобрить─► Одобрена
На рассмотрении ──отклонить► Отклонена
На рассмотрении ──отозвать─► Отозвана
Требуется доп. информация ──отправить──► На рассмотрении   (без лимита возвратов)
Требуется доп. информация ──отозвать───► Отозвана
Одобрена (поток A) ──сформировать кредит──► На оформлении
Одобрена (поток B) ──авто─────────────────► Ожидает решения/программы
Ожидает решения/программы ──привязка решения+программы──► На оформлении
На оформлении ──нотариус зарегистрировал договоры──► Оформлена
Отклонена ──пересоздать (клон)──► Новый (новая заявка)
Отозвана  ──пересоздать (клон)──► Новый (новая заявка)
Оформлена = терминал
```

Решения a–d (граф, 2026-07-04):
- **a.** Отзыв из «На рассмотрении» (пока комиссия не решила) — **разрешён**.
- **b.** «Одобрена → Оформлена» **не авто**. Между ними «На оформлении»: кредит сформирован,
  договоры (кредитный + залоговый, если есть залог) регистрируются у нотариуса. «Оформлена» — только
  после нотариальной регистрации.
- **c.** Отклонённую/отозванную заявку можно **пересоздать (клон)** → новая заявка в «Новый».
  Исходная остаётся в терминале (аудит). Клон vs полный тупик — клон.
- **d.** Возврат «Требуется доп. информация» — **без лимита** циклов.

### Статус-гейтинг тулбара (закрыто — P3-R7)

Все кнопки в тулбаре **заявки** жмёт **кредитный специалист**. Решения комиссии
(Одобрить/Отклонить/Вернуть) — **НЕ** в тулбаре заявки, а в тулбаре вкладки «Комиссия» (топик 5).

| Статус | Экшны тулбара заявки |
|--------|----------------------|
| Новый | Изменить · Отправить в комиссию · Отозвать · **Удалить** |
| На рассмотрении | Отозвать |
| Требуется доп. информация | Изменить · Отправить в комиссию · Отозвать |
| Одобрена | Сформировать кредит *(поток A)* |
| Ожидает решения/программы | Привязать решение+программу *(поток B)* |
| На оформлении | Зарегистрировать договоры |
| Отклонена | Пересоздать |
| Отозвана | Пересоздать |
| Оформлена | — (терминал) |

Решения (2026-07-04):
- **a. Удалить** — только в статусе **«Новый»** (черновик). В остальных запрещено (есть история/связи). → P3-R7.
- **b.** Решения комиссии {Одобрить, Отклонить, Вернуть} — **в тулбаре комиссии**, не заявки. Детально — топик 5.
- **c.** Все специалистские экшны (Отправить в комиссию, Сформировать кредит, Зарегистрировать договоры,
  Отозвать, Пересоздать, Изменить, Удалить) — жмёт **кредитный специалист**.

### Блокировка полей по статусу (закрыто — P3-10)

Правило редактируемости по статусу:

| Статус | Что редактируемо |
|--------|------------------|
| Новый | **всё** (по линзе наследования: single-value программы — read-only; multi-value — выбор из набора) |
| Требуется доп. информация | **всё как в «Новый»** (комиссия пишет комментарий что не так; специалист правит свободно) |
| Все прочие | ничего (полный read-only) |

**Иммутабельные поля** (нельзя менять кред. специалисту **никогда**, даже в «Новый»/«Треб. доп. инфо»):
- **Субъект** (заявитель) — фиксируется при создании.
- **Поля, изменённые комиссией** — после правки комиссией специалист их менять не может
  (третий слой diff-провенанса из топика 3 блокируется для специалиста).

**Телефон** — по общим правилам (edit только в «Новый»/«Треб. доп. инфо»), без спец-исключения.

→ закрывает **P3-10**. **Топик 4 закрыт.**

## Топик 5 — Связь «заявка ↔ комиссия» (закрыто)

### Одна комиссия + вкладка «Комиссия» (закрыто)

Было: вкладки «Кредитная комиссия» + «Залоговая комиссия» (2 тела, 2 голоса, параллельно).
Стало: **одна** вкладка «Комиссия», одно тело судит всё (кредит + залог, если есть), **один голос
на заявку, одно решение, один протокол** (требование 2026-07-04).

**Содержимое вкладки «Комиссия»:**
- **Шапка:** тело комиссии · № протокола · Дата заседания · Финальное решение.
- **Прогресс голосования** N% + tally: Одобрили · Отклонили · Воздержалось · Не проголосовали.
- **Члены комиссии** (грид): Роль (Член / Председатель) · Сотрудник · Решение · Крайний срок · Комментарий.
- **Тулбар решений** — по роли (см. ниже).

Решения (2026-07-04):
- **a. Один голос на заявку** целиком (залог — часть общего решения, не отдельный голос).
- **b. Два набора кнопок по роли:**
  - **Председатель** → **Финальное решение**: Одобрить · Отклонить · Вернуть.
  - **Член комиссии** → **голос** (совещательный): Одобрить · Отклонить · Воздержаться.
- **c.** Отдельной роли «залоговый эксперт» **не надо** — роли только Член / Председатель.

### «Отправить в комиссию» (закрыто)

Было: два экшна («Отправить в комиссию» + «Отправить в залоговую комиссию») + диалоги.
Стало: **один** экшн «Отправить в комиссию» → статус «На рассмотрении».

Решения (2026-07-04):
- **a. Выбор тела — из списка** (диалог «Выберите комиссию»); не авто.
- **b.** Дата заседания и состав членов **фиксируются при отправке**; **председатель или админ**
  может потом поменять.
- **c.** Повторная отправка после «Требуется доп. информация» — **та же** комиссия (та же запись
  голосования); **голоса сбрасываются** на повторный круг (подтверждено 2026-07-04).

### Governance комиссии (закрыто — P3-R4, без пункта d)

- **a. Финальное решение — вручную председателем**; голоса членов совещательные. Каждое сохранение
  пишет **аудит** (кто/когда/решение+причина, особенно при расхождении с tally).
- **b. Принцип 4 глаз — предупреждение, не блок:** автор/отправитель = член/председатель своей комиссии →
  не блокируем, показываем warning + метка в аудите.
- **c. Нет порога кворума** — председатель закрывает голосование в любой момент; tally (0/4) информативно.
- **~~d. Кредитная+залоговая параллельно~~ — СНЯТ** (одна комиссия, требование 2026-07-04).
- **e. Крайний срок члена — напоминание/эскалация, не авто-воздержание:** уведомить члена+председателя
  у дедлайна; голос остаётся открытым (не конфликтует с «председатель закрывает в любой момент»).

→ закрывает **P3-R4**. **Топик 5 закрыт.**
