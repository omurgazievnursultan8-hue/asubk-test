# Дизайн: запрос документов через интеграцию внешних систем — таб «Документы»

Дата: 2026-07-09
Файл-цель: `mockups/loan-application/loan-application.html`
Контекст: to-be вкладка «Документы» (CHECKPOINT 4, строки ~3887+). ЖЦ документа,
секции, роли, гейт «Отправить в комиссию».

## Проблема

Сейчас каждый документ комплекта закрывается только ручной загрузкой скана
(`docUpload`). Часть документов на деле доступна из госсистем через шину «Тундук»
и кредбюро — паспорт, ИНН, выписки, справки о доходах/налогах, кредитный отчёт.
Специалист должен уметь **запросить** такой документ из системы-источника, а не
только просить скан у заёмщика. Ручная загрузка остаётся как альтернатива.

## Решение (обзор)

Для документов с известным источником рядом с «Загрузить» появляется кнопка
**«Запросить из «X»»**. Запрос переводит документ в новый промежуточный статус
ЖЦ **«Запрошено»** (асинхронно — ждём ответа интеграции). Когда ответ пришёл,
файл приходит в статусе **«Загружен»** с пометкой источника, и дальше идёт
обычная 2-уровневая проверка (спец «Проверил» → головной филиал «Подтвердил»).

Решения, зафиксированные при брейнсторме:
- **Асинхронная модель** с промежуточным статусом «Запрошено» (а не мгновенное
  получение) — честно показывает природу интеграции.
- **Ответ приходит как «Загружен»**, а не сразу «Принят» — спец всё равно
  проверяет содержимое; единый флоу с ручной загрузкой, отличие только в бейдже
  источника.
- Интеграция — **не обязательный** путь: ручная загрузка всегда доступна рядом.

## 1. Маппинг источников

Новый конст `DOC_SOURCES` (ключ = `docId` → человекочитаемый ярлык системы):

| docId | Документ | Система-источник |
|---|---|---|
| `p1`,`p2` | Паспорт (лицевая/оборотная) | ГРС «Тундук» |
| `inn` | ИНН | ГНС «Тундук» |
| `reg` | Свидетельство о госрегистрации | Минюст «Тундук» |
| `egr` | Выписка из ЕГРЮЛ | Минюст «Тундук» |
| `inc` | Справка о доходах | Соцфонд |
| `tax` | Налоговые декларации | ГНС |
| `nod` | Справка об отсутствии задолженности | ГНС |
| `cbr` | **Кредитный отчёт (кредбюро)** — новая строка | Кредбюро |

- Интеграция только в секциях **`ident`** и **`fin`**. Печатные формы, договорные,
  залоговые, целевые — без интеграции.
- Новый документ `cbr` добавляется в секцию `fin` (Платёжеспособность),
  `req:false`, `ft:'both'`. Существующее «Согласие на запрос в кредбюро» (`kb`)
  остаётся ручным (его подписывает заёмщик, из системы не тянется).

## 2. Новый статус ЖЦ

Добавить в `DOC_STATUS`:

```js
requested: { label:'Запрошено', cls:'doc-st-reqd' },
```

- Входит в `OPEN_BLOCKS` (`s === 'requested'` → true), поэтому гейт «Отправить в
  комиссию» остаётся закрыт, пока ответ не пришёл. В `_docStats` `requested`
  считается открытым блоком (не `reqClosed`), но **не** `blocker` (это ожидание,
  не отказ) — чтобы формулировка гейта была «ждём ответа», а не «отклонён».
- Стиль бейджа `doc-st-reqd` — янтарный/ожидание (в ряд к прочим `doc-st-*`).

## 3. Хранение источника получения

На объекте-состоянии документа появляется признак, что файл получен из системы:
- поле `via` (ярлык системы) выставляется при `docReqFulfill`;
- используется в мете строки для бейджа «Получено из «X» · автоматически».

Для залоговых доков (плоский стор) интеграция не применяется — поле не нужно.

## 4. Действия и рендер строки (`docRow`)

Условие показа интеграционных кнопок: `DOC_SOURCES[docId]` задан **и** роль =
спец в режиме правки (`cap.editReq`) **и** секция обычная (не print/contract/coll).

Состояния и кнопки:
- **Открытый блок** (`required`/`rejected`/`expired`): к существующей
  «Загрузить»/«Перезагрузить» добавляется `[Запросить из «X»]` (ghost).
- **`requested`**: `[Отменить запрос]` (ghost) + демо `[⚙ Ответ получен]`
  (secondary, симуляция ответа интеграции для макета). Мета-строка:
  «Запрошено из «X» · <дата>».
- **После `uploaded`** (пришёл ответ): обычные кнопки «Проверил»/«Отклонить»
  + «Просмотр»/«Скачать». Мета: «Получено из «X» · автоматически» вместо/рядом
  со строкой «Файл: …».

Новые функции-обработчики (по образцу `docUpload`/`docAccept`):
```js
function docRequest(key){ d.st='requested'; d.reqAt='<demo date>'; ... }  // required→requested
function docReqCancel(key){ d.st='required'; delete d.reqAt; ... }        // отмена
function docReqFulfill(key){ d.st='uploaded'; d.via=DOC_SOURCES[id]; ... }// ответ пришёл
```
Все три перерисовывают вкладку и тулбар-гейт (`_docRerender`) и показывают тост.
`key` может быть простым `docId` (интеграция только в статических секциях, не в
залоговых `::`-ключах).

## 5. Гейт «Отправить в комиссию»

Логика `sendReady` не меняется — `requested` уже блокирует через `OPEN_BLOCKS`.
В `sendGateReason` при наличии запрошенных, но не пришедших доков дополнить
формулировку недобора: «… (ожидается ответ интеграции по N док.)». Это косметика
поверх существующей ветки `!st.met`.

## Влияние на существующий код

- `DOC_STATUS`, `OPEN_BLOCKS` — добавление одного статуса.
- `DOC_SECTIONS` секция `fin` — одна новая строка `cbr`.
- `_docStats` — учесть `requested` как открытый блок (уже покрыто `OPEN_BLOCKS`,
  проверить что не попадает в `blockers`).
- `docRow` — ветка интеграционных кнопок + мета источника.
- Новые функции `docRequest`/`docReqCancel`/`docReqFulfill`, конст `DOC_SOURCES`.
- CSS — класс бейджа `doc-st-reqd`.

Демо-сид (`_applyConfirmedSeed`) не трогаем: он выставляет обязательные в
`confirmed`; `cbr` необязательный, останется `required` — хороший кандидат
показать интеграционный флоу в демо.

## Вне скопа

- Реальные вызовы API/маппинг полей ответа — это макет, ответ симулируется кнопкой.
- Интеграция для залоговых/договорных/печатных документов.
- Автопроверка/доверенный авто-accept (решено: спец проверяет вручную).
