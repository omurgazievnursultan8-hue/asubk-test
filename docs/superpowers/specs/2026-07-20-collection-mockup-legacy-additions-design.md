# Взыскание — дозагрузка legacy-возможностей в макет (дизайн)

> Расширение существующего макета `mockups/collection/collection.html`.
> Базовый дизайн: `docs/superpowers/specs/2026-07-10-collection-mockup-design.md`.
> Источник «было»: `requirements/legacy/05-collection.md` (P5, UlutSoft, снято 2026-07-12).
> Backlog: `TODO.md` §«Фаза 9», `P9-R1…R9`. Проверка: `scripts/inspect/collection-check.mjs`.
> Дата: 2026-07-20.

## Зачем

Первая редакция макета перепроектировала **структуру** взыскания (процесс вместо
процедуры, фаза от меры-вехи, три измерения состояния, три оверлея). При этом из
legacy P5 выпали возможности, которые в спеке не обосновывались как отменённые —
они просто не попали в черновик. Ревизия legacy ↔ макет выявила 6 таких пробелов;
все 6 берём в макет. Ни один не отменяет принятых доменных решений — они
**достраивают** макет до функционального TO-BE, по которому команда строит модуль.

Что НЕ трогаем (осознанно перепроектировано, покрыто P9-R1…R8):
иерархию (процесс→фаза→журнал вместо процедура→фаза→событие), машину фаз,
отказ от селектора фазы, эстафету «Передачи дела» (≈ смена отделения legacy).

## Инвариант согласованности сумм

Сквозное требование ко всем финансовым данным макета: **три представления суммы
должны биться** —
1. bucket'ы требования (#1: основная + % + пеня + сборы),
2. `sum` меры в «Журнале мер»,
3. сумма по судебному акту (waterfall, P9-R9).

Демо-данные 6 процессов делаются согласованными вручную. Расхождение = макет врёт;
это дефект макета, а не «демо-условность».

## Общий подход

Один HTML-файл, данные в JS (`PROCESSES[]` + панель-билдеры). Все 6 — расширение
модели данных и добавление панелей/вкладок/списков. **Новых файлов нет.**
Демо-уровень сохраняется: действия-кнопки → тост, без рантайм-вычислений; результат
зашит в данные + заметка о правиле. Переиспользуется существующая дизайн-система
(`.cgrid`, `.pill`, строка `.total`, `.cgrid-empty`, модалки, тосты).

## #1 Финансовая разбивка требования — вкладка «Расчёт долга»

**Решение по размещению:** отдельная вкладка (не расширение «Охвата») — как legacy
вкладка 2 фазы; «Охват» отвечает за состав (какие кредиты), «Расчёт долга» — за
деньги. Разные заботы, разные границы.

**Модель.** У каждой пары кредит×предмет требования — разбивка:
```
debt: {
  principal: {accrued, paid, left},   // основная сумма
  interest:  {accrued, paid, left},   // проценты
  penalty:   {accrued, paid, left},   // пеня / фин. санкции
  fees:      {accrued, paid, left},   // сборы (госпошлина и пр.)
}
```
`left = accrued − paid` — производное, в данных не хранить (правило видно
разработчику, как `catOf`/`stageOf`).

**UI.** Таблица: строки = bucket'ы, колонки = Начислено · Погашено · Остаток,
итоговая строка (`.total`). Ниже — блок **waterfall**: порядок направления
погашения (при наличии судебного акта — сначала на сумму акта; иначе осн → % →
пеня → сборы), заметкой + компактной схемой стрелок. Опциональная демо-модалка
«Распределить погашение» показывает, как введённая сумма растекается по bucket'ам
(без реального пересчёта — предзаписанный результат).

**Истории в демо-данных** (каждый процесс иллюстрирует правило):
- **142** (Иск, просроч. сумма) — рост пени: осн 40 000 + % 2 600 + пеня растёт
  0→6 300, итого 42 600→48 900. Показывает, откуда прирост требования между мерами.
- **151** (залог, под паузой «реструктуризация») — bucket «пеня» помечен
  **ОТКРЫТЫЙ ВОПРОС: начисление санкций при паузе** (см. ниже). Макет видимо паркует
  нерешённый бизнес-вопрос, а не прячет его.
- **120** (мировое соглашение) — частичное погашение по новому графику: осн/%/пеня
  с ненулевым `paid`, остаток > 0.
- **104** (принятие имущества) — имущество принято на 295 000 при требовании
  310 000 → остаток 15 000; заметка о судьбе остатка (перекликается с orphan-
  остатком E2E-14 из legacy-прогона).
- **097** (безнадёжный долг) — после списания остаток 0; bucket-строки помечены
  «списано».

Закрывает: **P9-R9** (расчётные эффекты — waterfall, досрочное наступление долга
при извещении).

## #2 Судебные заседания — вкладка «Заседания»

**Решение по размещению:** отдельная вкладка (не вложенная таблица в «Журнале
мер») — как legacy отдельный «Список событий»; заседание — самостоятельная сущность
жизненного цикла судебной фазы, а не деталь строки меры.

**Модель.** `hearings[]` на процессе; каждое ссылается на меру-основание по её
номеру (`Иск` / `Извещение` / `Решение суда`):
```
{ measureNum, kind, place, when, participants:[], status, substatus }
```
- `kind`: «Судебный процесс» / «Извещение о назначении судебного процесса».
- `place`: районный суд (напр. «Ноокатский районный суд»).
- `status` + `substatus`: «на рассмотрении» + «отложено — неявка ответчика» /
  «назначено к рассмотрению по существу» / «вынесено решение».

**UI.** Таблица: Мера-основание · Вид события · Место · Дата+время · Участники ·
Статус (+подстатус). Кнопка «Назначить заседание» → демо-модалка. Заседания
привязаны к судебной стадии; на досудебных процессах вкладка показывает пустое
состояние (`.cgrid-empty`).

**Демо:** цепочка заседаний у **142** (Иск) с одним отложением по неявке.

## #3 Кросс-процессные очереди — 2 реестра + оживление сайдбара

Это не дубли списка процессов, а **очереди работы исполнителя**.

**Оживление сайдбара.** 4 пункта (Досудебный / Судебный / Исполнительное /
Отчуждение порядок) перестают быть тостами → **фильтруют список процессов по
стадии/разделу**. Активный пункт подсвечивается, заголовок отражает фильтр.

**Два новых реестра** (пункты сайдбара → отдельные list-views):
- **«Заседания»** — все заседания всех процессов, фильтр по **диапазону дат**
  (legacy §5, единственный список с датами). Колонки: Заёмщик · Процесс · Мера ·
  Вид · Место · Дата/время · Статус. Строка → переход в процесс (вкладка
  «Заседания»).
- **«Претензии»** — все претензии всех процессов, статус черновик/зарегистрирована
  (legacy §6). Колонки: Статус · Номер · Дата рег. · Заёмщик · Процесс · Сумма.
  Строка → переход в процесс.

Обе — переиспользуют разметку `view-list` (фильтр-хед + грид), новый рендер-билдер
на каждую.

## #4 Доказательная база — вложения на мере

**Модель.** `docs:[]` у меры: `{name, kind}` (напр. `{name:'СЭД-квитанция ПР-118.pdf',
kind:'подтверждение вручения'}`).

**UI.** В «Журнале мер» — колонка-скрепка со счётчиком (📎 N); клик → поповер/модалка
со списком вложений. В модалке «Зарегистрировать меру» — стаб загрузки файла
(input, без реальной загрузки). Закрывает требование **P9-R4** («способ + тип + дата
+ **скан**») — в текущем макете скан-слоя нет.

## #5 Ответственный + участники (ФИО)

**Модель.** `responsible` (ФИО) у меры; `participants[]` у заседания. Владелец-
подразделение (`owner`) остаётся — это разные уровни (подразделение ведёт процесс,
человек исполняет меру).

**UI.** Колонка «Ответственный» в «Журнале мер»; участники — в «Заседаниях».
Демо-ФИО в стиле legacy (Танаев У.Ш., Тукинова А.С., Эрнисов…).

## #6 Состояние претензии + аннулирование меры-вехи

**Состояние регистрации.** У мер-претензий — `regState: 'черновик' |
'зарегистрирована'`. Черновик: номер/дата пусты, pill «черновик» (перевод legacy
`Not registered`), доступны inline правка/удаление. Зарегистрированная — обычная
строка. Реестр «Претензии» (#3) фильтрует по этому состоянию.

**Аннулирование меры-вехи.** Действие «Аннулировать меру» в «Журнале мер». Если
мера была **вехой**, фаза **откатывается к предыдущей вехе** — единственный обратный
ход в forward-only машине фаз. Демо: тост + запись в «Историю» («Мера ИСК-77
аннулирована → фаза откачена к „Повторная претензия“») + заметка о правиле.
Обычная (не-веха) мера аннулируется без сдвига фазы.

## Открытые вопросы (макет фиксирует, не решает)

- **Пени при паузе** (перенос из базового спека, теперь ВИДИМ через #1): начисляются
  ли фин. санкции во время реструктуризации / мирового. Bucket «пеня» у 151 несёт
  явную метку вопроса.
- **Судьба остатка после принятия имущества** (#1, процесс 104): требование
  310 000, имущество принято на 295 000 — что с 15 000 (списание? довзыскание?).
  Legacy оставлял orphan-остаток (E2E-14).
- Из базового спека остаются: расширение обеспечения на поручительства/гарантии/
  депозит; флаг «право безакцептного списания» на договоре (P9-R7).
