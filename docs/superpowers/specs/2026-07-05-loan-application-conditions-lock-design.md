# Дизайн: блокировка/ограничение условий заявки по выбранной программе

**Дата:** 2026-07-05
**Файл-мокап:** `mockups/loan-application/loan-application.html` — форма создания заявки, route `#view-create`
**Отменяет:** пункт 4 спека `2026-07-05-loan-application-conditions-parity-design.md`
(«Программа = справка, никогда не блокирует»). Остальные пункты того спека (паритет виджетов,
плоские секции, объём, модель данных) остаются в силе.

## Проблема

Текущее поведение (спек паритета, п.4): после выбора программы все ~40 полей условий остаются
свободно редактируемыми; программа лишь предзаполняет пустые поля и показывает `⚑ отклонение`
(зачёркнутое значение программы) при расхождении. Оператор может ввести что угодно вне рамок
программы — заявка не гарантирует соответствие условиям программы.

Требование бизнеса изменилось: после выбора программы условия должны **блокироваться или
ограничиваться** рамками программы, а не быть просто справкой. Оператор не должен иметь возможности
завести заявку с условиями, противоречащими программе.

## Управляющее правило (после выбора программы, только `#view-create`)

Для каждого поля условий, в зависимости от того, что задаёт программа:

| Тип поля | Программа задаёт | Поведение в заявке |
|---|---|---|
| Списочное (enum) | 1 допустимое значение | **readonly**, подпись «Задано программой» |
| Списочное | 2+ допустимых | **обязательный select** только из допустимых |
| Списочное | значение оператора **вне** допустимого набора | старое стереть → обязательный select из допустимых |
| Сумма / срок / ставка | диапазон (min/max) | **ввод одного числа**, валидация ∈ [min, max] (границы = подсказка) |
| Фикс. не-списочное (валюта, метод погашения, база дней, фикс. штраф и т.п.) | 1 значение | **readonly** |
| Любое | программа НЕ задаёт значение | остаётся **редактируемым** (полный справочник / свободный ввод) |
| — | программа не выбрана | всё редактируемо (как сейчас до выбора) |

Область применения: **только форма создания** (`#view-create`). Детальная/редактирование
существующей заявки не затрагиваются этим изменением.

### Пограничные случаи

- **Значение введено до выбора программы, потом выбрана программа:**
  - значение входит в допустимые → остаётся выбранным (readonly либо предвыбранный пункт select);
  - значение вне допустимых → **стирается**, поле = обязательный select только из допустимых
    программой вариантов (оператор обязан выбрать заново).
- **Диапазон, введённое число вне [min,max]:** поле помечается invalid, блокирует сохранение
  (та же ленивая валидация P3-R6, что для суммы/срока).
- **Снятие программы:** снимаем блокировку/ограничение, поля снова полностью редактируемы;
  readonly-поля возвращают полный справочник/свободный ввод. Ранее введённые оператором значения,
  которые не были стёрты, сохраняются.

## Подход (выбран A)

Перестроить edit-gating внутри активного `renderCond` / `cdraft` (`#cf-cond-body`) — те же
виджеты-дословная-копия программы, но правило блокировки заменяет логику diff/«никогда не
блокирует». Источник истины один; метаданные допустимых значений уже есть в `PROGRAMS_MAP`
(массивы `vidKredita/naznachenie/istochnik/kreditLine/stavka/shtrafy/periodichnost`, скаляры
`currency/kind/purpose/line/repayAcct/rateType/...`, границы `amountMin/Max`, `termMin/Max`) и
`REF_VALUES` (полные справочники до выбора программы).

**Отклонены:**
- **B** — воскресить мёртвый `renderProgramConditions`/`#cf-progcond` и дорастить до 40 полей:
  ломает «дословную копию виджетов программы», плодит второй параллельный блок.
- **C** — пост-обработка DOM поверх `renderCond`: хрупко, рассинхрон с ре-рендером.

## Затрагиваемый код (`loan-application.html`)

- `renderCond()` (~стр. 3146) и его поле-построители — добавить режим «locked/constrained» на поле.
- `applyProgramSnapshot(prog)` (~стр. 3447) — вместо предзаполнения+diff теперь **вычисляет gating**
  каждого поля из `PROGRAMS_MAP[label]` и стирает не-конформные значения оператора.
- `selectProgram()` / `clearPicker('cf-program')` (~стр. 2985 / 2969) — подписи-хинты и вызов
  ре-рендера с новым правилом; при снятии программы — разблокировать.
- Валидация: список обязательных select программы (аналог `_progReqSelects`) участвует в проверке
  «Далее» (P3-R6, ленивая).
- Мёртвый блок `renderProgramConditions`/`REF_VALUES`/`PROG_ATTRS`/`selectFieldHTML`/`roFieldHTML`
  (~стр. 3029–3089) — переиспользовать идею (1→readonly, 2+→select, диапазон→число) в `renderCond`
  или удалить, чтобы не осталось второго параллельного механизма.
- `PROGRAMS_MAP` — при необходимости пополнить массивами допустимых значений те условия, которые
  сейчас заданы только скаляром, чтобы «2+ вариантов» имело данные (иначе всё скалярное = readonly).

## Diff-механика — судьба

Чипы `⚑ отклонение от программы` и зачёркнутое «программа: N» из спека паритета п.4 **убираются** —
расхождение теперь невозможно (поле либо readonly = значение программы, либо select из допустимых,
либо число в границах). Остаётся только: readonly-подпись «Задано программой» и хинт «Программа
допускает несколько — выберите один» для select.

## Проверка (ручная, мокап статичный)

1. Программа не выбрана → все секции редактируемы, переключатели/чипы/льготные работают.
2. Выбрать программу → списочные с 1 значением стали readonly «Задано программой»; с 2+ — select
   только из допустимых; сумма/срок/ставка-диапазон → поле числа с подсказкой границ.
3. До выбора ввести «Микрокредит» (нет в программе) → после выбора программы поле очищено, select
   из допустимых, требует выбора.
4. В поле-число суммы ввести вне [min,max] → invalid, «Далее» блокируется.
5. Снять программу → блокировки сняты, поля снова полностью редактируемы, справочники полные.
6. Сверка `#view-create` бок о бок с `loan-program.html`: те же классы/вёрстка/подписи/единицы.
7. Headless-верификация — обновить/дополнить `scripts/inspect/verify-create-conditions.mjs`
   под новое правило (readonly-состояния, наборы опций, стирание не-конформных).
