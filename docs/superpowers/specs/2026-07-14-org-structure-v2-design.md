# Оргструктура v2 — главная страница «Обзор»

**Дата:** 2026-07-14
**Артефакт:** `mockups/org-structure/org-structure-v2.html` (self-contained HTML, как `mockups/vedomstvo/vedomstvo.html`)
**Статус:** дизайн согласован, план реализации не написан

## 1. Задача

В v1 (`mockups/org-structure/org-structure.html`) модуль состоит из трёх равноправных видов —
«Подразделения» (дерево + карточка узла), «Должности», «Сотрудники подразделения». Точки входа,
которая показывает состояние структуры целиком, нет: чтобы увидеть вакансию руководителя или
истекающее и.о., надо обойти дерево узел за узлом.

v2 добавляет **главную страницу «Обзор»** — гибрид витрины и рабочего экрана: метрики сверху,
кликабельные рабочие списки снизу. Три существующих вида остаются как есть и становятся местом,
куда «Обзор» уводит по клику.

## 2. Границы

**В скоупе:** узлы дерева, должности, штатные позиции, назначения, и.о., ликвидация — то есть
собственные сущности модуля.

**Вне скоупа:** кураторство (модуль потребляет оргструктуру как справочник, обратной связи нет —
«слова "куратор" модуль не знает»), карточки сотрудников (`mockups/employee/`), ведомство
(`mockups/vedomstvo/` держит оргструктуру как read-only зеркало). Связь с ними — только гиперссылки.

Новых сущностей v2 не вводит. Все показатели «Обзора» — производные от модели v1:
`UNITS / BASE_PARENT / NAME_VER / PARENT_VER / TITLES / POSITIONS / EMPLOYEES / ASSIGN / ACTING`.

## 3. Сквозные механики (наследуются из v1)

- **Дата среза** (`state.asOf`) пронизывает весь резолв: `nameAt / parentAt / existsAt / liquidatedAt /
  childrenAt / headPosOf / actingOn / mainOn`. Все метрики и списки «Обзора» считаются НА ДАТУ, не на
  «сейчас».
- **Роль** (Кадровик / Наблюдатель) → `canEdit()`. Наблюдатель видит всё, но действия
  (`+ Подразделение`, «Назначить», «Ликвидировать») задизейблены.
- Дизайн-система — токены и классы `vedomstvo`: `.app / .sidebar / .topbar / .section / .metrics /
  .otree / .audit / .toast / .btn / .pill / .cgrid / .modal-*`, палитра `--asubk-*`.

## 4. Компоненты главной страницы

### Каркас

1. **`.sidebar`** — бренд + `NAV`. В группе «Организационная структура» новый первый пункт **«Обзор»**
   (active), затем существующие Подразделения · Должности · Сотрудники подразделения. `.sidebar-foot`.
2. **`.topbar`** — бургер, `.crumb-title` «Оргструктура · Обзор», **дата среза** (`input[type=date]` +
   чипы 2022 / 2023 / Сегодня), селектор **роли**.
3. **`.section-tools`** — строка контекста «Данные на ‹дата›» + кнопки **`+ Подразделение`** (primary,
   `disabled` для Наблюдателя) и **«Печать паспорта»**.

### Тело

4. **`.metrics` — 6 KPI-плиток.** Каждая кликабельна и ведёт в соответствующий рабочий список (§5) или
   отфильтрованный вид. Дельта — к предыдущему срезу (предыдущий чип даты).

   | Плитка | Что считает | Семантика |
   |---|---|---|
   | Действующих узлов | `existsAt` по типам: филиал / департамент / отдел / сектор | neutral |
   | Штат и факт | штатных позиций (`POSITIONS.units`) vs занято → **% укомплектованности** | info |
   | Вакансии | незанятых позиций; отдельно — **вакансий руководителя** | err, если вакансия руководителя > 0 |
   | И.о. | действующих `ACTING`; из них **истекают ≤ 30 дней** | warn |
   | Совместительства | `ASSIGN.kind = combine` | neutral |
   | Нарушения инварианта | `checkMainInvariant` — сотрудник с > 1 основным назначением | err |

5. **`.section` «Требует внимания»** — рабочие списки (табы-подразделы). Строка списка кликабельна →
   deep-link в карточку узла (`view=units`, выбранный узел, нужная вкладка).

   | Список | Колонки | Действие |
   |---|---|---|
   | Вакансия руководителя | узел · тип · родитель · вакантно с · дней | «Назначить» |
   | И.о. истекает / просрочено | позиция · сотрудник · до · основание | «Продлить» |
   | Нарушение «одно основное» | сотрудник · конфликтующие позиции | → карточка узла |
   | Узлы без штатных позиций | узел · тип · создан | → вкладка «Штатка» |
   | Ликвидация заблокирована | узел · сработавший гейт (дочерние / назначения / корень) | → вкладка «Ликвидация» |
   | Территория без обслуживающего узла | территория · с какой даты | warn, → дерево |

6. **`.otree` — мини-дерево на дату.** Read-only, 2 уровня + «раскрыть всё». Индикаторы v1: красная
   точка вакансии руководителя, зачёркивание ликвидированных, тег типа. Клик по узлу → карточка узла.

7. **«Штат и факт»** — компактная таблица/бары: по типам узлов (или топ-5 узлов по численности):
   штат · занято · вакансии.

8. **`.audit` — лента изменений за 30 дней** (до даты среза): создания, переименования (`NAME_VER`),
   перемещения в дереве (`PARENT_VER`), ликвидации, назначения. Колонки: когда · что · кто.

### Слои

9. **Модалка «Новое подразделение»** — переиспользуется из v1 без изменений: наименование · тип узла ·
   родитель в дереве · территория обслуживания; дата создания = дата среза.
10. **`.toast`** + печатный слой (`@media print` / `#printRoot`) — «Паспорт оргструктуры на дату».

## 5. Обработка ошибок и краевые случаи

- **Пустой срез** (дата раньше создания корня): дерево и все списки пусты → плашка «На выбранную дату
  структура не существует», метрики = 0, `+ Подразделение` доступна (создаст корень).
- **Вакансия корня** — не «нарушение», а ошибка конфигурации: отдельный красный баннер над метриками
  (как `.phead-banner` в v1), потому что рвёт всю цепочку ответственности.
- **Пустой рабочий список** — не скрывать таб, показывать «Нарушений нет» (иначе пустота читается как
  «не проверяли»).
- **Ликвидированные узлы** не попадают в метрики штата, но остаются в ленте изменений и в дереве
  (зачёркнутыми), если ликвидация произошла в пределах окна.

## 6. Проверка

Мокап исполняемый: открыть `org-structure-v2.html` в браузере, перебрать чипы дат (2022 / 2023 /
Сегодня) и обе роли. Критерий — метрики и списки пересчитываются от даты среза, каждая плитка и каждая
строка списка ведут в правильный вид/вкладку, у Наблюдателя все действия задизейблены.

## 7. Открытые вопросы

Нет. Спорные места закрыты на брейнштурме: главная = дашборд (не дерево и не карточка узла), гибрид
метрик и рабочих списков, скоуп — только свои сущности.
