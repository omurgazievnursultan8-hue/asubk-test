# Дизайн: блок «Условия кредита» в заявке = точная копия виджетов программы

**Дата:** 2026-07-05
**Файл-мокап:** `mockups/loan-application/loan-application.html` (форма создания заявки, route `#view-create`)
**Источник виджетов:** `mockups/loan-program/loan-program.html` (мастер-визард программы, вкладки 2–6 + вкладка 1)

## Проблема

Блок условий в форме создания заявки — усечённая кастомная подборка (~13 полей + 3 «голых»
чекбокса льготного периода). Программа определяет ~40 полей условий на вкладках 2–6 (Сумма/Срок,
Ставки, Штрафы, Льготный период, Платежи) + reference-поля вкладки 1. Заявка не покрывает ≈60 %
полей условий программы, а те, что есть, отличаются визуально и функционально.

Цель: поля условий в заявке — **точная копия** полей программы (вёрстка, вид, поведение), и все
поля условий программы присутствуют в заявке.

## Аудит паритета (что отсутствует в заявке)

Есть в заявке (reference-поля, редактируемы через `#cf-progcond`): Вид кредита, Назначение,
Источник финансирования, Кредитная линия, Процентная ставка (free-text), Штрафы (free-text),
Периодичность платежей; плюс Метод погашения, Метод расчёта дней, День месяца.

Отсутствует / расходится:

| Поле программы | Вкладка | Статус в заявке |
|---|---|---|
| Валюта (KGS/USD/EUR/RUB/KZT) | 1 | нет |
| Отрасль, Вид счёта погашения | 1 | нет |
| Ограничение суммы (Фикс/Диапазон) + фикс.суммы / min-max | 2 | нет (было одно число) |
| Ограничение срока (Фикс/Диапазон) + фикс.сроки / min-max | 2 | нет (было одно число) |
| Тип ставки (Фикс/Диапазон), плавающая ставка + тип + маржа | 3 | нет (free-text) |
| Штраф осн.сумма/проценты split, Фикс/Плавающая, маржа, Макс.размер штрафа % | 4 | нет (free-text) |
| Льгот.период: Тип, Продолжительность (multi), Условия предоставления, распределение | 5 | нет (3 чекбокса) |
| Способ графика, База дней в году, Очередь погашения, Обработка нерабочих, Привязка последнего платежа, Период освоение→1й платёж (min/max дней) | 6 | нет |

## Решения (утверждены с пользователем 2026-07-05)

1. **Модель полей:** точная копия виджетов программы (не плоские одиночные поля).
2. **Макет:** плоские карточки-секции (не мастер-вкладки).
3. **Объём:** только условия — вкладки программы 2–6 + reference-поля вкладки 1
   (валюта/источник/вид/назначение/линия/вид счёта). Без Документов и Залога.
4. **Программа = справка, никогда не блокирует** (управляющее правило, отменяет ранний вариант
   «закрыть/ограничить»):
   - Программа не выбрана → все поля редактируемы, значения по умолчанию нейтральные.
   - Программа выбрана, поле пустое/нетронутое → предзаполнить значением программы (default).
   - Поле остаётся редактируемым — **никогда не блокируется**.
   - Значение расходится со значением программы → эффективно значение оператора; значение
     программы показать **зачёркнутым** рядом + чип `⚑ отклонение от программы`.
   - Снятие программы → убрать справку/чипы, введённые значения сохранить.
5. **Сумма/срок:** envelope-виджеты программы (Фикс/Диапазон/списки) **плюс** отдельное поле
   «Запрашиваемая сумма/срок» (одно число), валидируемое в рамках envelope. Заявка показывает и
   envelope (условия), и конкретный запрос.

## Подход реализации

Выбран **A**: перенести код виджетов программы дословно из `loan-program.html`
(функции-построители `f/sel/combo/lookupF/msCombo`, переключатели Фикс/Диапазон, плавающая ставка,
штрафы, льготный период, платежи + соответствующий CSS) в форму создания заявки, разложить плоскими
карточками-секциями. Значения программы накладываются сверху слоем справка/diff. Так сохраняется
дословный визуальный и поведенческий паритет и остаётся self-contained HTML (конвенция мокапов).

Отклонённые: **B** (переписать с нуля — риск дрейфа от программы, нарушает «точную копию»);
**C** (общий подключаемый виджет для обоих файлов — ломает конвенцию self-contained HTML-мокапов).

## Структура секций (плоские карточки)

1. **Параметры** — Валюта, Источник финансирования, Вид кредита, Назначение, Кредитная линия,
   Вид счёта погашения.
2. **Сумма и срок** — Ограничение суммы (Фикс/Диапазон + фикс.суммы list / min-max);
   Ограничение срока (Фикс/Диапазон + фикс.сроки / min-max); **Запрашиваемая сумма** (число);
   **Запрашиваемый срок** (число).
3. **Ставки** — Тип ставки (Фикс/Диапазон), значения (multi-chip) / min-max; Использовать плавающую
   ставку → Тип плавающей + Маржа (mirror).
4. **Штрафы** — осн.сумма (Фикс/Плавающая + значение/маржа), проценты (Фикс/Плавающая +
   значение/маржа), Макс.размер штрафа %.
5. **Льготный период** — 3 блока (осн.сумма / начисление процентов / проценты): чекбокс → Тип
   (Фикс/Диапазон) → Продолжительность (multi) / from-to → Условия предоставления (textarea);
   + распределение отсроч.% (from/to) в блоке процентов.
6. **Платежи и расчёты** — Способ графика (периодичность / конкретные месяцы), Метод расчёта дней,
   База дней в году, Очередь погашения, День платежа, Обработка нерабочих дней, Привязка последнего
   платежа, Метод погашения, Период освоение→1й платёж (min/max дней).

Существующий `#cf-progcond` + 3 чекбокса заменяются этой структурой. Existing «Запрашиваемая
сумма/срок» переезжает в секцию 2.

## Модель данных

Черновик заявки получает ключи условий, зеркалящие `P()` программы: `amountType, amountList[],
amountMin/Max, termType, termList[], termMin/Max, requestedAmount, requestedTerm, rateType,
rateValues[], rateMin/Max, floatRate, floatType, floatMargin, penaltyMainType, penaltyMainVal,
penaltyMainFloatType, penaltyIntType, penaltyIntVal, penaltyIntFloatType, penaltyMaxPct,
graceMain/Accr/Int (+Type/Dur[]/DurFrom/DurTo/Cond), graceIntDistFrom/To, payMode, periodicity,
payMonths[], dayCountNum, dayCountBase, queue, payDay, weekend, lastPaymentAnchor, minDays, maxDays,
currency, source, kind, purpose, line, repayAcct`.

Плюс `_progSnapshot` — снимок значений выбранной программы (`PROGRAMS_MAP`) для diff.
Отклонение поля = `current !== snapshot[key]` при выбранной программе.

## Diff / отклонение — рендер

Каждое поле-обёртка после значения программы:
- если программа выбрана и `current === snapshot` → обычный вид (значение из программы как default);
- если расходится → значение оператора активно + строка `программа: <value зачёркнуто>` +
  чип `⚑ отклонение от программы`;
- если поле не задано программой → обычный редактируемый вид без справки.

Валидация суммы/срока: `requestedAmount` в границах envelope (min/max или ∈ amountList);
`requestedTerm` аналогично. При выбранной программе границы = из программы.

## Проверка (verification)

Мокап статичный (нет тестов). Ручная проверка сценариев в браузере:
1. Программа не выбрана → все 6 секций редактируемы, переключатели Фикс/Диапазон работают, multi-chip
   и льготные блоки разворачиваются.
2. Выбрать программу с фикс.ставкой → пустые поля предзаполнены значениями программы.
3. Изменить ставку вручную на отличное → чип «⚑ отклонение», `программа: N%` зачёркнуто, поле
   остаётся редактируемым.
4. Снять программу → справка/чипы исчезают, введённые значения сохранены.
5. Визуальная сверка «бок о бок» с `loan-program.html`: те же классы/вёрстка/подписи/единицы
   (%, мес., дней, сом).
6. `requestedAmount` вне envelope → ошибка валидации.

## Вне объёма (YAGNI)

Документы, Залоговое обеспечение, Отрасль как отдельная логика загрузки, вкладка Предпросмотр —
не входят. Реальная кодовая база приложения (Jmix/Vaadin) не затрагивается — это мокап-деливерабл
для команды разработки.
