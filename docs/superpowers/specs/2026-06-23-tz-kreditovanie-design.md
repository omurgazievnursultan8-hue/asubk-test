# Дизайн: ТЗ по подсистеме кредитования (as-is)

> Дата: 2026-06-23 · Ветка: `tech-spec-credit`

## Цель

Подготовить **техническое задание (функциональную спецификацию as-is)** по
подсистеме **«Система кредитования»** приложения АСУБК
(https://fkftest.okmot.kg/). Документ описывает текущее поведение системы как
есть — для передачи команде разработки/заказчику как эталон функциональности.

## Решения (зафиксированы при брейнсторме 2026-06-23)

| Параметр | Выбор |

## Границы

**Входит:** только подсистема «Система кредитования» — полный жизненный цикл:
`Госрешение → Кредитная программа → Заявка/Комиссия → Заёмщик → Кредит →
Транш/Освоение → Обслуживание`.

**Не входит:** Поручительства, Корпоративное управление, СУГС (субсидии),
Взыскание задолженности, Справочники/Сервисы/Администрирование — упоминаются
только как внешние связи, если на них ссылается подсистема кредитования.

## Единый шаблон каждого файла этапа

1. **Назначение** этапа — роль в жизненном цикле, кто работает
2. **Экраны** — список / деталь / создание, с URL и скриншотом-ссылкой
3. **Поля** — таблица: `поле · тип · обязательность · справочник · описание`
4. **Функции и действия** — кнопки, операции, переходы
5. **Бизнес-правила и расчёты** — формулы (аннуитет, резерв, разбивка
   платежа ОД+проценты и т.д.) с числовым примером
6. **Статусная модель** — где применимо (диаграмма статусов)
7. **Связи** с другими сущностями подсистемы

## Процесс работы

Полная ре-инспекция — основная трудоёмкость. По каждому этапу:
1. Пройти live-приложение через Playwright (переиспользовать/адаптировать
   скрипты из `scripts/inspect/`).
2. Зафиксировать поля, кнопки, правила, расчёты.
3. Написать раздел по шаблону.
4. Верифицировать ссылкой на скрипт + дату (конвенция репо:
   «проверено 2026-06-23, `scripts/inspect/...`»).

Порядок: index (каркас) → этапы 01→07 последовательно (каждый = инспекция +
написание). Этапы 03 (`/applications`) и 04 (`/borrowers`) приоритетны —
по ним нет отдельных верифицированных фич-доков.

## Критерии готовности

- `requirements/tz/index.md` + 7 файлов этапов написаны по шаблону.
- Каждое утверждение о поведении подкреплено ссылкой на инспекцию + дату.
- Все поля экранов перечислены с типом и обязательностью.
- Формулы расчётов приведены с числовым примером.
- Основной текст содержит только as-is (без дефектов/рекомендаций).
- `requirements/README.md` ссылается на новый каталог `tz/`.
