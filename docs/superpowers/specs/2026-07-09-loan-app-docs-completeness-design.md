# Вкладка «Документы» заявки — статусы комплектации и кнопки

> Дизайн-спек. Дата: 2026-07-09. Мокап: `mockups/loan-application/loan-application.html`, вкладка `#tab-2` (`renderDocs`).

## Контекст

Вкладка «Документы» уже реализована: аккордеон-секции, ЖЦ документа, счётчики
`N/M`, гейт отправки (`sendReady`), прогресс-бар, залоговые под-панели по
предметам. Задача — доработать **статусы комплектации** (именованные, не только
числа) и **кнопки** (три яруса: документ / пакет / роль-режим).

Актор-модель в мокапе: `_role` ∈ `spec` (кредитный специалист) | `com`
(член комиссии) | прочие (read-only). Режим правки спеца — `can(app).editReq`
(«Изменить»/draft). Комиссия голосует — `can(app).vote`.

## 1. ЖЦ документа — 6 хранимых статусов, «На проверке» производный

Хранимых статусов документа — **6** (было 7; «На проверке» больше не хранится):

| Ключ | Label | Когда | Владелец перехода |
|---|---|---|---|
| `required` | Требуется | файла нет | — |
| `uploaded` | Загружен | файл есть, заявка ещё не в комиссии | спец |
| `accepted` | Принят | комиссия приняла | комиссия |
| `rejected` | Отклонён | комиссия вернула (🔴 блокер) | комиссия |
| `expired` | Просрочен | срок истёк (🔴 блокер) | система |
| `waived` | Не требуется | освобождён спецом | спец |

**«На проверке» — производный вид, не хранимый статус.** Когда заявка в комиссии
(`app.status` ∈ голосующих стадий), документ со статусом `uploaded` отображается
как бейдж «На проверке» (`doc-st-rev`). Хранимое значение остаётся `uploaded`.
Это убирает «мёртвый» ручной переход `uploaded→review` (раньше никто его не
делал) и синхронит с реальным потоком: доки проверяет комиссия при голосовании.

Реализация: `DOC_STATUS` теряет ключ `review` как хранимый; вводится хелпер
`_dispStatus(app, st)` → возвращает `'review'` для `uploaded`, когда заявка на
рассмотрении комиссии, иначе исходный `st`. Бейдж рисуется по `_dispStatus`.
`OPEN_BLOCKS`, `_docStats`, гейты работают по **хранимому** `st` (не по виду).

## 2. Именованный статус комплектации (секция + пакет)

Рядом со счётчиком `N/M` — **чип-статус** комплекта. Один и тот же расчёт на
уровне секции и на уровне всего пакета (в верхнем баннере). Приоритет сверху
вниз (первое сработавшее — побеждает):

| Статус | Класс/цвет | Условие |
|---|---|---|
| Заблокировано 🔒 | `cs-lock` серый | секция договорная и заявка не одобрена |
| Есть замечания | `cs-block` 🔴 | ≥1 обязат. в `rejected`/`expired` |
| Укомплектовано ✓ | `cs-done` 🟢 | все обязат. `accepted` |
| На проверке | `cs-review` 🔵 | все обязат. загружены (не open-block), заявка в комиссии |
| Готов к отправке | `cs-ready` 🔵 | все обязат. загружены, заявка ещё черновик |
| В работе | `cs-wip` 🟡 | часть обязат. загружена |
| Не начато | `cs-none` серый | 0 обязат. загружено |

Если у секции нет обязательных (только опц.) — чип не рисуется, остаётся `M опц.`
(как сейчас). Счётчик `N/M` сохраняется как под-число рядом с чипом.

Хелпер `_complStatus({reqIn, reqDone, accepted, blockers, locked, atCommission})`
→ `{key, label, cls}`. Используется в:
- заголовке секции (`renderDocSection`) — заменяет голый цвет на `doc-count`;
- шапке залоговой под-панели предмета (`doc-item`);
- верхнем баннере пакета (`renderDocs`) — рядом с «Собрано обязательных N из M».

Для расчёта нужен подсчёт `accepted` (сейчас `_docStats` считает только
`reqClosed`/`blockers`). Добавить в `_docStats` и в поитемный подсчёт секции
поле `reqAccepted`.

`atCommission` — предикат «заявка на рассмотрении комиссии» из `app.status`
(тот же набор, что триггерит вид «На проверке»).

## 3. Гейт отправки (`sendReady`) — логика без изменений, слова выровнять

Логика верна и остаётся: «закрыт» = не (`required`/`rejected`/`expired`), т.е.
статуса `uploaded` **уже достаточно** для отправки (принимает комиссия позже).
Косметика:
- при `met && !coverage` пакет-чип = «Готов к отправке» (не «Укомплектовано»);
- тултип disabled-кнопки «Отправить» = `sendGateReason(app)` (есть) — без правок логики.

## 4. Кнопки

### 4a. Ярус A — по документу (`docRow`)

| Роль/режим · статус | Кнопки (слева→направо) |
|---|---|
| спец·draft · `required` | **Загрузить** (primary) · Не требуется |
| спец·draft · `uploaded` | Заменить · Просмотр · Скачать · Не требуется |
| спец·draft · `rejected`/`expired` | **Перезагрузить** (primary) · Просмотр · Скачать |
| спец·draft · `waived` | Вернуть *(undo waive)* |
| комиссия·vote · `uploaded` (вид «На проверке») | **Принять** (success) · **Отклонить** (danger) · Просмотр · Скачать |
| print-секция (спец·draft) | + Печать (перед Загрузить/Заменить) |
| contract-секция (спец, пост-одобрение) | + Сгенерировать |
| read-only (has file) | Просмотр · Скачать |

Новое против текущего:
- **Скачать** (`docDownload`) рядом с «Просмотр» на всех статусах «есть файл»;
- **Вернуть** (`docUnwaive`) для `waived` — сейчас waive необратим;
- «Не требуется» показываем для обязательных на `required`/`uploaded` (как сейчас).

### 4b. Ярус B — по пакету / секции

**Панель пакета** — новая строка кнопок над/под баннером в `renderDocs`
(видна спецу в режиме draft):
- **Отправить в комиссию** (primary) — гейт `sendReady`; при `false` — `disabled`
  + `title=sendGateReason`. *(Дублирует основное действие тулбара шапки —
  здесь как акцент на вкладке; можно указывать одну, если дубль лишний.)*
- **Запросить недостающие** (secondary) — собирает все обязат. `required` в
  список, показывает модалку-чеклист (мок: тост «Список из N док-тов отправлен
  заёмщику»).
- **Скачать пакет** (ghost) — мок-действие (тост «Формируется архив…»).
- **Опись** (ghost) — мок-действие печати описи комплекта.

**Заголовок секции** — без кнопок действий (чип-статус + `N/M` + шеврон).
Держим чистым.

### 4c. Ярус C — роль и режим (видимость)

| Актор | Видит |
|---|---|
| спец, **не** в правке | Просмотр/Скачать; общая кнопка «Изменить» входит в режим (существующий механизм) |
| спец, режим «Изменить» (draft) | полный ярус A + панель B |
| комиссия на голосовании | Принять/Отклонить + Просмотр/Скачать; upload/waive скрыты |
| read-only (прочие роли) | Просмотр/Скачать |
| договорные секции | спец действует без режима правки (пост-одобрение) — как сейчас |

**Правило hidden vs disabled:** upload/waive/replace скрываем по роли (нет прав —
не видно). Гейт-primary «Отправить» показываем **disabled с тултипом-причиной**
(видно, но нельзя — пользователь понимает, чего не хватает).
