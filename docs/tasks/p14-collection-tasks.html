Взыскание задолженности — задачи для команды разработки

20 задач, выстроенных в строгом порядке сборки: разработчик идёт сверху вниз — сначала ядро (каркас, модель «только факты», две ведущие оси, машина фаз, категория), затем виды, затем меры, сроки, гейты и особые состояния. Не начинать следующую, пока не закрыта предыдущая. Это сквозной модуль, живущий поверх любого кредита в просрочке, а не одна фаза кредитного цикла; живого экрана в приложении нет — модуль проектируется с нуля, поэтому дефектов стенда по нему тоже нет. Источник истины — TODO.md, раздел «Взыскание задолженности» (P14-R1…P14-R20); он разворачивает старый P9-R1…R9 (фаза 9) до фундамент-первого плана. Здесь то же самое человеческим языком: в чём боль → что сделать. Эталон интерфейса и поведения — готовый макет mockups/collection/collection.html v2 (проверка — node scripts/inspect/collection-check.mjs, jsdom).

Приоритет: 🔴 блокер (без этого модуль не работает) · 🟠 важно · 🟡 средне.

Порядок сборки — строгий. Номер шага в начале карточки — обязательная очерёдность; нумерация = порядок зависимостей: каждая задача опирается на уже сделанные и ни на одну из будущих. Ядро (шаги 1–7) — фундамент; без него меры, сроки, гейты и особые состояния некуда вешать. Значок Есть эталон в макете значит, что поведение уже отрисовано и проверено в mockups/collection/collection.html — разработке остаётся перенести его на боевую платформу (Jmix).

Дефектов живого стенда по этому модулю нет: экрана в приложении не существует, модуль спроектирован с нуля по нормативу «Порядок мониторинга, взыскания и урегулирования задолженности» (99 п.) + ТЗ АСУБК-3, разд. 2.5.
